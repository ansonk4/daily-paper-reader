## Abstract
Memory for large language model (LLM) agents has rapidly evolved from simple retrieval-augmented mechanisms into a data management system that supports persistent information storage, retrieval, update, consolidation, and dynamic lifecycle governance throughout agent execution. Despite this evolution, existing evaluations still benchmark agent memory mainly through end-to-end task success metrics (e.g., F1, BLEU), while treating the underlying system as a monolithic black box. As a result, critical system-level concerns, including operational costs, architectural trade-offs across memory modules, and robustness under dynamic knowledge updates, remain insufficiently explored. In this paper, we present a systematic experimental study of agent memory from a data management perspective. We propose an analytical framework that decomposes agent memory into four core modules: memory representation and storage, extraction, retrieval and routing, and maintenance. Under this framework, we evaluate 12 representative memory systems and two reference baselines across five benchmark workloads spanning 11 datasets. Our extensive end-to-end evaluation shows that no single architecture dominates across all scenarios; instead, effectiveness depends heavily on how well the memory structure aligns with the workload bottleneck. Furthermore, through fine-grained ablation studies, we quantify their individual effects on representation fidelity, retrieval precision, update correctness, and long-horizon stability. Finally, we reveal cost-performance trade-offs under realistic workloads, showing localized maintenance is more cost-efficient than global reorganization. Based on these findings, we identify promising directions towards building truly agent-native memory systems. The code is publicly available at https://github.com/OpenDataBox/MemoryData.

## 2. Proposed Methodology

### 2.1 Analytical Framework

The paper decomposes every agent memory system into a 4-tuple **M_sys = ⟨R, S, Q, U⟩**:

| Module | Role |
|---|---|
| **R — Representation & Storage** | Logical format (token sequence, graph/tree, heterogeneous composite) and physical store (in-context register, single-engine DB, multi-engine) |
| **S — Extraction** | Transforming heterogeneous inputs (dialogue, tool logs) into memory primitives via raw concatenation, schema-free, or schema-constrained extraction |
| **Q — Retrieval & Routing** | Locating relevant memory via native attention, dense semantic KNN, subgraph traversal, autonomous LLM planning, or multi-stage hybrid execution |
| **U — Maintenance** | Lifecycle policies: timestamp multi-versioning, capacity-driven eviction (FIFO / score-based), LLM-driven semantic consolidation (inline compaction / tool-driven CRUD), and parametric optimization |

### 2.2 Technical Taxonomy

Each module is further sub-categorized (see Table 1 of the paper). Key design axes include:

- **Representation granularity:** raw discrete tokens → continuous latent vectors → temporal knowledge graphs → hierarchical trees → heterogeneous composites (e.g., MemCube in MemOS).
- **Storage substrate:** transient in-context registers → specialized single-engine (Vector DB, Graph DB, Relational) → heterogeneous multi-engine.
- **Retrieval paradigm:** attention-based → dense semantic → topological traversal → agentic routing → hybrid (sequential or parallel ensemble).
- **Maintenance strategy:** append-only + timestamps → hard eviction → score-based eviction → LLM-driven merge/split → offline parametric fine-tuning.

### 2.3 Evaluation Flow

1. Select 12 representative systems spanning all taxonomy categories.
2. Decompose each system into its ⟨R, S, Q, U⟩ configuration (Table 1).
3. Run end-to-end benchmarks → 5 research questions (RQ1–RQ5).
4. Generate controlled variants (one-module-at-a-time edits) → fine-grained ablation (M1–M4).
5. Report task effectiveness, retrieval fidelity, update robustness, long-horizon stability, and operational cost.

No new model training or novel loss functions are introduced; the methodology is purely analytical and empirical.

## 6. Main Conclusions and Findings

| Finding | Key Insight |
|---|---|
| **1. Workload-Aligned Memory** | No single architecture dominates; effectiveness depends on memory-structure–workload-bottleneck alignment. Graph/temporal → cross-session reasoning; coarse-to-fine filtering → exact grounding; trace preservation → stateful execution. |
| **2. Evidence-Centric Organization** | Retrieval quality depends more on *how evidence is organized* than on top-1 ranking. Structured (linked/hierarchical) memory excels for scattered or temporally distant evidence. |
| **3. Temporal Update Fidelity** | Reliable post-update behavior is a *pipeline-level* design problem, not a model-capacity problem. Graph-based methods handle updates most reliably; append-only stores struggle with targeted overwrites. |
| **4. Horizon-Structured Memory** | As horizon grows, the challenge shifts from storing more to choosing the right abstraction. Pure long-context prompting and flat dense memory degrade sharply. |
| **5. Operational Scaling Rule** | Efficiency is governed by *maintenance scope*, not structure alone. Localized maintenance (LightMem, MemTree) is far more cost-efficient than global reorganization (Cognee, MemoryOS). |
| **6. Representation Granularity** | Preserving usable evidence matters more than compactness. Raw text outperforms summaries for exact detail recovery. |
| **7. Late-Filtering Principle** | Extraction should preserve context at write time; aggressive filtering at write time degrades downstream reasoning. |
| **8. Retrieval Strategy Guidance** | Moderate hybrid fusion and lightweight planning improve retrieval; extra reflection on top of planning adds overhead without gains. |
| **9. Maintenance Design Principle** | Conservative (selective) consolidation is the best default; delayed flushing and overly coarse summarization both degrade performance. |

## 7. Strengths

- **Novel perspective:** First systematic study of agent memory from a *data management* (rather than NLP/algorithm) viewpoint, treating memory as a standalone infrastructure object.
- **Comprehensive taxonomy:** The ⟨R, S, Q, U⟩ decomposition is principled, modular, and general enough to accommodate future systems.
- **Multi-dimensional evaluation:** Goes well beyond end-to-end F1/BLEU to measure retrieval fidelity, update robustness, long-horizon stability, and operational cost.
- **Controlled ablation:** Fine-grained module-level ablations (M1–M4) quantify each component's individual contribution, overcoming the "black box" limitation of prior work.
- **Reproducibility commitment:** Public release of code, taxonomy, and unified evaluation runner (MemoryData GitHub repository).
- **Practical guidance:** Findings directly inform system selection (e.g., "use graph-based for fact revision, coarse-to-fine for exact grounding, trace-preserving for stateful execution").

## 8. Weaknesses and Limitations

- **No compute disclosure:** GPU type, count, and hardware specifications for latency measurements are not reported, limiting reproducibility of cost results.
- **No statistical rigor:** No confidence intervals, p-values, or variance across runs are provided; results may be sensitive to LLM sampling randomness.
- **English-only, text-only scope:** All benchmarks are in English and limited to textual QA / procedural DB tasks. Multilingual, multimodal, and real-time streaming scenarios are unaddressed.
- **Limited agentic complexity:** Benchmarks focus on conversational memory and single-domain procedural tasks; complex multi-agent, tool-heavy, or open-web agentic workloads are not covered.
- **Judge-model bias:** LLM Judge Accuracy (used in LongMemEval) depends on the judge model's own reasoning capabilities, potentially conflating memory quality with judge quality.
- **Snapshot evaluation:** Systems are evaluated at a single point in time; longitudinal degradation under continuous real-world usage is not modeled.
- **No formal cost model:** Operational cost is measured empirically but not formalized (e.g., no asymptotic analysis of index-build time vs. query latency vs. memory size).
- **Selection bias risk:** The 12 chosen systems, while representative, are predominantly from academic or startup sources; production-scale industrial systems (e.g., Microsoft Copilot Memory, Google ADK Memory) are discussed but not benchmarked.
