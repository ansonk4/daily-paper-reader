## Abstract
Research on agent memory has matured rapidly, but almost entirely on the text side: few existing benchmarks ask, in an interactive environment, when an agent genuinely needs to remember what it saw rather than what it could write down. We introduce DMV-Bench (Code: https://github.com/yyyujintang/DMV-Bench), the first interactive benchmark for multimodal-agent visual memory. DMV-Bench is built on a controlled home-furnishing e-commerce catalogue of 1,000 product variants in which a text-leakage contract keeps the discriminative signal of each task in the pixels alone. Across a chain of autonomous shopping sessions, every visited product image carries a unique, pre-rendered incidental cue, and the agent is later asked to recall a particular cued product and navigate to its URL. Inspired by dual-coding theory, we propose DualMem, a memory architecture that maintains a visual and a verbal code in parallel. On DMV-Bench, DualMem outperforms a caption baseline and three recent multimodal agent-memory systems at every chain length J in {5, 10, 15, 50} on both Gemini 2.5 Flash and Qwen2.5-VL-7B, with the lead surviving controls for memory-bank size and encoding-position bias, and an asymmetric dual-coding regime in which vision carries the cue end-to-end while the verbal channel plays a smaller query-grounding role.

# DMV-Bench: Diagnosing Long-Horizon Multimodal Agents' Visual Memory with Incidental Cue Injection

## 1. Core Problem and Overall Meaning

- **Problem.** Agent-memory research has matured rapidly but almost entirely on the text side. Existing benchmarks rarely ask, in an interactive environment, when an agent genuinely needs to *remember what it saw* rather than what it could write down. In realistic web-agent use, a user may later refer back to a product by a purely visual detail (e.g., "the lamp with the triangular brass base"); a text-only memory can confirm a lamp was seen, but may have nothing to say about *which* one.
- **Why existing benchmarks fail to answer this.** They (i) conflate textual and visual recall (captions/alt-text leak the answer), (ii) make the discriminative detail nameable (so a text memory is not really stressed), and (iii) flag the evidence in advance and probe at short range, leaving unflagged, long-horizon visual retention largely unmeasured.
- **Overall meaning.** The paper introduces **DMV-Bench**, to the authors' knowledge the first interactive benchmark for-session visual agent memory, and **DualMem**, a dual-coding-inspired memory architecture. The work reframes the "when does an agent need visual memory?" question into a controlled diagnostic: inject unique, unrehearsed visual cues into product images, wipe context between sessions, and measure how many session boundaries a cue survives.

## 2. Proposed Methodology

### 2.1 DMV-Bench design

- **Environment.** A realistic modern-furniture e-commerce storefront (Next.js + Playwright) with four navigation levels (homepage, category, style, product detail), 10 categories × 10 styles × 10 variants = **1,000 products**, each bound to a frozen `urlHash`.
- **Incidental cue injection.** Every product image carries a unique, pre-rendered visual cue (a small colored object) baked into the pixel at build time by Nano-Banana image editing. Cues are **unannounced** (never mentioned in the session prompt), **identity-bound**, and **globally unique** (bijective object×color vocabulary).
- **L2-leakage contract.** The cue vocabulary appears in *no* text channel (title, description, alt-text, URL slug, meta-tags, reviews); a pre-release audit rejects any occurrence. This forces the discriminative signal to live in pixels alone, so a text-only memory can only answer if its captioner happened to describe an object the task never pointed out.
- **Task structure.** A DMV-Bench instance is a chain of $J$ autonomous shopping sessions ($J\in\{5,10,15\}$ plus a Monte Carlo pilot at $J=50$). Within each session a memoryless ReAct agent comparison-shops across ≥3 product categories over 22–28 steps. Between sessions the in-context conversation is wiped; only the memory bank crosses the boundary. After encoding, a read-only ReAct agent is probed to navigate back to a particular cued product (e.g., "take me back to the product with the teal sleep mask"); success is exact URL match.
- **Recall reach.** The diagnostic axis is $r = \text{recall session} - \text{visit session}$. A reach-$r$ probe asks the agent to recall a product whose cue survived $r$ context wipes. Sweeping $r$ turns a single accuracy into a retention curve.
- **Efficient evaluation: shared-prefix rollout tree.** Instead of re-running a full $J$-session chain per probe, the first session is run once, then $B=5$ child sessions branch from its end-of-session memory, each branching $B$ ways to depth $J$. A node is executed once; descendants reuse its memory snapshot. Cost is $(BJ-1)/(B-1)$ runs while yielding $\approx B^{J-1}$ distinct recall paths—roughly a $J\times$ saving over flat re-runs.

### 2.2 DualMem architecture

Inspired by dual-coding theory (Paivio, 1971), DualMem maintains a **visual** and a **verbal** code in parallel.

- **Encoding.** Each observed product page $o$ is dual-coded:
  - Visual signal $v_o$ via **SigLIP-2** (Zhai et al., 2025) over the raw image.
  - Verbal signal $t_o$ via **SBERT** (Reimers & Gurevych, 2019) over the page's VLM-generated caption.
  - Both are L2-normalized and stored as four channels $\{I_i, c_i, v_i, t_i\}$ in one memory bank.
- **Retrieval.** For a recall query $q$, the same two encoders produce $q_v$ and $q_t$. For each bank entry $i$, raw scores are inner products $s_v^{(i)} = \langle q_v, v_i\rangle$ and $s_t^{(i)} = \langle q_t, t_i\rangle$. They are made commensurate by **min-max normalization** within the bank:
$$\hat{x}^{(i)} = \frac{x^{(i)} - \min_j x^{(j)}}{\max_j x^{(j)} - \min_j x^{(j)}}$$
and fused with a tunable weight $\alpha$:
$$s^{(i)} = \alpha\,\hat{s}_v^{(i)} + (1-\alpha)\,\hat{s}_v^{(i)}$$
with $\alpha = 0.75$ as the operating point. The top entry $e^* = \arg\max_i s^{(i)}$ is selected.
- **Injection.** Both the raw image $I_{e^*}$ and the caption $c_{e^*}$ are injected back into the VLM navigator, which emits the final action (e.g., `go_to_URL_1234`).

## 3. Experimental Design

- **Back-ends.** All memory architectures are audited on two VLM back-ends: **Gemini 2.5 Flash** and **Qwen2.5-VL-7B-Instruct**, using identical prompts, trajectories, and memory injectors; only model weights differ.
- **Baselines (seven architectures on a shared ENCODE/RETRIEVE/INJECT interface).**
  - *Reference baselines:* **NoMemory** (discards everything), **TextOnly** (bare product class), **Caption** (VLM-generated caption; strongest text-only baseline).
  - *Prior multimodal external memories:* **WorldMM** (Yeo et al., 2026), **M2A** (Feng et al., 2026), **MMA** (Lu et al., 2026).
  - *Proposed:* **DualMem**.
- **Metrics.** Task Success Rate (TSR) = fraction of recall probes whose final navigate action exactly matches the ground-truth URL. Reported both overall and stratified by recall reach $r$.
- **Confound controls.** TSR is stratified by (i) memory-bank size at recall and (ii) encoding position $t$, to check that a lead is not an artifact of smaller bank or position bias.
- **Ablations.** At $J=5$ on Gemini 2.5 Flash the paper ablates: encoder (SigLIP-2 vs CLIP), retrieval mode (visual-only, verbal-only, hybrid), injection format (image-only, caption-only, image+caption), and a fine $\alpha$ sweep over $\{0, 0.25, 0.5, 0.75, 1\}$.
- **Statistical rigor.** Because the shared-prefix rollout tree nests probes inside chain trunks, the authors report **cluster bootstrap 95% CIs** (resampling chain trunks, the correct unit of independence) and a **paired cluster permutation test** (DualMem vs runner-up M2A) that permutes signs at the trunk level.

## 4. Resources and Compute

- The paper does **not** specify GPU type, GPU count, or training/encoding wall-clock time. DMV-Bench is a diagnostic benchmark run on two commercial/open VLMs (Gemini 2.5 Flash, Qwen2.5-VL-7B) with cached trajectories; compute is reported only in terms of number of runs via the rollout-tree savings, not hardware.

## 5. Number and Sufficiency of Experiments

- **Scale.** The benchmark ships **46,265 / 18,588** recall-probe tasks on Gemini 2.5 Flash / Qwen2.5-VL-7B. Per back-end, $N=550$ encoding sessions were generated. The Monte Carlo $J=50$ pilot uses 5 chains with sparse reach sampling ($n_r=2{,}407$ / $2{,}449$).
- **Sweeps.** Four chain lengths ($J\in\{5,10,15,50\}$), two back-ends, seven architectures, plus per-reach breakdowns, bank-size and position stratifications, and ablations (encoder, retrieval, injection, $\alpha$).
- **Fairness.** All baselines see **identical replayed trajectories** and probes; only the memory architecture differs. The shared ENCODE/RETRIEVE/INJECT coordinate system forces each prior system to be adapted under the same harness.
- **Statistics.** Cluster-aware CIs and a paired permutation test are provided; the DualMem lead over M2A is significant ($p\le 0.003$) on all six $J\in\{5,10,15\}$ cells across both back-ends. The $J=50$ cells are flagged as underpowered (5 trunks each).
- **Assessment.** The experiment set is large, controlled, and fairly compared; the main weakness is the lack of per-back-end $\alpha$ tuning and the small number of chains at $J=50$.

## 6. Main Conclusions and Findings

1. **DualMem is the strongest architecture.** It leads at every $J$ on both back-ends (e.g., 82.7% vs 65.7% for runner-up M2A at $J=5$ on Gemini; 81.1% vs 70.4% on Qwen). The lead survives controls for memory-bank size and encoding-position bias.
2. **Visual information is necessary.** NoMemory and TextOnly sit at 0% everywhere, confirming the L2-leakage contract holds and text alone cannot solve the task.
3. **Asymmetric dual coding.** Vision carries the cue end-to-end; the verbal channel plays a smaller query-grounding role. The $\alpha$ sweep peaks at $\alpha=0.75$ (82.7%), with pure-visual at 80.1% and pure-verbal collapsing to 59.5%. Image-only injection (75.9%) essentially ties image+caption (76.9%); caption-only collapses to 65.1%. Replacing SigLIP-2 with CLIP costs 11.5 points, confirming that visual-code discriminability drives the gain.
4. **Retention degrades gracefully with reach.** DualMem retains 68.3% at $J=50$ Monte Carlo, vs ~58–59% for Caption and M2A, showing the architecture scales to long horizons.

## 7. Strengths

- **Clean isolation of visual memory.** The L2-leakage contract and incidental-cue protocol are a genuinely novel way to force the discriminative signal into pixels alone, making the "when is visual memory needed?" question answerable.
- **Realistic interactive setting.** A live e-commerce site with real DOM snapshots and rendered images, not curated thumbnails or static QA.
- **Efficient, scalable evaluation.** The shared-prefix rollout tree gives roughly $J\times$ savings and yields tens of thousands of probes.
- **Rigorous statistics.** Cluster bootstrap CIs and trunk-level permutation tests correctly handle the non-independence from shared prefixes; underpowered cells are explicitly flagged.
- **Fair, apples-to-apples comparison.** All baselines share the same trajectories, prompts, and ENCODE/RETRIEVE/INJECT interface.
- **Thorough ablations.** Encoder, retrieval, injection, and $\alpha$ are each ablated, cleanly separating the cue-carrying role of vision from the grounding role of text.

## 8. Weaknesses and Limitations

- **Synthetic, single-domain catalogue.** The modern-furniture store with image-edited cues leaves transfer to other visual domains (outdoor scenes, documents, GUIs) untested.
- **Limited VLM coverage.** Only two back-ends (Gemini 2.5 Flash, Qwen2.5-VL-7B); broader cross-VLM coverage and a human performance ceiling are deferred.
- **No per-back-end $\alpha$ tuning.** $\alpha=0.75$ is swept only on Gemini at $J=5$ and applied unchanged to Qwen; per-back-end tuning could yield further gains.
- **Static fusion weight.** The paper itself notes that a more adaptive vision/verbal fusion (per-query weighting or a learned gate) is left to future work.
- **Underpowered long-horizon pilot.** The $J=50$ Monte Carlo cells use only 5 trunks, so the permutation test is underpowered there (e.g., $p=0.057$ on Qwen, $p=0.69$ on Gemini for DualMem vs M2A).
- **Instruction-following confound.** Qwen visits fewer distinct products per session (4.21 vs 11.02 for Gemini) and falls below the "≥3 categories" floor in 33.1% of sessions, reducing its probe count in a way orthogonal to memory quality.
- **No compute/hardware reporting.** GPU type, count, and wall-clock time are not specified, making cost reproducibility unclear.
