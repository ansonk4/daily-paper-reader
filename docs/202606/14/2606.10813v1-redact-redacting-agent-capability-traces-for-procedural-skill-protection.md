## 摘要
用户依赖执行轨迹来观察智能体行为、诊断故障并确保问责。这些轨迹包含丰富的程序细节，包括工具调用、中间决策和错误恢复逻辑。然而，这些细节可能暴露私有的程序技能，使得下游方法能够在不访问模型权重或技能文件的情况下恢复关键公式、阈值和策略。为了量化这种风险并评估保护措施，我们构建了\textsc{CapTraceBench}基准，包含75个专业的长时域任务和跨七个领域的154个精选技能。我们还介绍了\textsc{RedAct} https://github.com/XuShuwenn/RedAct，一个受保护的轨迹发布框架，它定位受保护的关键信息，在保留对验证者关键证据的同时重写轨迹，并为下游来源分析嵌入行为水印。在代表性的轨迹重用方法中，\textsc{RedAct}将原始轨迹上的归一化技能转移（NST）从44.7%–67.1%降低到低于无技能基准的水平，同时保留审计证据。其独立的行为水印达到了93.6%–100.0%的真实检测率，误报率最多为1.9%。这些结果将公共智能体轨迹定位为安全接口，并表明选择性编辑可以在不移除审计证据的情况下减少程序能力泄露。

## Abstract
Users rely on execution traces to observe agent behavior, diagnose failures, and ensure accountability. These traces contain rich procedural detail, including tool invocations, intermediate decisions, and error-recovery logic. Yet this detail can expose private procedural skills, allowing downstream methods to recover key formulas, thresholds, and strategies without access to model weights or skill files. To quantify this risk and evaluate protection, we construct \textsc{CapTraceBench}, a benchmark of 75 specialized long-horizon tasks and 154 curated skills across seven domains. We also introduce \textsc{RedAct} https://github.com/XuShuwenn/RedAct, a protected trace release framework that localizes protected key information, rewrites traces while preserving verifier-critical evidence, and embeds behavioral watermarks for downstream provenance analysis. Across representative trace reuse methods, \textsc{RedAct} reduces normalized skill transfer (NST) from 44.7--67.1\% on raw traces to below the no-skill baseline, while preserving audit evidence. Its standalone behavioral watermarks reach 93.6--100.0\% true detection with a false alarm rate of at most 1.9\%. These results frame public agent traces as security interfaces and show that selective redaction can reduce procedural capability leakage without removing audit evidence.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：用户依赖智能体执行轨迹（traces）来观察行为、诊断故障和确保问责，但这些轨迹包含丰富的程序细节（工具调用、中间决策、错误恢复逻辑），可能暴露私有的过程技能（procedural skills）。下游方法无需访问模型权重或技能文件，即可从轨迹中恢复关键公式、阈值、策略等，造成能力泄露。
- **整体含义**：公共智能体轨迹本质上是安全接口，需要在轨迹的透明度/可审计性与过程技能保护之间取得平衡。本文形式化了**黑盒轨迹披露**（black-box trace disclosure）问题，构建了评估基准 `CapTraceBench`，并提出保护框架 `RedAct`，以实现选择性编辑的同时保留审计证据。

## 2. 方法论：核心思想与关键技术细节

### 2.1 问题形式化
- 智能体 `πT` 基于任务 `x`、历史 `h` 和过程技能 `s` 产生轨迹 `τ`（包含推理步、动作、观测及最终输出 `y`）。
- 公开策略 `ϕ` 将轨迹映射为公开产物 `z = ϕ(x, τ)`。原始轨迹 `ϕ_raw` 暴露大部分过程，仅答案公开 `ϕ_ans` 只显示 `y`，而 `RedAct` 的 `ϕ_RA` 生成受保护轨迹。
- 下游重用方法在有限预算 `B` 下观测 `D_ϕ^B`，不能访问模型权重、私有技能文件、隐状态等。

### 2.2 RedAct 框架
RedAct 包含两层：

#### （1）关键信息引导的重写（Key-Information-Guided Rewriting）
- **前提**：技能所有者（部署 RedAct）可以检视技能包。
- **定位**（Key-Item Locator）：用 LLM `g_η(d, s) → K_prot` 从任务指令和技能包中提取受保护关键项（公式、常量、阈值、专用工具选择、验证流程等）。轻量人工审查去重。
- **重写**（Rewriter）：基于 `K_prot`，LLM `ρ_θ(d, τ, K_prot) → z_RA` 将暴露关键项的中间轮次抽象为任务级描述，保留最终输出、工具使用证据、执行顺序等审计所需信息。

#### （2）行为水印（Behavioral Watermarking）
- 为追踪下游重用，将功能中立的行为模式（hooks）嵌入选定轨迹。
- 水印方案 `W_h = (W_check_h, W_inject_h, W_detect_h)`：检查插入点、注入无害动作-观测模式、检测学生轨迹中的对应行为。
- 定义两类钩子家族：**Standalone hooks**（独立于观测，如 Ritual Marker、Env Check）；**Contextual hooks**（仅在特定工具结果或错误后插入，如 Cross Check、Error Anchoring）。
- 具体注入算法（Algorithm 2）：在满足条件的轨迹中按比例 `r` 采样，注入特定模式（如 Env Check 插入 `python3 --version` 等环境探测动作），确保不改变最终答案和验证结果。

### 3.1 基准与数据集

### 3.2 对比模型与重用法
- **闭源模型**：Claude Opus 4.6、Sonnet 4.6、Haiku 4.5（Anthropic）；GPT 5.2 Codex（OpenAI）；Gemini 3 Flash、Gemini 3 Pro（Google）。
- **开源模型**：Qwen3-8B、Qwen3-4B（用于轨迹微调和水印检测）。
- **下游轨迹重用法**：
  - 单智能体技能提取（Extracted Skills）
  - 多智能体技能进化（Evolved Skills）
  - 智能体工作流检索重用（Retrieval Reuse）
  - 轨迹微调（Trajectory Fine-tuning，用于水印检测场景）
- **披露条件**：No Skills（无技能基线）、Oracle Skills（真实技能）、Raw Traces（原始轨迹）、Protected Traces（RedAct 保护轨迹）。

### 3.3 评估指标
- **Success Rate (SR)**：任务最终验证通过率。
- **Step Success Rate (SSR)**：步骤级验证通过率，反映部分完成。
- **Normalized Skill Transfer (NST)**：当前条件 SSR 相对于无技能基线的增益，除以 Oracle 增益。越低表示技能转移越少。
- **Recovered Protected Information (RPI)**：下游工件中恢复的关键项比例。

## 6. 主要结论与发现

1. **原始轨迹是脆弱攻击面**：原始轨迹几乎闭合与 Oracle 技能的差距（平均 SSR 73.5% vs 无技能基线 68.0%），使技能提取、进化、检索均接近 Oracle 效果（NST 44.7%-67.1%）。
2. **RedAct 有效抑制重用**：受保护轨迹下，三种重用方法的 SSR 均低于无技能基线（67.5%、66.3%、65.5% vs 68.0%），NST 降至非正数（-5.9% 及以下），RPI 下降 37-48%。
3. **水印提供来源证据**：独立水印（Env Check、Ritual Marker）真检测率 93.6%-100%，误报率 ≤1.9%；上下文水印检测率较低（16-32%）但零误报。
4. **保留审计证据**：受保护轨迹保留 91-96.6% 的最终答案、工具名、验证器路径等审计关键信息；移除 70.6% 的敏感关键项。
5. **关键信息引导重写优于通用重写**：通用重写仍残留正 NST，而关键信息引导使 NST 显著为负。
6. **硬任务泄露更严重**：原始轨迹在困难任务上增益更大（SSR +6.5-10.5%），保护后压缩至最多 +4.8%，部分方法甚至低于无技能基线。

## 7. 优点

- **问题新颖**：首次系统研究智能体轨迹中的过程技能保护问题，形式化为黑盒轨迹披露。
- **基准构建**：CapTraceBench 覆盖多领域、多难度，包含 154 个真实技能，可复现性强。
- **框架设计巧妙**：两层结构（重写+水印），在保护与审计之间取得平衡；重写利用技能所有者先验知识定位关键信息，水印提供法律/工程可用的来源追踪。
- **实验全面**：跨多个前沿闭源模型和开源模型，覆盖四种主流重用方式，并包含消融、鲁棒性分析、人评。
- **保护效果好**：显著降低 NST 至负值，同时保持轨迹可审计性。
- **水印实用**：独立水印检测率高、误报率极低，可实现实际部署中的重用检测。

## 8. 不足与局限

- **基准规模有限**：仅 75 个任务，且为可控实验室环境，真实部署中的噪声、动态工具环境、弱验证反馈等未充分模拟。

- **攻击模型局限**：仅考虑黑盒场景（下游仅见公开轨迹），未评估灰盒攻击（如访问部分技能文件或模型内部）或更高级的对抗删除水印手段（如除去固定标记、继续训练）。

- **缺乏真实部署验证**：未在真实用户或商业系统中评估框架可用性和用户接受度。
