## Abstract
We present HORIZON, a self-evolving agent framework that treats hardware design as repository-level code evolution. A Markdown harness is compiled into a project pack containing domain knowledge, an executable evaluator, an acceptance predicate, and a git/runtime policy; a hands-free agent loop then evolves an isolated git worktree, using repository operations for state management, tracing, and replay. This extends prior works of repository-scale self-evolution from EDA software systems, to hardware-design artifacts themselves. We evaluate our approach on ChipBench, RTLLM, Verilog-Eval, and nine CVDP categories, achieving 100\% benchmark completion across all suites with a fully hands-free agentic loop. However, we do not claim that agentic AI for hardware design is solved: these benchmarks are controlled proxies for a much broader engineering problem in chip design. Section~\ref{sec:discuss} examines the limitations of the current study and highlights open research challenges.

# Agentic Hardware Design as Repository-Level Code Evolution — Detailed Summary

## 1. Core Problem, Motivation, and Background

- **Problem statement.** Hardware design (especially RTL design) is hard to automate because correctness depends on cycle-level behavior, reset conventions, bit-widths, interface protocols, and simulator feedback — plausible-looking Verilog is not enough. Single-turn code generation cannot close the loop between candidate artifacts and executable verification.
- **Motivation.** Prior repository-scale self-evolution work (AlphaEvolve, SATLUTION, ABCEvo) evolved EDA *software* — the programs engineers run — under executable, evidence-gated loops. The missing step is to apply the same paradigm to the *hardware-design artifacts themselves* (RTL sources, testbenches, verification artifacts).
- **Research question.** Can hardware design itself be managed as repository-level code evolution, with a hands-free agent loop driving a version-controlled worktree to convergence under an executable acceptance gate?
- **Context within the field.** The paper sits at the intersection of:
  - RTL-specialized LLMs and data (VeriGen, RTLCoder, ChipNeMo, OriGen, CraftRTL, ScaleRTL) that improve first-attempt accuracy but do not define an agentic iteration loop.
  - Iterative/agentic RTL tools (AutoChip, RTLFixer, VerilogCoder, MAGE, ACE-RTL) that add tool use and repair but are built around per-module generation-and-repair pipelines.
  - Next-generation benchmarks (CVDP) that expose verification-oriented tasks where single-shot models still struggle.
  - Git-as-substrate agent designs (EvolveGit, Git Context Controller) that use version control as the scaffolding for agentic exploration.

## 2. Proposed Methodology

### 2.1 Core idea

HORIZON treats a hardware-design problem as a self-contained, version-controlled, automatically evaluated repository rather than a one-shot prompt:

1. A human author writes a **structured Markdown harness** containing the goal, domain knowledge, evaluator specification, and acceptance predicate.
2. A **bootstrap agent** compiles the harness into a **project pack** — a control plane that fixes the mission, domain skills, executable evaluator, correctness gate, and git/runtime policy.
3. A **hands-free agent loop** then evolves an isolated **git worktree**: it edits artifacts, runs the evaluator, scores the result, and either commits the new version or logs the failure. Git is the state-management, tracing, and replay substrate.

### 2.2 Formal objects

Let the input harness be *m*. A bootstrap tool loop *G<sub>ϕ</sub>* constructs a project pack

*p = G<sub>ϕ</sub>(m) = (π<sub>agent</sub>, E<sub>p</sub>, A<sub>p</sub>, Γ<sub>p</sub>, Ω<sub>p</sub>)*   (1)

where:
- *π<sub>agent</sub>* — agent policy prompt and tool contract,
- *E<sub>p</sub>* — executable evaluator/harness (compile, simulate, coverage, assertion, testbench checks),
- *A<sub>p</sub>* — acceptance predicate (the pass/fail gate),
- *Γ<sub>p</sub>* — version-control and artifact policy,
- *Ω<sub>p</sub>* — domain skills and repository instructions.

### 2.4 Agent loop and trace buffer

- The loop is fully hands-free after the initial harness: bootstrap → generate/edit → evaluate → accept/reject → log → next iteration.
- Each outer transition contains an inner trajectory of depth *K<sub>t</sub>* (variable-length): the agent reads state, plans, edits the worktree, invokes tools, interprets failures, repairs or submits.
- **Git-native tracing**: `git diff --cached` inspects staged edits; each accepted attempt becomes a `git commit` whose message and attached `git notes` carry the evaluator verdict and reward; `git log` recovers the full trajectory. Successful commits are positive repair examples; rejected attempts are negative examples. The repository history *is* the experience buffer.

### 2.5 Memory and session reuse

- There is no true Markov state; memory is handled pragmatically. The operative objective is to maximize cached tokens relative to freshly billed input/output tokens.
- HORIZON reuses a **persistent model session** across iterations so the harness, project pack, stable sources, and accumulated debugging context are served from the provider's prompt cache. Newly billed tokens are dominated by the current diff, the latest evaluator output, and the agent's response.
- The agent may condition on session memory *h<sub>t</sub>*:
  *a<sub>t</sub> ∼ π<sub>θ</sub>(· | s<sub>t</sub>, h<sub>t</sub>)*,   *h<sub>t+1</sub> = M(h<sub>t</sub>, s<sub>t</sub>, a<sub>t</sub>, y<sub>t</sub>)*   (8)
  but the source of truth remains the git worktree, project pack, evaluator outputs, and versioned trace. Campaign and review memories are kept separate so review stays an independent check.

## 3. Experimental Design

### 3.1 Model and environment

- **Backbone model**: GPT-5.3, fixed throughout all experiments.
- **Host environment**: AMD EPYC 9334 32-Core processor, 512 GB RAM.
- **Evaluators**: each suite's native open-source flows; commercial-EDA simulators where required (CVDP CID 012–014).
- **Mode**: single-agent, fully hands-free.

- The CVDP portion covers 9 of the 13 CVDP categories (code-generation and verification-generation sides), including completion, specification-to-RTL, modification, reuse, linting/QoR, stimulus generation, checker generation, assertion generation, and debugging.

### 3.3 Protocol and metrics

- **Iteration** = one automated outer step: agent edits the worktree, runs the evaluator, and either commits a passing version or logs a rejection.
- **Best-so-far pass rate** = fraction of tasks for which a passing version has been committed by a given iteration.
- **Earliest-best iteration** = first iteration that attains a benchmark's maximum observed pass rate.
- **Coverage** (for CID 012 and CID 014 where parsed coverage data exists) is reported as an observational signal, *not* as an optimization target — the acceptance gate is the CVDP pass condition.
- **Token consumption** is measured as cumulative tokens through the earliest-best iteration, split into cached vs. freshly billed input/output tokens.

### 3.4 Compared methods

- The paper does **not** run a head-to-head comparison against other agentic RTL systems. Iteration-0 pass rates reflect the state of the repository after the first step of HORIZON's own agentic evolution process (not standalone model Pass@1), and are reported to characterize the starting point of the loop rather than to benchmark against other methods.

## 6. Main Conclusions and Findings

1. **100% benchmark completion**. HORIZON drives every evaluated suite — ChipBench, RTLLM-2.0, Verilog-Eval v2, and all nine CVDP categories — to a 100% pass rate under a fully hands-free agentic loop. The only residual miss (one ChipBench task) is traced to a specification–harness defect in the original benchmark.
2. **Convergence paths vary widely**. Legacy suites saturate within 1–5 iterations; CVDP categories require up to 82 iterations (CID 002). A very low first-iteration pass rate does not necessarily imply slow convergence (e.g., CID 013 climbs steadily from 3.8% to 100% in 19 iterations).
3. **Token cost is concentrated in hard categories**. The three legacy suites consume only 2.9% of total tokens; CID 002 alone consumes 26.7%. The final few percentage points on difficult categories absorb a disproportionate share of the budget.
4. **Cached tokens dominate**. ~91% of all tokens are cached input, keeping marginal cost per additional repair iteration low.
5. **Coverage is a secondary signal**. For CID 012, average per-design coverage at convergence is 97.9% (not 100%) because the acceptance gate stops each design as soon as it passes; the loop lifts a low-coverage tail rather than only nudging already-high-coverage designs.
6. **The main research bottleneck shifts**. Once executable feedback makes correctness converge, the key metrics become **convergence efficiency** (token/time cost) and **verification quality** (robustness of the generated tests), not final pass rate.
7. **Agentic hardware design is not claimed to be solved**. The benchmarks are controlled proxies; real chip projects involve incomplete specs, changing constraints, downstream integration, human review, PPA tradeoffs, and validation targets not captured here.

## 7. Strengths

- **Novel paradigm extension**. First system (to the authors' knowledge) to carry repository-level, evidence-gated self-evolution from EDA software to hardware-design artifacts themselves, and to complete all evaluated RTL benchmark suites end-to-end without human intervention.
- **Clean, general formulation**. The project-pack abstraction and repository-traced semi-Markov framing are domain-agnostic; the same machinery is intended to host RTL, EDA-software, and methodology/flow tasks.
- **Git-native design**. Using git as the state-management, tracing, and replay substrate is principled and practical: diffs, commits, logs, and notes give a free, replayable experience trace and keep the repository history as the experience buffer.
- **Honest reporting**. The authors explicitly flag the ChipBench harness defect, distinguish iteration-0 results from standalone Pass@1, report coverage as observational rather than as an optimized objective, and devote a full section to limitations including reward-hacking risk and long-turnaround reward.
- **Rich diagnostic data**. The paper reports not just final pass rates but full pass-rate trajectories, cumulative token trajectories (absolute and normalized), and per-design coverage curves, giving a detailed picture of *how* convergence happens.
- **Efficient session reuse**. The persistent-session, cache-maximizing design keeps ~91% of tokens as cached input, demonstrating a practical path to low marginal cost per repair iteration.

## 8. Weaknesses and Limitations

- **No baseline comparisons**. The paper does not compare HORIZON against other agentic RTL systems (AutoChip, RTLFixer, VerilogCoder, MAGE, ACE-RTL) or against frozen single-shot models under the same token budget. The 100% completion is demonstrated but not contextualized.
- **Single run per suite**. No variance, confidence intervals, or repeated trials are reported. Results demonstrate feasibility but not statistical reliability.
- **No ablations**. The contribution of individual components (harness design, trace depth *D*, acceptance predicate, session reuse) is not isolated.
- **Reward-hacking / over-solving risk**. The agent has full access to simulator messages, failure traces, and evaluator artifacts during repair. A passing result may mean "satisfies the visible harness under the exposed traces" rather than "satisfies the specification under all reasonable tests." The paper acknowledges this but does not implement a mitigation.
- **No hidden/held-out test separation**. Unlike SWE-bench's structural test-withholding protocol, the benchmarks used here do not separate debugging feedback from final scoring. There is no mechanism to detect benchmark-specific adaptation.
- **Long-turnaround reward not addressed**. RTL pass/fail evaluations in the paper complete quickly enough for iterative repair. For PPA optimization or EDA-tool self-evolution, reward turnaround can be hours to days, fundamentally changing the agentic problem; the paper identifies this as an open challenge but does not solve it.
- **Coverage not optimized**. The acceptance gate is pass/fail, not coverage; the loop stops refining a design as soon as it passes. Coverage is reported as a diagnostic, not a target.
- **Compute details incomplete**. GPU type, count, and wall-clock time per campaign are not reported, making it hard to assess reproducibility or cost.
- **Generalizability claim is aspirational**. The paper frames HORIZON as a general host for chip-design work (architecture, microarchitecture, physical design, EDA software, methodology), but only the RTL instantiation is empirically validated.
- **Single backbone model**. All experiments use GPT-5.3; the framework's behavior with open-weight or smaller models is unknown.
