## Abstract
AI agents are driving a new software paradigm, with the ability to autonomously call tools, extract information, manage memory, and complete tasks that span applications and data sources. Most existing end-user operating systems, however, are designed for application-centric workflows and offer little native support for AI agents. This mismatch limits the wider adoption of agents and leads to execution overhead and safety risks when running agents on conventional systems. While the concept of agent-native operating systems is emerging, the research community lacks an open testbed to explore the architectural primitives desired for agent-mediated interaction. We present AOHP (Android Open Harness Project), an OS-level agent harness built on the Android Open Source Project (AOSP). The core design principle of AOHP is to treat agents as first-class OS actors, enabling adaptive user interfaces and agent-friendly runtime environments. AOHP preserves the mature Android software and hardware ecosystem while introducing three agent-oriented system mechanisms: personalized service composition, efficient agent interfaces, and secure information flow. Based on preliminary experiments on challenging tasks covering key capabilities of OS agents, AOHP shows clear advantages in task completion (+21.12% completion rate), execution cost (-51.55% token cost), and security-policy compliance.

# AOHP: Detailed Technical Summary

## 1. Core Problem, Motivation, and Background

- **Problem:** Conventional operating systems (e.g., stock Android) are designed for *app-centric* workflows—interfaces are fixed by developers, GUIs are rendered for human visual operation, and permission models enforce protections only at app boundaries. This creates a fundamental architectural mismatch with AI agents, which need to process structured text more efficiently than pixels, run multiple tasks in parallel, retain long-lived context, and make tool calls faster than humans can inspect.
- **Consequences:** The mismatch limits wider agent adoption, introduces significant execution overhead (excessive GUI navigation, brittle cross-app handoffs), and creates security gaps where sensitive data cannot be tracked across an agent's context and tool calls.
- **Motivation:** The research community lacks an open testbed to explore the architectural primitives needed for *agent-mediated interaction*. AOHP (Android Open Harness Project) is proposed as an open-source, OS-level agent harness built on AOSP to fill this gap.
- **Goal:** Redesign Android so that agents are treated as *first-class OS actors*, enabling adaptive user interfaces and agent-friendly runtime environments while preserving full compatibility with the existing Android app ecosystem.

## 2. Proposed Methodology

### 2.1 Core Idea
AOHP re-architects Android around three design principles:
1. **Personalized Service Composition:** Interaction surfaces are generated on demand rather than being fully predefined. The OS agent synthesizes task-level service entrances (e.g., a "shopping" entrance aggregating search, comparison, and purchase across multiple apps).
2. **Efficient Agent Interfaces:** The OS provides agent-native primitives—parallel background execution, structured UI, native sandbox runtime, unified file shortcuts, and event stream abstractions—to reduce visual-processing overhead and rigid sequential execution.
3. **Secure Information Flow:** Sensitive data is treated as OS-controlled state. Private plaintext is replaced with typed references (vault tokens) before reaching the agent; trusted system components mediate all plaintext operations with full audit trails.

### 2.2 Architecture (Four Vertical Layers)
- **Bottom – Android Ecosystem Layer:** Retains existing apps, system services, hardware resources, and platform APIs as the compatibility base.
- **Second – Unified Interaction Interface:** Normalizes traditional Android and agent interfaces into four invocation modes: **API, CLI, Structured UI, and Rendered GUI**. Agents select compact symbolic paths when available and fall back to visual operation only when compatibility requires it.
- **Third – AOHP Capabilities Layer:** Reorganizes services and agent functions into:
  - **System Memory:** Stores preferences, task state, histories, and policy decisions outside any single app. Distinguishes persistent profile memory, task-local memory, and sensitive memory.
  - **Skills:** Reusable service capabilities and execution routines with input/output schemas, preconditions, side effects, and policy labels.
  - **UI Utilities:** Support construction of generated entrances and task-specific interaction surfaces.
- **Top – Personalized Service Composition Layer:** Assembles capabilities into user-facing application surfaces. Each generated entrance contains a *task schema*, a *service graph*, and a *presentation policy*.

### 2.3 Horizontal Cross-Layer Mechanisms
- **Efficient Agent Interfaces (Section 5):**
  - *Parallel Background Interaction:* Decouples execution from the physical display via lightweight virtual displays, allowing agents to run waiting-heavy workflows in the background.
  - *Agent-aware UI Enhancement:* Abstracts GUIs into structured representations with lower redundancy and richer semantics, retaining rendered GUI fallback.
  - *Native Sandbox Runtime:* An OS-managed execution substrate where agents can run computation, transformation, and tooling independently of app-facing interfaces.
  - *Unified File Shortcut:* Treats files as first-class task objects at the OS boundary, unifying GUI file-producing steps and programmatic file-consuming steps into a single cross-app data plane.
  - *Event Stream Abstraction:* Supports subscription-based access to dynamic notification capture (buffering transient toasts/push notifications) and sensor streaming (accelerometer, gyroscope, microphone, camera).
- **Secure Information Flow (Section 6):**
  - *Policy Enforcement:* Evaluates data source, requested purpose, destination, action sensitivity, and approval state at runtime—not just static permissions.
  - *Sensitive Source Sanitization:* Replaces plaintext with typed placeholders (e.g., `<payment-card:uuid>`, `<home-address:uuid>`) before content enters the agent context. A data vault stores sensitive values behind opaque identifiers.
  - *Trusted Vault and Execution:* Agents submit intents to a trusted vault executor using placeholders; the executor checks policy, obtains user approval when necessary, and performs operations inside the trusted environment.
  - *Data-Flow Taint Tracking:* Associates taint metadata with sensitive data as it flows through copying, transformation, composition, and transfer, enabling audit trails and enforcement at system exits.

- **Completion Rate:** AOHP raises average completion rate from 54.44% to 75.56% (+21.12 points), fully solving 7 more tasks than the baseline (20 vs. 13 fully solved).
- **Efficiency (on 11 commonly-solved tasks):**
  - Tool calls reduced by 44.64% (129 vs. 233)
  - Duration reduced by 44.21% (18.93 min vs. 33.94 min)
  - Token consumption reduced by 51.55% (3.44M vs. 7.10M)
  - LLM requests reduced by 47.62% (143 vs. 273)
- **Security:** All 5 security test cases passed as expected—sensitive data is properly redacted, sensitive actions require user consent, unsupported access fails closed, and taint metadata is preserved through event streams.
- **Key insight:** Gains concentrate on tasks involving transient notifications, fine-grained in-app GUI manipulation, and multi-step cross-app or memory-dependent workflows.

## 7. Strengths

- **Novel problem formulation:** Identifies and formalizes the architectural mismatch between app-centric OSes and agent workflows—a timely and important contribution.
- **Comprehensive system design:** Three well-motivated, complementary mechanisms (personalized composition, efficient interfaces, secure information flow) that address multiple dimensions of the problem simultaneously.
- **Practical substrate:** Building on AOSP ensures real-world applicability and full app/hardware compatibility, rather than a research prototype disconnected from practice.
- **Strong empirical results:** Large, consistent improvements across completion rate, efficiency, and security metrics.
- **Open-source release:** Source code publicly available at https://github.com/aohp-os/aohp, enabling reproducibility and community building.
- **Clean experimental design:** Controlled comparison with the same agent on two OS environments; efficiency analysis restricted to commonly-solved tasks to avoid confounds.
- **Security model is principled:** Combines source sanitization, taint tracking, trusted vault execution, and policy enforcement into a coherent framework.

## 8. Weaknesses and Limitations

- **Single agent evaluation:** Only OpenClaw is tested; results may not generalize to other agent architectures or frameworks.
- **Benchmark scale:** 30 tasks is moderate; larger-scale benchmarks would strengthen statistical confidence.
- **Security evaluation scope:** The security case study uses a single custom annotated payment app with 5 test cases—not a comprehensive security audit across diverse real-world apps.
- **Compatibility coverage gaps:** The paper acknowledges that apps with custom rendering, anti-automation logic, and undocumented behavior may not be fully supported.
- **Capability discovery dependency:** AOHP depends on accurate service descriptions, side-effect labels, and policy metadata, which may require significant manual annotation for legacy apps.
- **Resource scheduling not addressed:** Background execution on mobile devices involves thermal, memory, and battery constraints that are not yet managed by scheduling policies.
- **Policy usability:** Fine-grained information-flow control requires user approvals that could become excessive; the paper acknowledges this as future work.
- **No user study:** The evaluation is purely task-based; no human evaluation of user experience, approval burden, or interface quality is reported.
- **Limited device testing:** The paper does not specify the range of Android devices or form factors on which AOHP was tested.
