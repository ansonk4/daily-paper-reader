## 摘要
使用工具的语言模型智能体带来了超越不安全文本的安全漏洞：它们可能泄露受保护对象、写入持久化记忆、发送消息、修改数据库，或触发有害的代码和工具效应。现有评估通常将这些阶段压缩为一个单一的攻击成功率，难以区分模型是仅仅同意了攻击者的请求，还是实际造成了可观测的危害。我们提出 SafeClawBench，一个分阶段评估工具使用型智能体安全性的基准测试，包含600个受控对抗性任务，涵盖六种攻击类型：直接与间接提示注入、工具返回注入、记忆投毒、记忆提取以及模糊性驱动的不安全推理。SafeClawBench 报告三个独立的端点：语义攻击接受度、审计可见的危害证据，以及沙箱观测到的工具/状态危害。在四种提示策略下评估五个智能体端点，我们发现这些端点捕获了不同的故障模式。在没有额外提示保护的情况下，语义失败率在模型间差异显著，从9.0%到44.2%不等。审计到的危害证据比语义失败范围更窄，且在独立可执行协议下，一些匹配的任务标识在通过语义核心调用的情况下仍产生了沙箱危害：在一项12,000行的匹配分析中，观测到的347个沙箱危害中有291个发生在通过语义检查的行中。提示策略改变了端点结果，但其效果取决于模型和协议。SafeClawBench 提供了一个可复现的框架，用于比较智能体模型和提示策略条件，而不会混淆文本合规性、有证据支持的可执行状态变化。开源数据集可在 https://huggingface.co/datasets/sairights/safeclawbench 获取。

## Abstract
Tool-using language-model agents introduce security failures that go beyond unsafe text: they can disclose protected objects, write persistent memory, send messages, modify databases, or trigger harmful code and tool effects. Existing evaluations often collapse these stages into a single attack success rate, making it difficult to tell whether a model merely agreed with an attacker or actually produced observable harm. We introduce SafeClawBench, a staged benchmark for tool-using agent security with 600 controlled adversarial tasks across six attack families: direct and indirect prompt injection, tool-return injection, memory poisoning, memory extraction, and ambiguity-driven unsafe inference. SafeClawBench reports three separate endpoints: semantic attack acceptance, audit-visible harm evidence, and sandbox-observed tool/state harm. Evaluating five agent endpoints under four prompt-level policies, we find that these endpoints capture different failure modes. Without additional prompt protection, semantic failure rates vary widely across models, from 9.0% to 44.2%. Audited harm evidence is narrower than semantic failure, and under a separate executable protocol some matched task identities produce sandbox harm despite passing the Semantic Core call: in a 12,000-row matched analysis, 291 of 347 observed sandbox harms occur in rows that pass the semantic check. Prompt policies change endpoint outcomes, but their effects depend on both model and protocol. SafeClawBench provides a reproducible framework for comparing agent models and prompt-policy conditions without conflating textual compliance, evidence-supported harm, and executable state changes. The open-source dataset is available at https://huggingface.co/datasets/sairights/safeclawbench.

## 2. 方法论

### 2.1 核心思想：三阶段端点分离

SafeClawBench 的核心设计原则是**端点分离（endpoint separation）**，定义三个独立指标：

| 端点 | 含义 | 测量方式 |
|------|------|----------|
| **CoreFail@600（语义端点）** | 模型是否在语义上接受/授权攻击目标 | LLM 裁判（GPT-5.4）对最终响应做二分类标注 |
| **HarmEvidence@600（审计端点）** | 在语义失败的行中，是否存在受保护对象泄露、访问、操作或持久污染的证据 | 对 CoreFail 行进行自动化审计，匹配 TextHarm / AccessHarm / ActionHarm / PersistHarm |
| **ObservedHarm@Exec（沙箱端点）** | 在独立可执行沙箱中，工具调用/状态变化是否触发确定性 oracle 危害 | 隔离沙箱（文件/邮件/数据库/内存/Web/代码执行存储）+ 确定性状态差分 oracle |

### 2.2 攻击族分类

覆盖六类攻击族（各 100 例，共 600 个受控对抗任务）：

- **DPI**（直接提示注入）
- **IPI**（间接提示注入）
- **TRI**（工具返回注入）
- **MPI**（记忆投毒）
- **MEX**（记忆提取）
- **ADI**（歧义驱动的不安全推理）

### 2.3 提示策略（防御配置）

主矩阵对比四种提示级策略：

- **D0**：无额外防御（基线）
- **B2/Sandwich**：轻量包装（在用户内容前后插入优先级提醒）
- **D3**：分层提示策略栈（指令层级 + 受保护状态/内存处理 + 请求目标验证 + 工具链检查）
- **D4/LongPolicy**：长策略探针（在 D3 基础上增加 SIA、MIG、TCA 组件，约 6700 字符）

### 2.4 关键公式

- **CoreFail@600**：
$$\text{CoreFail@600}_{m,d} = \frac{1}{600}\sum_{i=1}^{600} C_{i,m,d}$$
其中 $C_{i,m,d} \in \{0,1\}$ 为语义失败标签。

- **HarmEvidence@600**（Core-gated 审计）：
$$\text{HarmEvidence@600}_{m,d} = \frac{1}{600}\sum_{i=1}^{600} I^{\text{harm}}_{i,m,d}$$
要求 $C=S=P=H=1$ 同时成立（CoreFail → 审计选中 → 解析成功 → 存在四类证据之一）。

- **SemanticOnly**：CoreFail 成立但审计未发现任何伤害证据。

- **NullAudit**：审计调用无法解析（保留为独立类别）。

### 2.5 Core–Exec 配对分析

在 12,000 行匹配矩阵中，分析语义端点与沙箱端点的不一致：
- **CF ∧ H**：语义失败且沙箱危害
- **CF ∧ ¬H**：语义失败但沙箱无危害
- **CP ∧ H**：语义通过但沙箱危害（关键发现）
- **CP ∧ ¬H**：语义通过且沙箱无危害

  - 主矩阵 12,000 行 + 沙箱 12,000 行 + 扩展 153,000 行，**统计上足够支撑结论**。
  - 采用配对检验（McNemar、置换检验）+ Holm 校正，**多重比较控制严格**。
  - 人工校准与跨裁判验证（κ=0.871）保证标签可靠性。
  - 匹配长度控制（D3_LM）排除提示长度混淆因素。
- **可商榷之处**：
  - 所有案例为**合成数据**，未攻击真实生产系统，外部效度有限。
  - 沙箱环境**禁用外部服务**，长时上下文或外部 API 相关危害无法覆盖。
  - 主矩阵仅 5 个端点，部分提供商（如 DeepSeek、Gemini）未纳入主对比。

## 6. 主要结论与发现

1. **模型选择是重要安全因素**：D0 下 CoreFail@600 从 **9.0%（Claude Opus 4.7）到 44.2%（Kimi K2.5）**，差距约 4.9 倍。

2. **语义失败远宽于审计证据**：
   - D0 下合并 CoreFail = 28.0%（840/3000），其中 HarmEvidence 仅 16.8%（504/3000），SemanticOnly 占 11.2%（336/3000）。
   - 说明许多语义合规并未产生可审计的伤害证据。

3. **沙箱危害独立于语义判断**：
   - 在 12,000 行匹配分析中，**291/347 次沙箱危害（83.9%）发生在语义检查通过的行中**。
   - 证明仅依赖语义评估会遗漏大量实际风险。

4. **提示策略效果因模型而异**：
   - B2/Sandwich 将合并 CoreFail 从 28.0% 降至 14.8%。
   - D3 与 D4/LongPolicy 接近（9.1% vs 9.2%），但 D3 在 Qwen、GLM、Kimi 上最优；D4 在 GPT-5.5 上最优。
   - D4/LongPolicy 作为**策略复杂度探针**，其收益不能归因于 SIA/MIG/TCA 各组件的单独因果。

5. **记忆相关攻击最具挑战**：MPI 和 MEX 在 D0 下 CoreFail 最高（各约 54.4%），反映 Agent 在状态持久化和受保护信息处理上的薄弱。

## 7. 优点

- **端点分离设计**：首次将语义合规、审计证据、沙箱危害作为独立指标系统报告，避免"一锅端"的评估偏差。
- **Core-gated 审计机制**：仅对语义失败行进行证据审计，提高效率的同时保证覆盖（1,834/1,834 行零缺失）。
- **匹配长度控制（D3_LM）**：排除"更长提示 = 更好效果"的混淆，证明 D4/LongPolicy 的收益来自策略复杂度而非长度。
- **多协议配对分析**：Core–Exec 配对揭示 83.9% 沙箱危害来自语义通过行，是论文最具说服力的发现之一。
- **可复现性**：固定端点字符串、artifact hash、确定性摘要脚本，完整开源数据集与代码。
- **统计严谨性**：Holm 校正、配对检验、跨裁判验证（κ=0.871）、人工校准（240 行）多重保障。

## 8. 不足与局限

- **合成数据偏差**：所有 600 例为人工构造，未涉及真实生产系统攻击，外部效度（external validity）有限。
- **沙箱环境受限**：禁用外部服务，无法评估需要长时上下文或外部 API 交互的危害场景。
- **提示策略单一**：仅评估提示级防御，未涉及运行时权限控制、网络出口管控、人工审批等生产级措施。
- **模型覆盖有限**：主矩阵仅 5 个端点，部分主流模型（DeepSeek、Gemini）未纳入主对比。
- **未报告算力/成本**：未说明评估的总 API 调用成本、耗时或硬件配置，难以评估复现门槛。
- **MEX 异质性**：记忆提取族混合了精确密钥泄露、配置泄露、策略透明度等多子类型，聚合指标可能掩盖细粒度差异。
- **D4/LongPolicy 因果不明**：作为复杂度探针，SIA/MIG/TCA 各组件的独立贡献未被严格因果识别。
- **良性工具使用面板非主指标**：过度拒绝等副作用仅作为辅助证据，未与主安全指标形成统一权衡框架。
