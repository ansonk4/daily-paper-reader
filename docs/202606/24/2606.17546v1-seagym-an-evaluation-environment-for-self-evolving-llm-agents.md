## 摘要
自我进化的大语言模型智能体主要通过改变其智能体框架来改进：即围绕基础模型的结构化执行层，包括提示词、记忆、工具、中间件、运行时状态以及模型-工具交互循环。现有评估通常将此过程简化为孤立的任务得分或单一的时序曲线，掩盖了某项更新是否产生了可复用的改进、对近期任务过拟合、增加了成本或损害了原有行为。我们提出SEAGym，一种用于衡量智能体框架更新的评估环境，涵盖训练、验证、测试、回放和成本记录。SEAGym将兼容Harbor的基准测试转化为动态的自我进化任务源，提供训练批次、冻结的更新-验证、留出的域内和域外迁移视图、回放诊断以及保存的快照与指标记录。我们在Terminal-Bench 2.0和HLE上实例化SEAGym，在统一的轮次/批次协议下比较了ACE、TF-GRPO和AHE。结果表明，这些评估视图为演化过程提供了互补信号：频繁更新可能无法提升留出性能，有用的中间快照可能在后续发生退化，且任务源多样性和模型后端可能影响框架的可靠性。

## Abstract
Self-evolving LLM-based agents improve mainly by changing their agent harness: the structured execution layer around a base model, including prompts, memory, tools, middleware, runtime state, and the model-tool interaction loop. Existing evaluations often reduce this process to isolated task scores or a single sequential curve, obscuring whether an update produces reusable improvement, overfits recent tasks, increases cost, or harms older behavior. We introduce SEAGym, an evaluation environment for measuring agent harness updates across training, validation, test, replay, and cost records. SEAGym turns Harbor-compatible benchmarks into dynamic self-evolution task sources with train batches, frozen update-validation, held-out ID and OOD transfer views, replay diagnostics, and saved snapshot and metric records. Instantiating SEAGym on Terminal-Bench 2.0 and HLE, we compare ACE, TF-GRPO, and AHE under a shared epoch/batch protocol. The results show that these evaluation views provide complementary signals about the evolution process: frequent updates may fail to improve held-out performance, useful intermediate snapshots may collapse later, and source diversity and model backend can affect harness reliability.

## 1. 核心问题与整体含义

- **研究背景**：当前基于大语言模型（LLM）的智能体越来越多地采用“自我进化”机制——即通过更新其“智能体框架”（agent harness），包括提示词、记忆、工具、中间件、运行时状态等组件，来持续提升性能。
- **核心问题**：现有评估方法通常将智能体的进化过程简化为最终任务得分或单一学习曲线，无法区分：
  - 某项更新是否带来**可复用的改进**；
  - 是否对近期任务**过拟合**；
  - 是否**增加成本**（如 token 使用量）；
  - 是否**损害旧有能力**（即“灾难性遗忘”）。
- **整体含义**：论文提出 **SEAGym**，一个系统性的评估环境，用于全面衡量智能体框架更新的动态过程，涵盖训练、验证、测试、回放和成本记录，旨在为自我进化智能体提供更科学、可诊断的评估标准。

## 2. 方法论

### 核心思想
- 将自我进化建模为一种 **RL 风格的 MDP 过程**：
  - 状态 $ s_t \in S $：包含当前智能体快照（模型 + 可变 harness）、调度位置和任务上下文。
  - 动作：由智能体自身决定如何更新其 harness。
  - 环境：SEAGym 负责任务采样、反馈、调度和快照评估。
- 智能体快照定义为 $ A_t = (M, H_t) $，其中 $ M $ 是固定基础模型，$ H_t $ 是可变的 harness 状态（如提示词、记忆、工具等）。
- 更新规则为：  
  $$
  H_{t+1} = U(H_t, B_t, T_t, F_t)
  $$  
  即智能体根据当前批次任务 $ B_t $、轨迹 $ T_t $ 和反馈 $ F_t $ 来更新自身 harness。

### 关键技术细节
- **多视图评估机制**：SEAGym 提供五种评估视图：
  - **更新验证**（Update-Validation）：冻结中间快照，评估其在验证集上的表现；
  - **域内迁移**（ID Transfer）：测试在未见但同分布任务上的泛化能力；
  - **域外迁移**（OOD Transfer）：测试在分布偏移或跨领域任务上的迁移能力；
  - **回放诊断**（Replay Diagnostics）：检测对旧任务的保留或遗忘情况；
  - **成本记录**：统计 token 使用量、工具调用次数、运行时间等。
- **调度参数化**：支持配置状态持久性、任务重用、批次大小、更新时机等，适配单任务适应、在线迁移、批量学习等多种场景。
- **与 Harbor 框架集成**：复用 Harbor 的任务执行能力（容器化环境、验证器、并行作业），在其上构建动态自进化任务源。

### 算法流程（简化）
1. 初始化任务索引和分割（训练/验证/测试）；
2. 对每个 epoch：
   - 从训练集中采样一个批次；
   - 运行任务并收集轨迹和反馈；
   - 智能体根据自身规则更新 harness；
   - 冻结当前快照并评估其在验证集上的表现；
3. 最终评估所有快照在 ID、OOD 和回放视图上的性能；
4. 计算指标（成功率、增益、遗忘率、成本等）并保存记录。

## 3. 实验设计

### 数据集与场景
- **基准测试来源**：
  - **Terminal-Bench 2.0**：提供命令行和软件工程任务，强调执行、工具使用和容器化验证；
  - **HLE（Humanity's Last Exam）**：专家级问答任务，选取数学和物理作为源任务，CS/AI 和工程任务作为 OOD 迁移测试。
- **任务划分**：
  - 训练集：80 个任务（Terminal-Bench + HLE 数学/物理）；
  - 验证集：35 个；
  - 域内测试集：55 个；
  - OOD 测试集：80 个（HLE CS/AI + 工程）。

### 对比方法
- 比较了三种自进化机制：
  - **ACE**：通过反思轨迹生成技能库（skillbook），更新提示词可见的过程提醒；
  - **TF-GRPO**：基于分组 rollout 证据更新经验记忆，存储任务族策略；
  - **AHE**：直接编辑智能体 harness（包括工具、中间件、运行时行为），实现执行层级的修改。

## 4. 资源与算力

- **模型后端**：主实验使用 **DeepSeek-V4-Flash**；消融实验还包括 **GLM-5.1** 和 **GPT-5.4**。

  - 单次运行 token 消耗约 **1.0–2.6B rollout tokens**，更新操作额外消耗 **17–125M tokens**；
  - 运行时间（任务执行部分）约 **7–18 小时**（如 AHE 主运行 14h06m）；
  - 并发数：16 个任务并行执行，单个任务超时设为 1800 秒。
- **未明确说明 GPU 型号和数量**，但实验基于 Harbor 框架在容器化环境中运行，推测使用了多节点分布式部署。

## 5. 实验数量与充分性

- **实验组数**：共进行了 **12 组主要实验**，包括：
  - 3 种方法 × 1 组基线 = 3 组；
  - 3 种 batch size（10/20/40/80）消融 = 4 组；
  - 2 种数据源（混合 vs HLE-only）消融 = 2 组；
  - 3 种模型后端（DeepSeek/GLM/GPT-5.4）跨模型迁移 = 3 组。

  - ✅ **覆盖全面**：涵盖方法对比、超参数敏感性、数据多样性、模型迁移性；

## 6. 主要结论与发现

- **关键结论**：
  - **频繁更新 ≠ 性能提升**：Batch=10 和 80 时，验证和 ID 性能反而下降，说明更新频率需适中；
  - **中间快照可能坍缩**：AHE 在第 4  epoch 后出现严重运行时错误（消息协议不匹配），导致回放性能从 42/80 降至 6/80；
  - **源多样性影响恢复能力**：单一 HLE 数据训练导致最终快照完全失效（0% 准确率），而混合源训练能从中恢复；
  - **跨模型迁移不对称**：DeepSeek 训练的 harness 对 GLM 有正向迁移（+7.3 ID），但对 GPT-5.4 为负向（-3.6 ID）；
  - **成本与性能权衡**：TF-GRPO 的 rollout token 最高（2.33M/task），但 OOD 表现最差（-2.5%）。
- **核心观点**：**不能仅依赖最终得分或验证曲线评估自进化智能体**，必须结合多视图诊断来理解更新机制的有效性和稳定性。

## 7. 优点

- **方法设计亮点**：
  - 提出 **多视图评估范式**，将静态基准转化为动态自进化任务源，填补领域空白；
  - 引入 **冻结快照机制**，避免验证集泄露到更新过程；
  - 支持 **多种自进化机制接入**，保持方法无关性；
  - 提供 **可复现的记录系统**，允许离线重新计算指标。
- **实验设计亮点**：
  - 引入 **任务级回放网格**，直观展示每个任务在不同快照下的成功/失败/错误；
  - 分析 **更新 artifact**（如技能库、经验记忆、harness 补丁），解释性能变化的机制；
  - 跨模型实验揭示 **harness 迁移的方向性和脆弱性**。

## 8. 不足与局限

- **实验覆盖局限**：
  - 仅测试了三种自进化方法，未涵盖 RL 微调、混合进化、多智能体协同等；
