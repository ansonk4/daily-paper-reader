## 摘要
大语言模型（LLM）智能体在协作场景中的大规模部署，对具备权限管控、权威冲突解决及未授权数据防护能力的多用户、多主体交互机制提出了迫切需求。然而，当前LLM的单用户训练范式与多主体治理所需的硬性约束之间存在根本性错配，使得基于概率的提示词防护在多轮对抗性交互中极为脆弱。我们的核心洞察是：治理约束——谁被授权、哪些内容受限制、哪条指令优先——是确定性的运行时变量，应由执行钩子（execution hooks）强制实施，而非交由LLM自行处理。我们提出\textbf{Harness-MU}，首个模型无关、零微调的多用户LLM智能体基础设施框架。通过将语言生成与安全编排解耦，Harness-MU在最大化合规需求满足的同时，确保了权限边界不可突破。在\textit{Muses-Bench}基准上，跨越四个前沿开源与闭源模型的测试表明，Harness-MU在所有访问控制攻击下均实现了隐私保护，效用得分较标准基线提升0.28至0.39，指令遵循准确率最高提升48.9个百分点。Harness-MU推进了\textit{框架工程}（Harness Engineering）的理念，确立了系统性基础设施对于解决LLM多主体治理挑战的必要性。代码与数据已开源：https://github.com/YuanJrShiuan/Harness-MulUser。

## Abstract
The increasing deployment of large language model (LLM) agents in collaborative workflows demands robust multi-user, multi-principal interaction mechanisms capable of enforcing access permissions, resolving authoritative conflicts, and preventing unauthorized data disclosure. However, a fundamental mismatch exists between the single-user training paradigm of contemporary LLMs and the hard constraints required for multi-principal governance, rendering probabilistic, prompt-based safeguards vulnerable under multi-turn adversarial interactions.Our key insight is that governance constraints -- who is authorized, what is restricted, and whose instructions take precedence -- are deterministic runtime variables that should be enforced by execution hooks rather than entrusted to the LLM. We present \textbf{Harness-MU}, the first model-agnostic, zero-tuning infrastructure framework for multi-user LLM agents. By decoupling language generation from safety orchestration, Harness-MU guarantees unbreakable permission boundaries while maximizing compliant demand satisfaction. Across four frontier open-weight and proprietary models on the \textit{Muses-Bench} benchmark, Harness-MU achieves the goal of privacy preservation across all access-control attacks, outperforming the standard baseline by 0.28--0.39 in utility score and improving instruction-following accuracy by up to 48.9 percentage points. Harness-MU advances the philosophy of \textit{Harness Engineering}, establishing that systematic infrastructure is essential for solving LLM multi-principal governance challenges. The code and data are available at https://github.com/YuanJrShiuan/Harness-MulUser.

## 1. 核心问题与整体含义

- **研究背景**：大语言模型（LLM）智能体正被广泛部署于组织和团队环境中，需要同时服务多个用户。这些用户往往具有**不同的角色、非对称的权威层级和严格的隐私边界**。
- **核心问题**：当前LLM的训练和架构基于**单主体-智能体（SPA）范式**，当多个用户与同一个LLM交互时，会产生三种系统性故障：
  - 未授权访问敏感信息（隐私泄露）
  - 冲突指令的解决混乱或失效
  - 无法严格优先执行高权威指令
- **根本矛盾**：LLM的概率性token生成机制本质上无法保证确定性的硬约束（隐私和权威），尤其在多轮对抗交互中，基于提示的安全防护会被系统性地绕过。
- **核心洞察**：治理约束（谁被授权、什么被限制、谁的指令优先）应作为**确定性运行时变量**，由**执行钩子（execution hooks）**在基础设施层强制执行，而非交给LLM自身处理。

## 2. 方法论

### 2.1 核心思想

- **解耦架构**：将语言生成与安全编排完全分离，构建一个**模型无关、零微调**的运行时基础设施框架。
- **设计哲学**：遵循Harness Engineering范式——智能体的最终可靠性取决于系统基础设施，而非单纯的模型能力。

### 2.2 四模块解耦运行时流水线

1. **Gatekeeper（准入控制层）**：
   - 在任何模型调用之前，查询状态相关的语义权限算子Auth，为每个活跃用户生成**不可变的权限记录** $p_k = \langle u_k, \alpha_k, A_k, R_k \rangle$。
   - 仅从声明的组织策略状态计算权限集，不从语言上下文推断。

2. **Mediator（策略编排层）**：
   - 将耦合的全局多主体目标分解为**隔离的局部执行向量**。
   - 为每个用户计算 $\pi_k = \langle w_k(\alpha), G^\star \cap G_k, \Omega_k \rangle$（权威权重、过滤后的硬目标、行为姿态）。
   - 不直接与LLM交互，而是将策略指令注入到下游Worker线程中。

3. **Worker Collective（隔离生成层）**：
   - 为每个用户实例化**并行的单主体执行环境**。
   - 每个Worker仅看到自己的上下文视图 $\Gamma_k$，从不合并不同用户的历史记录，从根本上消除跨用户的提示注入攻击面。

4. **ComplianceChecker（可行域投影层）**：
   - 作为确定性的非线性投影算子，在生成后立即拦截原始候选动作向量。
   - 通过子串匹配重新评估隐私对齐方程，任何越界触发**fail-closed截断**，用固定的拒绝响应覆盖泄露内容。
   - 在指令遵循场景中，运行IHEval检查器，将失败的约束反馈给Mediator进行修正。

### 2.3 形式化框架

- 将多主体-智能体（MPA）交互形式化为广义决策框架，定义了用户空间、私有上下文空间、状态依赖的权限算子。
- 优化目标为**权威加权效用最大化**，约束条件为零泄露隐私边界和硬目标满足（公式3）。
- 证明了LLM的概率性token分布本质上无法满足该优化边界的约束要求。

### 2.4 六组件框架实例化

按照Meng et al. (2026)的框架分类法，实例化了 $H_{MU} = \langle E, T, C, S, L, V \rangle$：执行循环、工具注册表、上下文管理器、状态存储、生命周期钩子、评估接口。

## 6. 主要结论与发现

1. **绝对隐私保护**：Harness-MU在所有评估的底层模型上实现了**100%隐私保护**（Privacy = 1.000，零方差），彻底截断了所有泄露向量。

2. **效用不降反升**：
   - Overall得分提升：Qwen +0.240，DeepSeek +0.427
   - GPT基线Utility从0.257恢复到0.589（+0.332）
   - 证明了框架层隔离可以**同时提升安全性和有用性**。

3. **指令遵循显著提升**：
   - GPT的Aligned Accuracy从0.422提升到0.912（+48.9pp）
   - Conflict Accuracy从0.309提升到0.781（+47.2pp）

4. **ComplianceChecker的关键作用**：
   - 在DeepSeek上修正了6.8%的泄露（raw privacy 0.932 → 1.000）
   - 触发率为32.9%，且**Utility增量为零**（无误拦截）

5. **逆向缩放现象**：修正循环在基础模型可靠性最弱时收益最大（GPT触发率66.8%，DeepSeek 27.3%）。

6. **延迟-成本权衡**：虽然总Token消耗增加（29.8K → 151.6K），但由于Worker并行执行，实际墙钟时间**降低11%**（148s → 131s）。

## 8. 不足与局限

- **对抗模型单一**：所有多轮AC评估中攻击者固定为DeepSeek-V4-Pro，可能低估匹配/同质攻击者-防御者场景下的攻击效能。
