## 摘要
在智能体记忆管线中，LLM的位置——位于检索已知事实（已广泛基准测试）的召回平面与通过替换、释放、清除（基本未经测试）改变事实的控制平面之间——决定了系统能够恢复哪些遗忘失效模式。通过在一个包含385个案例的对抗性表面上比较十三种系统配置，我们观察到三种具有部分互补覆盖范围的放置机制：确定性原语足以处理词法/时间类别，但无法规范化（标识符混淆5%，跨语言0%）；写入时LLM可以恢复规范化（100%），但无法处理意图感知删除（前缀冲突和复合事实0%）；变异时钩子可以恢复意图感知删除（78-85%），并同时提升几乎所有类别（总体91.7-93.2%，每次385案例运行成本0.17美元，每个案例变异延迟2.3秒对比确定性64-191毫秒，召回路径不变）。我们通过ForgetEval（一个包含1000个案例的模板化套件加385个案例的对抗层，其中132个手工制作+253个LLM起草并由预言机验证，通过确定性子串匹配评分，并配有一个六方法适配器协议，采用诚实的N/A评分，可在130行代码中接入异构记忆存储）展示了这种权衡。接纳性由10名标注者的IAA（Fleiss' kappa = 0.958）和一个包含77个案例的外部作者子集（四名盲贡献者）证实，该子集复现了规范化不对称性并放大了联合放置提升（+27.8个百分点）。生产故障主要是遗忘故障而非召回故障，但现有基准仅衡量召回。ForgetEval和所有适配器均以MIT许可证发布。

## Abstract
Where an LLM sits in an agent memory pipeline -- between the recall plane that retrieves stored facts (extensively benchmarked) and the control plane that mutates them via supersede, release, purge (largely untested) -- shapes which forgetting failure modes the system recovers. Comparing thirteen system configurations on a 385-case adversarial surface, we observe three placement regimes with partly complementary coverage: deterministic primitives suffice for lexical/temporal categories but fail canonicalization (5% on identifier-obfuscation, 0% on cross-lingual); inscribe-time LLM recovers canonicalization (100%) but cannot help intent-aware deletion (0% on prefix-collision and compound-fact); a mutation-time hook recovers intent-aware deletion (78-85%) and brightens nearly all categories simultaneously (91.7-93.2% overall, $0.17 per 385-case run, 2.3s/case mutation latency vs. 64-191ms/case deterministic, recall path unchanged).   We expose the trade-off via ForgetEval, a 1000-case templated suite plus a 385-case adversarial layer (132 hand-crafted + 253 LLM-drafted oracle-validated) scored by deterministic substring match, paired with a six-method Adapter Protocol with honest N/A scoring that lets heterogeneous memory stores enter in 130 lines. Admission is corroborated by 10-annotator IAA (Fleiss' kappa = 0.958) and a 77-case external-authored subset (four blind contributors) that replicates the canonicalization asymmetry and amplifies the joint-placement lift (+27.8 pt). Production failures are predominantly forgetting failures rather than recall failures, yet existing benchmarks measure only recall. ForgetEval and all adapters are released under MIT.

## 1. 核心问题与整体含义（研究动机和背景）

- **生产失败的主要形式是遗忘失败**：如轮换的密码仍被推荐、GDPR删除的记录仍出现、用户职位在不同会话中矛盾、一次性验证码永久存在。这些失败中，检索成功了（检索到了正确的事实），但应用希望检索不到任何东西——这是控制平面（control-plane）的缺口。
- **现有基准的缺陷**：所有现有的LLM智能体记忆基准（LongMemEval、MTEB、BEIR、LOCOMO等）只测量**召回（recall）**，即“检索正确事实”的能力，从未测试记忆系统能不能被命令**忘记（forget）**。
- **核心研究问题**：LLM在记忆管道中的放置位置（在召回平面与控制平面之间）如何塑造遗忘失败模式？不同放置机制是否互补？

## 2. 论文提出的方法论

### 2.1 核心思想：放置位置决定遗忘特性

- LLM可以放置在三个关键位置：
  - **确定性原语**（无LLM）：纯添加/删除/搜索，依赖词法或向量匹配。
  - **写入时LLM（inscribe-time LLM）**：在事实写入时用LLM提取标签、实体链接，用于读取时规范化。
  - **变异时钩子（mutation-time hook）**：在替换、释放、清除操作时调用LLM，进行规划与执行。
- 不同位置恢复不同的失败模式子集，具有部分互补性。

### 2.2 ForgetEval基准

- **结构**：包含1385个案例，分为：
  - **模板套件（1000个案例）**：覆盖5个家族（超时、衰退、失忆、清除、漂移），每个家族有4个子模板，确定性生成。
  - **对抗层（385个案例）**：包含132个手工核心+253个LLM起草预言机验证的案例，覆盖10个攻击类别（如子串陷阱、前缀冲突、释义替换、否定陷阱、时间限定、共享属性、复合事实、标识符混淆、跨语言标识符、递归替换）。
- **评分**：使用确定性子串匹配（must_contain ⊆ top-10，must_not_contain ∩ top-10 = ∅），无需LLM评判。
- **数据接纳**：两道过滤（结构过滤+独立LLM评判），由Qwen-2.5-72B执行，多标注者IAA验证（Fleiss' κ = 0.958）。

### 2.3 Adapter协议

- 六个方法：reset、inscribe、recall_texts（必需），supersede、release、purge（可选），未实现返回NotImplementedError（计为N/A）。
- 允许异构记忆系统在约130行代码内接入，行为契约确保可比性。

## 8. 不足与局限

### 8.1 实验覆盖局限

- **语言与脚本**：仅英语；跨语言类别仅有9个脚本家族，每个仅1-12例，无法区分性能差异。

### 8.2 方法局限

- **子串评分盲区**：否定陷阱类别中，“Dana does not have production access”包含“has production access”子串，导致误判。需要未来引入NLI意识评判。

### 8.3 应用限制

- **缺乏真实部署实证**：仅结构映射到GDPR泄漏、密码管理失败等，未实证验证高分系统生产泄漏更少。

- **LLM质量敏感性**：钩子提升依赖LLM的JSON跟随能力（DeepSeek-V3提升28-30pt，Qwen仅13pt，Llama失败）。

### 8.4 其他

- **单体实验**：部分结论（架构无关性）仅基于3个后端和2个LLM家族，需更多复现。
- **时间与成本**：基准套件385案例，但A-MEM每案例14秒，可扩展性受限。
