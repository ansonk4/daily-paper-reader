## 摘要
随着大语言模型（LLM）智能体日益自主地选择工具，其在不同权限级别工具之间的选择已成为一个与安全性相关的重要问题。然而，以往的工具选择研究主要关注与安全无关的元数据偏好，对权限敏感的选择行为缺乏深入探讨。为填补这一空白，本文研究了过度特权化的工具选择问题，即尽管存在足够的低权限替代方案，智能体仍选择或升级至更高权限的工具。我们提出了ToolPrivBench基准，用于评估智能体在存在足够低权限替代方案时是否仍选择高权限工具，涵盖初始选择和短暂工具故障后的升级行为。在八个领域和五种反复出现的风险模式中，我们发现过度特权化的工具选择在大语言模型智能体中普遍存在，且会因短暂故障而进一步加剧。我们还发现，通用的安全对齐并不能可靠地迁移至最低权限工具的选择，而提示级别的控制在短暂故障下仅能提供有限的缓解。因此，我们提出了一种权限感知的后训练防御方法，教导智能体优先选择足够的低权限工具，仅在必要时才进行升级。缓解实验表明，该防御方法在保持通用能力的同时，显著减少了不必要的高权限工具使用。

## Abstract
As LLM agents increasingly select tools autonomously, their choices among tools with different privileges become safety-relevant. However, prior tool-selection studies focus on safety-agnostic metadata preferences, leaving privilege-sensitive choices underexplored. To address this gap, we study over-privileged tool selection, in which an agent selects or escalates to a higher-privilege tool despite a sufficient lower-privilege alternative. We introduce ToolPrivBench to evaluate whether agents choose higher-privilege tools despite sufficient lower-privilege alternatives, measuring both initial selection and escalation after transient tool failures. Across eight domains and five recurring risk patterns, we find that over-privileged tool selection is common among mainstream LLM agents and is further amplified by transient failures. We further find that general safety alignment does not reliably transfer to least-privilege tool choice, while prompt-level controls provide only limited mitigation under transient failures. We therefore introduce a privilege-aware post-training defense that teaches agents to prefer sufficient lower-privilege tools and escalate only when necessary. Our mitigation experiments show that this defense substantially reduces unnecessary high-privilege tool use while preserving general capabilities.

## 1. 核心问题与整体含义

- **核心问题**：LLM智能体在自主选择工具时，即使存在足以完成任务的低权限工具，仍倾向于选择或升级至高权限工具——即“过度特权化工具选择”（Over-Privileged Tool Selection）。
- **背景与动机**：
  - LLM智能体正从对话助手转向高度自主的Agent（如vibe coding场景），Agent不仅决定“做什么”，还决定“用什么工具做”。
  - 现有工具选择研究主要关注元数据偏好（如提供商身份、描述措辞），忽视权限维度的安全关键选择。
  - 现实中大量“vibe-coded”生产应用存在过度宽松的访问控制、私有数据暴露等权限相关问题，但缺乏系统研究。
  - 核心安全关切：Agent选择“更灵活、更强大”的高权限工具来完成本可由低权限工具完成的任务，扩大了错误、滥用或安全漏洞的“爆炸半径”。

## 2. 方法论

### 2.1 核心思想

- **问题形式化定义**：对任务实例 $x$，设可用工具集为 $T$，$p(t)$ 为工具权限级别，$S(x)$ 为足以完成 $x$ 的工具子集。在第 $i$ 步，若存在未被排除的低权限充分工具 $t_\ell \in U_i(x)$ 且 $p(t_\ell) < p(t_h)$，Agent却选择了 $t_h$，则判定为过度特权化选择。
- **两种行为模式**：
  - **激进选择（Aggressive Selection, PED=0）**：首次选择即直接使用高权限工具。
  - **过早升级（Premature Escalation, PED≥1）**：低权限工具遭遇短暂故障后，未充分探索其他低权限替代方案即升级至高权限工具。

### 2.2 评估指标

- **OPUR@k（Over-Privileged Tool Use Rate@k）**：在 $k$ 轮交互内，Agent在仍有低权限充分工具可用时使用高权限工具的比例。
- **PED（Pre-Escalation Exploration Depth）**：首次使用高权限工具前尝试的不同低权限工具数量；PED=0为激进选择，PED≥1为过早升级。

### 2.3 缓解方法：权限感知后训练（Privilege-Aware Post-Training）

- **训练数据构建**：
  - 遵循与评估基准相同的设计原则，构建独立的训练场景（含标准工具和风险工具）。
  - 标准工具可能返回短暂故障，要求Agent学会重试或探索替代方案而非立即升级。
  - **SFT数据**：由Qwen3.5-397B生成理想轨迹，包含权限分析推理过程（放在`<think>`标签内），展示如何区分权限级别、区分短暂故障与能力不足。
  - **RL数据**：使用与SFT和评估集不重叠的查询集，仅提供用户请求和工具环境，不提供监督轨迹，迫使模型通过交互和奖励学习。
- **训练流程**：
  1. **SFT阶段**：使用LoRA（rank=16, α=32）在1,994条多轮工具调用轨迹上微调2个epoch，学习率2×10⁻⁵，有效batch size=64。
  2. **RL阶段**：基于GRPO算法，在模拟多轮工具交互环境中优化策略。奖励函数编码有序偏好：使用标准工具完成 > 经充分低权限探索后升级 > 过早使用风险工具（惩罚最重）。KL正则化系数0.05，学习率1×10⁻⁶，240个rollout。
- **关键发现**：直接对预训练模型进行RL训练会导致优化不稳定（奖励坍塌至零），SFT初始化对RL成功至关重要。

## 3. 实验设计

### 3.1 基准构建（ToolPrivBench）

- **领域与风险分类**：
  - 8个应用领域：Coding、Infrastructure、Business、Database、Education、Government、Healthcare、Media。
  - 5种风险类型：Authority Escalation（权限提升）、Scope Expansion（范围扩展）、Temporal Persistence（时间持久性）、Safety Bypass（安全绕过）、Data Over-Exposure（数据过度暴露）。
- **构建流程**：
  1. 分析APIGen数据集中的约60K API调用，提取3,600个唯一工具，进行嵌入聚类（BGE-M3）得到20个语义簇。
  2. 按5级权限量表（L1-L5）标注工具，筛选高权限密度（L3-L5占比高）的簇作为领域种子。
  3. 为每个领域-风险组合合成用户任务和6个工具（3个标准+3个风险），所有工具独立充分。
  4. 自动化过滤（格式预检、描述中性检查、去重）→ 双模型自动验证（Gemini 2.5 Pro + GPT-5.2）→ 人工审核。
- **最终规模**：544个评估场景，分布广泛。

### 3.2 评估协议

- 每个场景提供6个工具（3低权限+3高权限），所有工具独立充分。
- 标准工具首次调用返回短暂故障（如连接错误），后续调用成功；风险工具首次调用即成功。
- 交互上限5轮，记录完整工具调用轨迹，计算OPUR@5和PED。

### 3.3 对比方法

- **基线评估**：11个主流LLM（Claude 4.6 Sonnet、GPT-5.2、GLM-5、Gemini 3 Flash、Kimi K2.5、DeepSeek-v3.2、Qwen3.5-397B、Grok 4.1 Fast、MiniMax-M2.7、LLaMA-3.1-8B、Qwen3-8B）。
- **安全对齐对比**：AgentAlign框架（Zhang et al., 2025b），测试其对OPUR的影响。
- **缓解对比**：
  - 提示工程（在系统提示中添加安全原则块，要求优先使用最小权限工具）。
  - 权限感知后训练（本文方法）。
- **通用能力评估**：MMLU、GSM8K、MetaTool基准，检验缓解方法是否损害通用能力。

## 6. 主要结论与发现

- **发现I**：过度特权化工具选择在主流LLM中普遍存在，但严重程度因模型而异。6/11模型OPUR超过30%，小模型（Qwen3-8B: 64.9%、LLaMA-3.1-8B: 55.9%）尤为严重；对齐前沿模型（Claude 4.6 Sonnet、GPT-5.2、GLM-5）低于10%但仍存在。
- **发现II**：工具故障显著加剧权限升级。GPT-5.2的PED=0仅5次，但PED=1达13次，PED=2激增至35次；类似模式在DeepSeek-v3.2、Grok 4.1 Fast、Kimi K2.5、Qwen系列中一致出现。
- **领域效应**：Infrastructure领域OPUR最高，Media和Database也较脆弱；Healthcare和Government对对齐前沿模型表现出较低升级倾向。
- **风险类型不对称**：Authority Escalation和Safety Bypass最常见，Scope Expansion最少见——模型偏好直接放宽执行约束的工具。
- **安全对齐失效**：AgentAlign大幅降低有害分数，但OPUR仅小幅下降（Ministral）甚至上升（Qwen），表明通用安全对齐无法可靠迁移至最小权限选择。
- **提示工程有限有效**：提示控制降低OPUR，但在多轮故障后效果减弱。
- **权限感知后训练有效**：在Qwen3-8B上OPUR从64.9%降至27.0%，Qwen3-4B-Thinking从66.0%降至18.9%，且通用能力保留率95.8%-100.2%。

## 7. 优点

- **问题定义清晰**：首次系统研究LLM Agent的过度特权化工具选择问题，填补工具选择研究与Agent安全之间的空白。
- **基准设计严谨**：通过双模型验证+人工审核确保工具独立充分，消除能力混淆；自动化过滤和修复流程提升构建效率。
- **评估协议合理**：同时捕捉激进选择和过早升级两种行为，OPUR和PED指标直观且互补。
- **缓解方法创新**：权限感知后训练结合SFT+RL，奖励函数设计精细（有序偏好，区分充分探索后的合理升级与过早升级）。
- **SFT初始化必要性发现**：揭示预训练模型直接进行RL会因缺乏工具调用先验知识导致优化坍塌，对Agent训练有实践指导意义。
- **全面性**：跨8领域×5风险类型，11个模型，消融实验和定性分析结合。

## 8. 不足与局限

- **模拟环境局限**：出于安全和可控性考虑，仅在模拟环境中评估，未使用真实生产工具或外部服务，无法覆盖部署Agent环境的全部复杂性。
- **工具独立充分假设**：所有工具均可独立完成任务，可能高估过度授权率；真实场景中工具能力部分重叠，需更长序列分析。
- **交互轮次限制**：5轮上限可能遗漏更长horizon中的权限升级行为，未来需扩展至多工具工作流和更长序列。
- **机制分析不足**：对过度特权化选择背后的认知机制（如不确定性下的决策偏差）缺乏深入探索。
- **训练规模有限**：SFT仅1,994条轨迹，RL仅1,199个提示，可能限制泛化能力。
- **模型覆盖偏倚**：缓解实验仅在Qwen3系列（4B/8B/4B-Thinking）上进行，未验证对其他模型家族的普适性。
- **风险类型分布不均**：Authority Escalation（139个场景）和Safety Bypass（116个）多于Scope Expansion（99个）和Temporal Persistence（91个），可能影响统计比较的公平性。
- **未涉及多工具组合场景**：仅评估单工具选择，未覆盖需要多工具协作的复杂工作流。
