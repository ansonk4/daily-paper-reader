## 摘要
AI安全代理日益依赖检索增强生成（RAG）来利用外部安全知识进行漏洞分析和攻击推理。这带来了一种新的风险：中毒的文档可能被转化为不正确的攻击行为。然而，此前关于RAG投毒的研究大多聚焦于问答场景中的答案篡改，而对采取行动的安全代理知之甚少。本文旨在通过针对真实挑战和AI代理精心构造的投毒内容来揭示这些特征。首先，我们展示了注入到公开风格安全知识源中的单篇被投毒文档——我们称之为“中毒手册”——如何改变基于RAG的AI安全代理的行为。在11个夺旗挑战、3个前沿大模型系列、2个模型代际和11个真实CVE的实验中，我们发现投毒内容的采纳是系统性的而非随机的。为了解释这一模式，我们引入了验证边界（VB），一种基于代理可用于反驳所检索声明的证据的三级经验分类。最后，我们评估了验证提示和多源检索，表明两者在存在更强证据时有效，但在稀疏证据和零日条件下效果减弱。

## Abstract
AI security agents increasingly rely on Retrieval-Augmented Generation (RAG) to use external security knowledge for vulnerability analysis and exploit reasoning. This creates a new risk: poisoned write-ups can be operationalized into incorrect exploit behavior. Yet, prior work on RAG poisoning has mostly studied answer corruption in QA settings, much less is known about action-taking security agents. This paper aims to reveal such characteristics with crafted poisons about real-world challenges and AI agents. First, we demonstrate how a crafted single poisoned write-up injected into public-style security knowledge sources which we denote as Poisoned Playbooks, alters the behavior of RAG-based AI security agents. Across 11 CTF challenges, 3 frontier LLM families, 2 model generations, and 11 real-world CVEs, we find that poison adoption is systematic rather than random. To explain this pattern, we introduce the Verification Boundary (VB), a 3-level empirical classification based on what evidence the agent can use to refute a retrieved claim. Finally, we evaluate verification prompting and multi-source retrieval, showing that both help when stronger evidence exists, but weaken under sparse-evidence and zero-day conditions.

## 1. 核心问题与整体含义

- **研究背景**：AI安全代理（如漏洞分析、渗透测试代理）越来越依赖检索增强生成（RAG）来获取外部安全知识（如CTF write-up、exploit数据库、博客等）。这些知识源时效性强但策展薄弱，容易被操纵。

- **核心贡献**：
  - 提出"**中毒手册**"（Poisoned Playbooks）作为研究载体——模拟真实安全文档风格的投毒 write-up

## 2. 方法论

### 2.1 核心思想

- **威胁模型**：攻击者预先将投毒文档注入公共安全知识源（GitHub、博客、论坛等），下游代理通过正常RAG检索获取这些文档并将其纳入推理。攻击者无需运行时目标系统。
- **中毒手册设计**：
  - 每篇围绕一个**虚假声明**设计，通常涉及外部依赖、环境假设或不可从即时工件验证的利用条件
  - 模拟真实 write-up 风格：第一人称叙事、伪造定量证据、"先承认后反驳"结构、完整替代利用路径
  - 制作成本：每篇约30-60分钟，但可1:N规模影响所有下游代理

### 2.2 验证边界（Verification Boundary, VB）

三级经验分类，基于代理可用于反驳检索声明的证据类型：

| 级别 | 名称 | 定义 | 采纳行为 |
|------|------|------|----------|
| **L1** | Code-Verifiable | 可用本地工件（源码、配置等）验证 | 几乎零采纳 |
| **L2** | Knowledge-Verifiable | 本地工件无法判断，但模型参数知识可能拒绝 | 模型依赖性强 |
| **L3** | Runtime-Dependent | 需要外部依赖/运行时信息，超出代理可观察上下文 | 几乎普遍采纳 |

形式化定义：

$$L(c; S, K) = \begin{cases} L1 & \text{if } \text{verifiable}(c, S), \\ L2 & \text{if } \neg\text{verifiable}(c, S) \land \text{assessable}(c, K), \\ L3 & \text{otherwise}. \end{cases}$$

其中 $S$ 为上下文可观察工件，$K$ 为模型参数知识。

### 2.3 缓解策略

- **验证提示**（Verification Prompting）：显式指示代理区分可验证声明与运行时依赖声明
- **多源检索**（Multi-Source Retrieval）：检索并比较多份候选源，降低单篇投毒文档的影响

## 6. 主要结论与发现

1. **投毒采纳是系统性而非随机的**：单篇投毒文档即可系统性改变代理行为，但模式取决于证据可用性
2. **验证边界（VB）是核心解释变量**：
   - L1（代码可验证）：近零采纳
   - L2（知识可验证）：模型依赖性强，新模型可能拒绝
   - L3（运行时依赖）：跨所有模型和代际普遍采纳
3. **模型改进的局限性**：模型升级可扩展L2区域，但不改变L3的结构性脆弱性——代理无法反驳其无法观察的内容
4. **信息垄断是强预测因子**：全信息垄断下L3声明一致采纳；强竞争证据下降至零
5. **稀疏证据/零日场景最脆弱**：早期披露阶段缺乏权威反证，投毒最有效
6. **三种行为结果模式**：
   - **假阴性诱导**：承认漏洞存在但论证不可行，导致忽略真实漏洞
   - **目标转移**：重定向至不存在的利用路径
   - **推理腐败**：保留正确漏洞类但破坏利用逻辑
7. **缓解策略有条件有效**：验证提示和多源检索在强证据下有效，稀疏证据/零日下效果减弱
