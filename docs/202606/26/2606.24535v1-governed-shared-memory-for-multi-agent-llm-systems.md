## 摘要
多智能体大语言模型环境需要强大的共享知识管理机制。本文形式化了舰队内存问题，并识别出四种基础故障模式：未授权泄露、陈旧传播、矛盾持续和溯源崩溃。为解决这些问题，我们定义了显式的系统级原语：作用域检索、时间超链、溯源跟踪和策略治理的内存传播。这些原语在MemClaw——一个生产级多租户内存服务——中实现，并通过ArgusFleet（一种可复现的测试框架）在四个治理维度上进行评估。该研究未采用基线对比，而是对生产环境中的实时服务进行测量，强调真实架构洞见与负面结果。关键评估结果：溯源——成功重建了100%深度为四的派生链路，每跳延迟低于秒级，且写入者身份正确。传播——实现了舰队内高可见性，且零跨舰队泄露。在强写入模式下，写入到可见的延迟被优化至单次搜索往返。发现的生产架构问题：非对称作用域执行——租户隔离有效，但子租户作用域最初在针对智能体作用域凭证的直接按ID获取请求中被绕过（在研究期间已披露并修复）。流水线排序冲突——虽然矛盾超链适用于已提交的写入，但同步的近重复门控可能在异步矛盾检测器评估之前过早地拒绝矛盾写入。结论：仅靠长上下文检索不足以满足生产多智能体内存的需求。受治理的共享内存需要显式的系统级抽象，而实时评估对于暴露设计阶段遗漏的执行和流水线排序故障至关重要。

## Abstract
Multi-agent LLM environments require robust mechanisms for shared knowledge management. This paper formalizes the fleet-memory problem and identifies four foundational failure modes: unauthorized leakage, stale propagation, contradiction persistence, and provenance collapse. To address these, we define explicit systems-level primitives: scoped retrieval, temporal supersession, provenance tracking, and policy-governed memory propagation. These primitives are implemented in MemClaw, a production multi-tenant memory service, and evaluated via ArgusFleet, a reproducible harness testing four governance dimensions. Rather than a baseline comparison, this study measures a live production service, emphasizing real-world architectural insights and negative results. Key Evaluation Results Provenance: Successfully reconstructed 100% of depth-four derivation chains with correct writer identity at sub-second per-hop latency. Propagation: Demonstrated high intra-fleet visibility with zero cross-fleet leakage. Under strong write mode, write-to-visible latency was optimized to a single search round-trip. Production Architectural Issues Discovered Asymmetric Scope Enforcement: Tenant isolation held, but sub-tenant scope was initially bypassed on direct GET-by-id requests for agent-scoped credentials (disclosed and remediated during the study). Pipeline Ordering Conflict: While contradiction supersession works for admitted writes, a synchronous near-duplicate gate can prematurely reject contradictory writes before the asynchronous contradiction detector can evaluate them. Conclusion: Long-context retrieval alone is insufficient for production multi-agent memory. Governed shared memory demands explicit systems-level abstractions, and live evaluation is vital to expose enforcement and pipeline-ordering failures missed by design-only treatments.

## 2. 方法论：核心思想与关键技术细节

### 2.1 舰队内存形式化定义

- 将舰队内存系统建模为五元组：$F = (A, M, G, P, T)$
  - $A$：交互智能体集合
  - $M$：共享内存基底
  - $G$：治理与策略层
  - $P$：溯源元数据
  - $T$：时间排序与超链语义
- 写入操作定义为五元组：$w_i = (a_i, c_i, s_i, t_i, p_i)$，分别对应写入智能体、内容、可见性作用域、时间排序和溯源元数据。
- 检索操作定义为：$r_j = f(q_j, a_j, g_j, t_j)$，检索正确性不仅依赖语义相似度，还依赖**授权、状态新鲜度、矛盾解决、溯源有效性和同步保证**。

### 2.2 四种故障模式

| 故障模式 | 含义 | 示例 |
|---|---|---|
| 未授权泄露（Unauthorized Leakage） | 智能体检索到超出其授权作用域的内存 | 客服智能体检索到仅给财务智能体的账单备注 |
| 陈旧传播（Stale Propagation） | 内存更新未能在智能体间正确同步 | 一个智能体更新了地址，另一个仍读取旧状态 |
| 矛盾持续（Contradiction Persistence） | 冲突内存共存且无解决机制 | 两个不兼容的用户偏好同时可被检索 |
| 溯源崩塌（Provenance Collapse） | 检索到的内存无法追溯到来源 | 无法知道某条事实由谁写入 |

### 2.3 四项核心架构原语

1. **作用域检索（Scoped Retrieval）**：定义四级内存作用域——智能体本地（Agent-local）、团队共享（Team-shared）、租户全局（Tenant-global）、受限策略（Restrict）。检索操作必须满足治理策略，形式化表示为不变式 **Inv-Scope**：$w_i \in r_j \Rightarrow \text{auth}(a_j, s_i, G)$。
2. **时间超链（Temporal Supersession）**：内存被视为演化状态而非不可变追加。每条内存携带创建时间戳、超链引用、矛盾标记和置信度状态。矛盾写入通过时间排序和策略感知的解决规则处理。
3. **溯源图（Provenance Graph）**：每条内存对象存储写入者身份、来源系统、派生历史和修改谱系，支持审计、调试、可追溯性和合规验证。
4. **策略治理传播（Policy-Governed Propagation）**：检索流程分为五步——语义候选生成 → 策略过滤 → 时间解决 → 溯源增强 → 排序交付。跨智能体内存重新分配受信任级别门控（trust_level ≥ 3）。

### 2.4 参考实现：MemClaw

- 多租户生产级记忆服务，通过 REST API 暴露治理能力。
- 认证使用 `mc_` 前缀的 API 密钥，凭证类型（租户级 vs 智能体级）绑定在凭证记录上。
- 智能体级凭证由网关解析调用者身份并注入 `X-Agent-ID`（客户端提供的被剥离），租户边界在所有路径上强制执行。
- 核心操作包括：写入（POST /memories）、语义搜索（POST /search）、实体物化（POST /entities/upsert）、按ID检索（GET /memories/{id}）、内存重新分配（POST /redistribute）。
- 写入模式：`fast`、`strong`、`auto`、`stm`，不同模式在一致性和延迟间权衡。

### 2.5 评估框架：ArgusFleet

- 开源 Python 3.12 评估工具包，包含四个实验、异步 REST 客户端、类型化领域模型、结构化事件日志和可复现报告管线。
- 生成 JSONL 事件追踪、CSV 摘要和 Matplotlib 图表，所有数字从同一规范事件流重新生成，无需重新请求服务器即可复现分析。
- 确定性：每个工作负载设置种子，计划结构可复现，每次运行内容用唯一随机数盐化。
- 异步执行：基于 httpx 和 tenacity，带指数退避处理瞬态错误、5xx 响应和 429 速率限制。

## 6. 主要结论与发现

### 6.1 正面结果

- **溯源（Provenance）**：所有 50 条深度为 4 的派生链路**100% 正确重建**，每跳获取延迟 p50 = 291ms，p95 = 491ms，p99 = 1,076ms。写入者身份在存储往返中完整保留。
- **传播（Propagation）**：舰队内可见性 117/120 = 0.975（95% Wilson CI [0.929, 0.991]），跨舰队泄露率 0/80（95% Wilson CI [0.000, 0.046]）。强写模式下写入-可见窗口 p50 = 0.83s，p95 = 1.63s——本质上是**一次搜索往返**，一致性成本在写入时支付而非读取时。
- **矛盾（Contradiction）**：条件于两次写入均被接受时，超链正确率 90/90 = 100%。整体检测率 98/200 = 0.490，差距完全由写入准入效应解释（同步去重门控拒绝了 206/400 次写入）。

### 6.2 负面发现（生产架构问题）

- **非对称作用域执行**：租户隔离在所有路径上强制执行，但子租户作用域（舰队/智能体）在 GET-by-id 路径上最初**未被强制执行**——处理程序仅检查 `enforce_readable_tenant` 和 `memory.tenant_id == tenant_id`，完全忽略了调用者的舰队/智能体身份、行的可见性作用域和信任级别。这是一个**预测 → 测量 → 修复循环**的形式化预测验证。
- **去重使矛盾检测饥饿**：同步近重复门控（pre-commit, 同步）可能在异步矛盾检测器（post-commit, 异步）评估之前拒绝矛盾写入。自然措辞的矛盾（"X 是 A" vs "X 是 B"）文本几乎相同，最可能被门控拒绝——检测器存在的目的恰恰是解决这些写入。
- **跨智能体传播特权门控**：`POST /memories/redistribute` 要求 trust_level ≥ 3，将舰队内正常传播与管理性迁移混为一谈。

### 6.3 方法论发现

- **一致性成本在写入时支付，非读取时**：强写模式下强制的丰富化是同步的，读取端无最终一致性尾部，但写入延迟在负载下出现重尾（p99 = 19,319ms）。
- **朴素批量探测调度会严重高估可见性延迟**：早期版本报告数十秒的 p50，实为客户端调度伪影而非服务器行为。

## 7. 优点

- **问题定义的前沿性与重要性**：首次将多智能体 LLM 共享内存形式化为**受治理的分布式系统问题**，填补了单智能体记忆系统与多智能体需求之间的鸿沟。
- **形式化驱动的预测-测量-修复循环**：论文不仅识别了故障模式，还通过形式化（Inv-Scope 不变式）**预测了具体违反**，然后通过实证测量验证，最终披露并修复——展示了形式化方法在真实系统中的价值。
- **负面结果作为核心贡献**：论文明确将负面结果（子租户绕过、流水线排序冲突）视为贡献，这在学术出版中罕见且诚实，对社区具有极高价值。
- **可复现评估框架**：ArgusFleet 完全开源，所有 JSONL 事件追踪公开提交，支持确定性复现和重分析，评估设计严谨。
- **多维度双信号测量**：泄露率（安全信号）和遗漏率（可用性信号），提供更完整的治理画像。
- **方法论反思**：对朴素批量探测调度会高估可见性延迟的反思，纠正了早期版本的错误测量，展示了严谨的自我批判精神。
- **生产级实证**：直接对生产服务进行黑盒测量，而非实验室原型，结果更具现实参考价值。

## 8. 不足与局限

- **自评估偏差**：ArgusFleet 和 MemClaw 由同一团队构建，非独立审计。虽然通过公开追踪缓解，但独立复现会更有说服力。
- **单租户限制**：所有测量来自单一 MemClaw 新配置租户，**跨租户泄漏探测表面未测试**——这是最强治理声明却无法在当前配置中验证。
- **无基线对比**：未与长上下文配置或其他记忆实现对比，无法量化相对改进幅度。
- **工作负载规模有限**：每个实验 $N = 200$，泄露率 Wilson 区间在 164/164 时为 [0.977, 1.000]，跨舰队泄露率 0/80 上界约 0.046。更强声明需要数量级更多的探测和对抗性构造。
- **窗口阶段规模小**：写入-可见窗口在 250ms 粒度下仅测量 8 个事实，无法解析亚 250ms 值，更大规模多区域扫描会增强尾部估计。
- **聚焦探测非穷举**：批量泄露率使用租户级密钥收集（按设计为租户级），子租户问题通过智能体级凭证的聚焦探测解决。原始观察是小规模聚焦探测，修复后复探较大（96 个探测），但更完整的对抗性矩阵（多智能体、多作用域、多可见性级别）会增强鲁棒性。
- **陈旧性测量部分**：矛盾实验建立了陈旧解析侧（旧行标记为 outdated，新行携带 supersedes_id），但陈旧读取检查依赖语义搜索，其丰富化窗口现在直接测量约 1s——在 20s 稳定期内，0 更多反映正确解析而非搜索延迟。
- **去重与矛盾检测交互未完全解决**：论文明确指出了流水线排序问题，但未提出或验证修复方案，仅提出两种可能的重排策略。
