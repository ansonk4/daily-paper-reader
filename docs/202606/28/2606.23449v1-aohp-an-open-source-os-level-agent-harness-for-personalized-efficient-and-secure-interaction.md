## Abstract
AI agents are driving a new software paradigm, with the ability to autonomously call tools, extract information, manage memory, and complete tasks that span applications and data sources. Most existing end-user operating systems, however, are designed for application-centric workflows and offer little native support for AI agents. This mismatch limits the wider adoption of agents and leads to execution overhead and safety risks when running agents on conventional systems. While the concept of agent-native operating systems is emerging, the research community lacks an open testbed to explore the architectural primitives desired for agent-mediated interaction. We present AOHP (Android Open Harness Project), an OS-level agent harness built on the Android Open Source Project (AOSP). The core design principle of AOHP is to treat agents as first-class OS actors, enabling adaptive user interfaces and agent-friendly runtime environments. AOHP preserves the mature Android software and hardware ecosystem while introducing three agent-oriented system mechanisms: personalized service composition, efficient agent interfaces, and secure information flow. Based on preliminary experiments on challenging tasks covering key capabilities of OS agents, AOHP shows clear advantages in task completion (+21.12% completion rate), execution cost (-51.55% token cost), and security-policy compliance.

# AOHP: An Open-Source OS-Level Agent Harness for Personalized, Efficient and Secure Interaction

## 1. Core Problem and Overall Meaning

- **Problem**: Conventional end-user operating systems (e.g., stock Android) are designed for app-centric workflows with fixed GUIs, foreground-serialised app lifecycles, and app-boundary permission models. This creates a structural mismatch with AI agents that need to observe system state, invoke tools across apps, run parallel workflows, and retain long-lived context. The mismatch causes:
  - High execution overhead (slow, token-heavy GUI navigation).
  - Security gaps (private plaintext can leak across agent tool calls and app boundaries).
  - No open testbed for exploring "agent-native" OS primitives.
- **Motivation**: As AI agents become a major workload, the OS should treat them as first-class actors, proactively compose services around user intent, expose efficient non-GUI interfaces, and enforce fine-grained information-flow control across app and tool boundaries.
- **Overall meaning**: AOHP (Android Open Harness Project) is an open-source, OS-level agent harness built on AOSP that redesigns Android to be "agent-native" while preserving hardware support, app compatibility, and the existing ecosystem. It introduces three agent-oriented mechanisms: personalized service composition, efficient agent interfaces, and secure information flow.

## 2. Proposed Methodology

### 2.1 Core Design Principles
- **On-demand UI composition**: The OS agent discovers and recomposes capabilities across providers into personalised, task-level interaction surfaces driven by user intent, context, and state.
- **Compatible, efficient, interface-neutral agent-service interaction**: A unified substrate preserves legacy Android interaction while supporting agent-oriented service logic with lower overhead and higher accuracy.
- **Sensitive data isolated by default**: Agents never observe private plaintext by default; the system enforces fine-grained information-flow tracking, approval, and audit.

### 2.2 System Architecture (Four Layers)
1. **Android ecosystem layer** – unchanged apps, system services, hardware, and platform APIs as the compatibility base.
2. **Unified interaction interface layer** – normalises traditional and agent interfaces into four invocation modes:
   - API
   - CLI
   - Structured UI
   - Rendered GUI (fallback)
   Agents select a compact symbolic path when available and fall back to visual operation only when necessary.
3. **AOHP capabilities layer** – reorganises services and agent functions into:
   - **System memory**: preferences, task state, histories, policy decisions stored outside any single app.
   - **Skills**: reusable service capabilities and execution routines.
   - **UI utilities**: support construction of generated entrances and task-specific interaction surfaces.
4. **Personalized service composition layer** – assembles the above capabilities into user-facing, personalised service entrances tailored to the current task.

### 2.3 Cross-Layer Mechanisms
- **Efficient agent interfaces**: optimise how agents access system resources, app state, files, events, and isolated execution substrates.
- **Secure information flow**: sandboxes sensitive values via information-flow tracking, sanitisation, vault execution, and audit.

### 2.4 Personalized Service Composition
- **Generated service entrances**: user-facing shells backed by OS-managed service composition. Each entrance contains:
  - **Task schema**: what the user is trying to accomplish (e.g., "compare shoes under a budget").
  - **Service graph**: mapping of the task to concrete service capabilities.
  - **Presentation policy**: which intermediate results are shown to the user vs. kept agent-internal.
- **Capability discovery and composition**: AOHP discovers capabilities across API, CLI, and GUI channels. Each capability is represented with input/output schemas, preconditions, side effects, and policy labels. The OS agent composes them into higher-level workflows, constrained by policy (e.g., product search can be parallelised; purchase submission requires explicit confirmation).
- **Cross-service personalization via system memory**:
  - **Persistent profile memory**: stable preferences.
  - **Task-local memory**: temporary state (candidates, partial forms).
  - **Sensitive memory**: private values stored behind sandbox indices.
  Memory is OS-mediated so personalization can be shared, audited, and revoked without depending on each app's private data model.

### 2.5 Efficient Agent Interfaces
- **Parallel background interaction**: Lightweight virtual displays decouple execution from the physical screen, allowing agents to run waiting-heavy or independent workflows in the background without preempting the foreground session.
- **Agent-aware UI enhancement**: GUIs are abstracted into structured representations with lower redundancy and richer semantics, retaining rendered GUI fallback for visual components.
- **Native sandbox runtime**: An OS-managed sandbox runtime can be created and reset as an execution surface independent of app-facing interfaces. Agents execute code, process data, and host long-running services inside the sandbox, returning structured results to the task context.
- **Unified file shortcut**: Files are treated as first-class task objects at the OS boundary. GUI interactions that affect storage are reflected back as structured file observations; agents can hand a resolved artifact to another interface either programmatically or by launching the appropriate system UI flow. This unifies file-producing GUI steps and file-consuming programmatic steps into a single cross-app data plane.
- **Event stream abstraction**: Agents can subscribe, process, and unsubscribe from continuous data sources. Two supported stream types:
  - **Dynamic notification capture**: A notification buffer intercepts and retains transient system events (toasts, pop-ups, push notifications) so agents do not miss critical UI context.
  - **Sensor streaming access**: Hardware sensor data (accelerometer, gyroscope, microphone, camera) is streamed so agents can process real-time physical states without repeated polling.

### 2.6 Secure Information Flow
- **Policy enforcement over runtime data use**: For each sensitive operation, the policy layer evaluates data source, requested purpose, destination, action sensitivity, and approval state. Ordinary non-sensitive flows proceed; private-data transfers and state-changing actions require consent or are blocked. Approval prompts explain source, purpose, destination, and downstream effect.
- **Sensitive source sanitisation**: Before sensitive content enters the agent context, AOHP replaces plaintext with typed placeholders such as `<payment-card:uuid>` or `<home-address:uuid>`. A data vault stores sensitive values behind opaque identifiers. Supported sources include application pages, files, event streams, API responses, system memory, and user interactions. Developer-provided annotations can mark sensitive fields; otherwise, AOHP applies conservative detection rules.
- **Trusted vault and execution**: When an agent needs to operate on sensitive information, it submits an intent to a trusted vault executor using the placeholder. The executor checks policy, obtains user approval when necessary, and performs operations (formatting, comparison, validation, composition) inside the trusted environment. If the result is still sensitive, the executor returns another placeholder. The executor also mediates sensitive transfers to external interfaces (e.g., filling an approved GUI field directly or substituting plaintext at the API/CLI boundary).
- **Data-flow taint tracking**: Once sensitive data enters AOHP, it is associated with taint metadata that follows the value through copying, transformation, composition, and transfer. At system exits and policy-relevant boundaries, AOHP checks tainted data before it is displayed, stored, submitted, or transmitted. The taint path provides an audit trail explaining which source reached which sink through which task steps.

## 3. Experimental Design

- **Agent under test**: OpenClaw agent.
- **Baselines**:
  - **Stock Android (app-centric baseline)**: the agent receives conventional GUI observations and operates through ordinary app interactions.
  - **AOHP (agent-native setting)**: the same agent additionally uses generated service entrances, structured UI observations, system CLIs/APIs, virtual execution, and sandboxed information flow.
- **Benchmark**: 30 real-world mobile tasks built exclusively from real applications, spanning six categories (5 tasks each):
  1. GUI operation
  2. Non-GUI operation
  3. Event capture
  4. Multi-source information retrieval
  5. Memory management (two-stage: stage A executes the task, stage B asks memory-related questions)
  6. Hybrid (capability composition into longer cross-capability workflows)
- **Metrics**:
  - Task completion (checkpoint-weighted completion rate)
  - Success count
  - Tool calls
  - Duration
  - Token usage
  - LLM requests
  - Security-policy behaviour
- **Security evaluation**: A purpose-built annotated payment application exposing account information, payment credentials, transfer inputs, confirmation actions, invoice files, and transaction-related event streams. Case-driven tests exercise source sanitisation, sink and action mediation, vault-token use, taint propagation, file handling, event handling, and fail-closed behaviour.

## 5. Number and Sufficiency of Experiments

- **Scale**: 30 benchmark tasks run under two settings (stock Android and AOHP). Security evaluation uses 5 annotated test cases.
- **Sufficiency and fairness**:
  - The same agent (OpenClaw) is used in both settings, isolating the effect of the OS substrate.
  - Efficiency analysis is restricted to the 11 tasks solved completely by both settings, removing the confound of unequal task difficulty and isolating the cost of producing the same successful outcome.
  - Tasks are checkpoint-weighted, so partially completed tasks still earn partial credit, reducing binary scoring bias.
  - The benchmark covers six capability categories including hybrid composition, which matches AOHP's target setting.
- **Limitations in sufficiency**:
  - Only one agent (OpenClaw) is evaluated; results may not generalise to all agent architectures.
  - The security evaluation is a case study on a single annotated payment app, not a large-scale adversarial benchmark.
  - No direct comparison with other agent-native OS proposals or other agent frameworks.

## 6. Main Conclusions and Findings

- **Task completion**: AOHP raises the average completion rate from 54.44% (stock Android) to 75.56%, a +21.12-point gain. AOHP fully solves 20 tasks (vs. 13) and partially completes 5 (vs. 7).
- **Efficiency** (on the 11 commonly solved tasks):
  - Tool calls: 129 vs. 233 (−44.64%).
  - Duration: 18.93 min vs. 33.94 min (−44.21%).
  - Tokens: 3.44M vs. 7.10M (−51.55%).
  - LLM requests: 143 vs. 273 (−47.62%).
  - Input tokens down 51.50%; output tokens down 57.48%.
- **Security**: In the prototype evaluation on the annotated payment app, AOHP enforced all five tested security cases (sensitive display, ordinary actions, sensitive actions, unsupported access fail-closed, sensitive events) as expected.
- **Overall**: AOHP demonstrates that treating agents as first-class OS actors yields large gains in completion, efficiency, and security while preserving Android compatibility.

## 7. Strengths in Method or Experimental Design

- **OS-level, not just agent-level contribution**: AOHP redesigns the OS substrate itself rather than proposing a stronger agent policy, addressing a gap in the research community.
- **Preservation of ecosystem**: Built on AOSP, it retains hardware support, open-source framework, and app compatibility.
- **Three coherent, well-motivated mechanisms**: Personalized service composition, efficient agent interfaces, and secure information flow each address a distinct mismatch between app-centric OSes and agent workflows.
- **Unified interaction model**: Four invocation modes (API, CLI, Structured UI, Rendered GUI) let agents use the most efficient path available.
- **Fair efficiency analysis**: Restricting cost comparison to tasks solved completely by both settings avoids confounding difficulty with cost.
- **Granular, checkpoint-weighted scoring**: Partial credit reduces binary bias in task completion.
- **Security by design**: Combines source sanitisation, trusted vault execution, taint tracking, and audit trails, moving enforcement to the OS boundary.
- **Open-source**: Provides an open testbed for future agent-native OS research.

## 8. Weaknesses and Limitations

- **Compatibility coverage**: Apps with custom rendering, anti-automation logic, or undocumented behaviour may not be fully supported. The paper itself notes the need for stronger structured UI extraction, predictable rendered-GUI fallback, and clearer compatibility guidelines.
- **Capability discovery dependency**: AOHP depends on accurate service descriptions, side-effect labels, and policy metadata. Legacy apps may require manual annotation; automatic capability inference is not yet realised.
- **Resource scheduling not fully addressed**: Background execution via virtual displays and sandbox runtimes must still respect mobile resource, thermal, and memory limits. The paper acknowledges the need for scheduling policies.
- **Policy usability**: Fine-grained information-flow control requires user approvals that must be understandable and not excessively interruptive. Approval UI, trace review, and policy explanations need further work.
- **Limited agent and adversarial evaluation**: Only one agent (OpenClaw) is tested; security evaluation is a case study on a single annotated app rather than a broad adversarial benchmark.
- **No comparison with other agent-native systems**: The paper does not empirically compare against other emerging agent-native OS proposals or alternative agent frameworks.
- **Prototype maturity**: Results are described as preliminary; a fuller prototype is needed to validate scheduling, compatibility at scale, and real-world deployment constraints.
