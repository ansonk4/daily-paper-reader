Abstract —Recent advances in large language models (LLMs) have enabled vibe coding, an emerging software development paradigm in which users create applications primarily through natural-language interactions with AI agents. Due to its low barrier to entry, vibe coding is rapidly gaining adoption in practice. Unlike conventional AI-assisted programming, where developers remain responsible for implementation and code review, vibe coding delegates a substantial portion of develop-ment to AI systems. This shift raises a fundamental question: how (in)secure are applications developed through vibe coding? In this paper, we conduct a systematic study of the security of vibe-coded applications. We collect a large corpus of real-world applications developed using popular AI agents (e.g., Claude Code) and design a vulnerability analysis framework that combines agent-assisted code auditing with human validation. Using this framework, we examine the prevalence, severity, and root causes of vulnerabilities in the deployed vibe-coded applications. Our study reveals several key findings: (1) vibe-coded applications exhibit recurring vulnerability patterns that differ from those commonly observed in conventional software development workflows, including placeholder logic, unfiltered input, and secret exposure; (2) these vulnerabilities arise from systematic limitations of AI agents throughout the vibe-coding lifecycle, such as memory loss, locally optimized objectives and insufficient security knowledge; and (3) while advances in LLM capabilities and improved prompting strategies can reduce the incidence of vulnerabilities, they do not eliminate the underlying security risks. Overall, our study provides an empirical understanding of the security landscape of vibe-coded applications and lays the groundwork for addressing the security challenges introduced by the growing delegation of software development to AI systems. 

## 1. Introduction 

Recent advances in large language models (LLMs) are fundamentally reshaping how software is developed [1]. What began as code-completion assistants [2], [3] ca-pable of generating individual functions or features has rapidly evolved into increasingly autonomous AI coding agents [4], [5] that design, implement, and deploy com-plete, runnable software applications from natural language instructions. This evolution has given rise to a new devel-opment paradigm commonly referred to as vibe coding [6], [7]. In this paradigm, users describe high-level requirements in natural language and rely on AI coding agents or vibe-coding platforms to generate entire software applications from scratch, with minimal human intervention. Since its introduction, vibe coding has rapidly gained traction across the software development ecosystem, evolv-ing from an experimental concept into a production reality. The appeal of vibe coding lies in substantially lowering the technical barrier to application creation by abstracting away much of the implementation process. As a result, individ-uals with limited programming experience can now create and deploy applications that previously required significant development expertise. Platforms such as Claude Code [5] and Lovable [4] have attracted millions of users and enabled the creation of applications at an unprecedented scale. For example, Lovable reported more than eight million users and over 100,000 new applications created every day as of early 2026 [8]. Moreover, more than 10% of the applications are deployed as live, publicly accessible systems, as we later show in our study. With its popularity, vibe coding raises a critical concern: 

how (in)secure are vibe-coded applications in practice? 

Unlike traditional software development, vibe coding also allows non-expert individuals to create and directly de-ploy applications without fully understanding the underlying implementation. Therefore, security-critical decisions made by AI coding agents may remain unaudited. While mod-ern agents are increasingly designed to incorporate secure programming practices, such as authentication mechanisms, input validation, and common security safeguards [9], [10], practitioners continue to report applications that expose sensitive credentials, implement flawed access-control logic, construct vulnerable database queries, or deploy insecure configurations [11], [12], [13]. More importantly, as vibe coding scales, a single vulnerability generated by coding agents may be replicated across thousands of applications. Although a growing body of work has examined the se-curity of LLM-generated code [7], [14], [15], [16], existing studies fall short of capturing the security risks of vibe-coded applications. Prior research has primarily focused on evaluating the security of AI-assisted programming. In this paradigm, AI coding agents operate within established codebases and are tasked with completing specific develop-

ment activities, such as fixing bugs or implementing new features [17], [18], where software development is largely reduced to mere code-filling tasks. This paradigm fundamen-tally differs from vibe coding, where AI coding agents are instructed to create entire software applications from scratch. In vibe-coded applications, security vulnerabilities may arise not only from insecure code generation, but also from flawed architectural choices, poorly defined trust boundaries, and insecure integrations among components selected and assembled by LLM agents. Therefore, findings from prior studies on LLM-generated code cannot be generalized to characterize the security of vibe-coded applications. 

Our Work. To this end, we conduct the first systematic study of the security of vibe-coded applications. We con-struct VIBE APPS , a large-scale corpus of 10,517 open-source vibe-coded applications. From this corpus, we ran-domly sample 200 deployed applications that are publicly accessible and use them as the target of our security anal-ysis. We then develop a human-in-the-loop vulnerability-discovery workflow, combining agent-assisted candidate dis-covery with exploitability triage and manual validation. Us-ing this workflow, we identify 1,471 vulnerabilities across the sampled applications, forming our vulnerability dataset VIBE VULNS . To the best of our knowledge, this is the first large-scale dataset for studying the security of real-world vibe-coded applications. We seek to answer the following research questions:  

> •

RQ.1: What is the landscape of real-world vibe-coded applications? We characterize the ecosystem of vibe-coded applications, including their scales, development characteristics, and application categories, providing a foundation for subsequent security analysis.  

> •

RQ.2: How prevalent and severe are vulnerabilities in deployed vibe-coded applications? We measure the prevalence, severity, and distribution of their vulnerabili-ties to assess their overall security posture.  

> •

RQ.3: What factors contribute to these vulnerabilities? 

We investigate the root causes of identified vulnerabilities and uncover recurring factors that lead to vulnerabilities.  

> •

RQ.4: What conditions can mitigate the vulnerabilities of vibe-coded applications? We examine whether ad-vanced agent harnesses, stronger foundation models, and carefully-crafted user prompts can mitigate vulnerabilities in generated applications. Our key findings are summarized as follows. For RQ.1, we find that vibe coding is often used to develop large-scale applications within a short time frame. Moreover, web applications form the dominant application category, and more than 12% of these applications are publicly deployed. For RQ.2, we find that deployed vibe-coded applications commonly contain security risks, including broken access control, cryptographic failures, injection, and secret expo-sure. For RQ.3, we show that these vulnerabilities arise from recurring failure modes in the vibe-coding lifecycle— memory defects, objective defects, and knowledge defects— rather than from isolated code-generation mistakes alone. For RQ.4, nearly all investigated conditions mitigate secu-rity vulnerabilities to varying degrees, where lightweight prompt engineering techniques with production and self-check requests are the most effective. Nevertheless, mit-igating vulnerabilities introduced by different root causes requires different mitigation strategies. 

Contributions. We make the following contributions:  

> •

We construct the first large-scale dataset of 10,517 real-world vibe-coded applications. The dataset captures a diverse ecosystem of AI-generated software and serves as a foundation for future research in this domain.  

> •

Leveraging this dataset, we present the first large-scale empirical study of security vulnerabilities in vibe-coded applications, providing a systematic characterization of the prevalence, distribution, and patterns of security risks in vibe-coding software development.  

> •

Our findings provide actionable guidance for securing vibe-coded applications and mitigating their vulnerabil-ities. Our results highlight that securing the vibe-coded applications requires improvements throughout the whole development lifecycle, moving beyond focusing on code-generation errors alone. 

## 2. Background and Related Work 

AI-Assisted Programming and Vibe Coding. LLMs are becoming a standard tool in modern software development. Current coding assistants support a wide range of tasks, including code completion, bug fixing, and even repository-level modification. We refer to this development paradigm as 

AI-assisted programming [1]: developers remain responsible for application design, implementation decisions, and code review, while LLMs provide code-level assistance. Industry-grade tools such as GitHub Copilot are now used by mil-lions of developers [19], leading to a substantial fraction of code in production repositories being generated with LLM assistance [20]. Building on these advances, vibe coding has emerged as an AI-led development paradigm. Here, users describe desired requirements for applications in natural language, and LLM agents autonomously design, implement, and it-eratively refine code to produce complete, runnable appli-cations with little or no human inspection [21]. Unlike AI-assisted programming, where humans remain the primary authors and decision-makers, vibe coding shifts substantial authorship and oversight toward AI agents, leaving hu-mans primarily specifying high-level goals and providing feedback. This distinction is central to our study because security-relevant decisions in vibe-coded applications may be made as part of an AI-led development workflow rather than through explicit human design and review. 

Security of AI-Assisted Programming. In recent years, the security of AI-generated code has attracted growing attention. However, most existing studies focus on AI-assisted programming , where LLMs generate code snip-pets or modifications integrated into existing codebases. 2Early work, including A SLEEP [14] and S ECODE PLT [22], evaluated the security of AI-generated code on synthetic CWE-based tasks involving individual functions or single-file programs. More recent studies, such as A.S.E [15], S E-

> CURE

VIBE BENCH [16], and S US VIBES [7], have expanded the scope to repository-level settings, where coding agents modify multiple files within established codebases. Despite this increased realism, the common characteristic of these studies is that AI-generated code is evaluated in the context of existing software systems. Evidence regarding the security of AI assistance is mixed. A large body of work has shown that LLMs frequently generate vulnerable code. For example, Pearce et al. [14] found that approximately 40% of GitHub Copilot completions for CWE-targeted prompts contained security vulnerabilities. Subsequent studies have reported similar findings across a variety of models, programming languages, and task settings [23], [24], [25], [26], [27], [28], [29]. However, the presence of vulnerabilities in generated code does not necessarily imply that AI-assisted develop-ment produces less secure software, since developers may identify and correct these issues during review. To investi-gate this question, Perry et al. [30] conducted a controlled user study and found that participants with access to an AI assistant introduced significantly more vulnerabilities while simultaneously expressing greater confidence in their solutions. Collectively, these findings suggest that LLMs often generate insecure code and that the ultimate security outcome depends critically on the expertise, vigilance, and review practices of the human developer. 

Security of Vibe Coding. Despite the growing body of literature, no prior study has examined the security of real-world applications deployed through vibe coding. Existing research has focused either on AI-generated code integrated into human-authored codebases [25], [26] or on benchmark tasks conducted in controlled experimental settings [7], [15], [16], [31]. Consequently, the current evidence base is largely restricted to localized vulnerabilities—typically spe-cific CWEs within individual code files—evaluated against a predefined oracle. Such evaluations cannot capture how application security emerges from the interplay of architec-tural decisions, implementation choices, refactoring actions, and deployment configurations across an entire software system. Yet these higher-level decisions are precisely where autonomous coding agents exercise the greatest degree of control in the vibe-coding paradigm. This gap is becoming increasingly consequential as vibe coding enters the mainstream. Platforms such as Lovable and Claude Code already enable millions of users, many with limited software engineering expertise, to build and deploy production applications. Therefore, security risks in these systems have immediate implications for real users. To address this gap, we provide the first empirical characteriza-tion of the security landscape of real-world vibe-coded ap-plications. By analyzing vulnerabilities in publicly deployed systems, our study moves beyond controlled benchmarks and offers the first understanding of vulnerabilities emerging in applications operating on the public Internet. 

## 3. Dataset Construction 

In this section, we construct a high-quality vulnerabil-ity dataset of vibe-coded applications to investigate their security risks. To this end, we first construct V IBE APPS ,a representative corpus of real-world vibe-coded applica-tions. We then develop a vulnerability-discovery framework that combines agentic code auditing with human-in-the-loop validation to discover their vulnerabilities. The resulting vulnerabilities constitute the dataset V IBE VULNS , which serves as the foundation for our security analysis. 

3.1. Vibe-Coded Application Collection 

To collect real-world vibe-coded applications, the pri-mary challenge lies in how to identify such applications, as they are not explicitly labeled as vibe-coded. To address this, we design a three-stage pipeline. We first discover potential vibe-coded applications using fingerprints associated with AI coding agents. We then filter these candidates according to a set of criteria to obtain a high-quality subset. Finally, we determine applications to be vibe-coded based on the degree of AI contributions. 

Stage 1: Potential Application Discovery. Although there are no explicit indicators for vibe-coded applications, the repositories that have AI-generated code usually contain some fingerprints left by the corresponding coding agents. For example, repositories using Claude Code include the configuration directory .claude/ , while those developed with Lovable contain metadata <meta name="author" content="Lovable" /> . These fingerprints provide useful signals for discovering potential vibe-coded appli-cations. Based on this observation, we query the GitHub Code Search API to collect the repositories from GitHub that contain the relevant fingerprints. In this study, we focus on two popular AI coding agents: Claude Code, representing command-line-based agents, and Lovable, representing AI-based development platforms. They capture distinct devel-opment workflows and provide broader coverage of the current vibe-coding ecosystems. Although we leverage the fingerprints of these two coding agents to discover candidate applications, in practice, many repositories are developed using multiple AI coding agents. Therefore, our collected applications are not limited to those developed exclusively by Claude Code or Lovable. This step results in an initial dataset of 74,800 candidate applications. 

Stage 2: High-Quality Application Selection. To ensure that our study is conducted on high-quality applications, we further select those from the initial dataset that are likely to be used in practice. Specifically, we select the repositories that satisfy the following criteria: (1) they provide suffi-ciently complete documentation, with a README contain-ing at least 100 characters; (2) they have 500 to 1M lines of 3code and 10 to 5,000 commits, where the lower bounds filter out small repositories with trivial functionality and the upper bounds filter out the extremely large repositories that typi-cally contain substantial redundant implementations (e.g., third-party libraries) based on our observation; and (3) they are implemented as applications or tools, thereby excluding repositories intended primarily for documentation, tutorials, or illustrative examples (a detailed category breakdown is provided in Appendix B, Table 7). By applying these, we selected 37,962 applications from the initial dataset. 

Stage 3: Vibe-Coded Application Determination. After the previous two steps, we obtain a high-quality dataset of real-world applications that contain AI-generated code. We further determine whether an application is vibe-coded. The vibe-coded applications should be created by AI from scratch, instead of filling features in existing codebases. To capture this property, we require the first commit of a repository to be AI-authored. In addition, vibe-coded appli-cations should be developed predominantly by AI coding agents, with minimal human intervention. To quantify the degree of AI involvement, we measure AI contributions in two metrics: the proportion of AI-authored commits and the proportion of AI-authored code lines. We classify an application as vibe-coded only if both metrics exceed 85%. To support this determination, we need to identify AI-authored commits within repositories. We label code commits as AI-authored based on two complementary signals. First, we leverage explicit metadata. Many coding agents and vibe-coding platforms leave their identification in commit messages, author fields, or commit trailers, such as "Co-authored-by: Claude <noreply@anthropic.com>" or "Generated with Claude Code" . Any commit containing such markers is directly labeled as AI-authored. Second, for commits without explicit metadata, we identify AI-authored commits based on their message style. AI-generated commit messages often exhibit recognizable patterns, including emoji-prefixed summaries, highly structured bullet lists, and unusually detailed descriptions of code changes. We use these stylistic patterns to infer AI authorship for commits that lack explicit agent identification. We finally collected 10,517 applications, forming our dataset VIBE APPS for analyzing the landscape of vibe coding. 

3.2. Vulnerability Discovery 

After collecting vibe-coded applications, we construct their vulnerability dataset for security analysis. However, discovering vulnerabilities across these applications poses another challenge: they vary substantially in scale , pro-gramming language , technology stack , and even application type (as shown in Section 4). No conventional vulnerability-detection tool can effectively cover such a diverse set of applications. To address the challenges introduced by differ-ent scales , programming languages , and technology stacks ,we adopt code-auditing agents for vulnerability discovery. Nevertheless, auditing different types of applications re-quires different agent capabilities and auditing workflows. Therefore, we focus on the dominant type of vibe-coded applications (from Section 4): web applications. Among the 10,517 collected applications, 9,935 (over 94%) are web applications, making their vulnerabilities broadly represen-tative of the overall security landscape of vibe coding. Based on this, we develop the vulnerability-discovery framework that combines multi-agent code auditing with structured human-in-the-loop validation. Specifically, we first leverage a multi-agent system to analyze each reposi-tory and generate structured vulnerability reports. We then perform both AI-assisted and manual validation to eliminate redundancy and false positives, producing the final set of vulnerabilities for analysis. 

3.2.1. Multi-Agent Code Auditing. We leverage LLM-based code auditors to identify potential vulnerabilities in each repository. To maximize vulnerability coverage, we adopt a 2 × 2 auditing design that combines two agent frameworks with two security auditing pipelines, yielding four independent audit reports for each repository. The two agent frameworks are Claude Code with Claude Sonnet 4.6 and GitHub Copilot with GPT-5.3-Codex. Each agent is paired with two complementary security skill sets: OpenAI’s official security-best-practices skill [32], which provides guidance for detecting web application vulnerabilities across a range of technology stacks, and Antigravity Awesome Skills [33], a widely adopted collection of security skills with nearly 40k GitHub stars that integrates SAST tools and vulnerability-detection rules. Each skill set is organized around a primary security-analysis agent and a collection of technology-specific subagents. The primary agent, defined by the top-level 

SKILL.md , first analyzes the repository structure, frame-work usage, and technology stack. Based on this assessment, it automatically invokes specialized subagents tailored to the detected technologies. These subagents implement security auditing strategies specific to individual frameworks and ecosystems, such as React and Next.js . This hierarchi-cal architecture helps reduce missed vulnerabilities arising from the limitations of any single model or auditing strategy. We apply this 2×2 design to audit the web applications. To ensure that the discovered vulnerabilities reflect realistic security exposure, we restrict our analysis to applications with publicly accessible deployments. These account for 1,170 of the 9,935 web applications (nearly 12%, as shown in Section 4). Auditing all deployed applications would require substantial computational resources and manual val-idation effort. Therefore, we randomly sampled 200 applica-tions from the 1,170 deployed repositories for detailed secu-rity auditing. As a result, across these applications, the four auditing configurations generated 9,353 raw vulnerability reports. The auditing process consumed 7.98 billion tokens and 130 hours of compute time, averaging 9.98 million tokens and 9.7 minutes per audit run. 43.2.2. Vulnerability Deduplication and Validation. The raw vulnerability reports contain substantial redundancy, as the four auditors frequently identify the same vulnerability using different descriptions. To address this, we use Claude-Sonnet-4.6 to review the four audit reports for each reposi-tory, deduplicate overlapping candidate vulnerabilities, and merge semantically similar reports into canonical entries. We then use the same model to filter out the non-actionable items (Table 6), resulting in 1,934 candidate vulnerabilities. To assess whether these candidates are true positives, we use both agent and human validation. We first built a validation agent that investigates the exploitability of each reported issue. The agent examines the relevant source files, traces whether user-controlled input can reach the reported sink, and checks for any sanitization or guard conditions along the execution path. It then attempts to construct a harmless proof-of-concept probe to the deployed application that would trigger the vulnerability if it were exploitable. Candidate vulnerabilities with clearly invalid exploitation paths are discarded, while plausible or uncertain cases are forwarded for human review. This process reduced the can-didate set to 1,513 potential vulnerabilities. Each remaining candidate is then independently re-viewed by two authors with security expertise. They review its audit report, inspected the relevant application source code, and examined the deployed website to verify ex-ploitability. A candidate is only considered a vulnerability if both authors classify it as exploitable. Furthermore, the two authors assign each vulnerability a severity level (Low, Medium, High, or Critical) following OWASP risk-rating criteria [34], and map it to the corresponding OWASP Top 10 category [35]. Disagreements are resolved through discussion; inter-rater agreement was Cohen’s κ = 0 .87 .This produced 1,471 vulnerabilities and forms our dataset VIBE VULNS for security analysis. All vulnerabilities were responsibly disclosed to the respective maintainers. 

## 4. RQ.1: What Is the Landscape of Real-World Vibe-Coded Applications? 

To answer RQ.1, we characterize the 10,517 repositories in our corpus V IBE APPS along four dimensions: reposi-tory scale and development characteristics, application cate-gories, technology stacks, and deployment status. This char-acterization establishes the landscape of real-world vibe-coded applications and motivate our subsequent security analysis of deployed web applications. 

Repository Scale and Development Characteristics. Real-world vibe-coded applications are often large-scale software projects. The median repository contains 8,351 lines of code (LoC) across 101 files, while repositories at the 90th percentile reach 22,714 LoC and 212 files. Overall, 94.5% of repositories contain more than 5,000 LoC, and 36.0% exceed 10,000 LoC. Despite their considerable scale, these applications are typically developed within a short time. Measured from the first to the last commit, the median devel-opment span is 9.8 days. Furthermore, 66.3% of repositories have development spans shorter than 30 days, and 84.3% are completed within 90 days. However, these short devel-opment spans do not imply limited development activities: 65.1% of repositories contain at least 25 commits, indicating substantial development activity within compressed periods. To better characterize the development process, we ex-amine the relationships between repository scale, develop-ment span, and commit counts. Repository scale is moder-ately correlated with commit count (Spearman ρ = 0 .67 

for LoC and ρ = 0.62 for file count), suggesting that larger repositories are generally associated with more in-tensive development activity. In contrast, commit count is more weakly correlated with development span (Spearman 

ρ = 0 .51 ), indicating that elapsed time explains compar-atively little of the variation in development effort. More-over, commit density declines sharply as development span increases (Spearman ρ = −0.65 ). The shortest-span quartile of repositories sustains a median of 11.9 commits per day, whereas the longest-span quartile drops to 1.2 commits per day, indicating that vibe-coded applications are typically built in short, high-intensity periods. Taken together, these findings reveal a distinctive scale-and-speed profile: many vibe-coded applications combine large codebases with highly concentrated development activ-ity. Rather than emerging through the gradual accumulation of small code generations, they are often produced through intense bursts of project-level iteration. This development pattern indicates that few human reviews and hardening are likely to be applied before deployment. This confirms our motivation to study the security of vibe-coded applications. 

Application Categories, Development Languages and Technology Stacks. We present the categories of vibe-coded applications, and their most-widely used program-ming languages and technology stacks in Figure 1. It is interesting to note that vibe-coded applications are highly biased towards web applications, which account for 94.5%. In contrast, the second-largest category, Games, only repre-sents 1.2% of the repositories. The distributions of devel-opment languages and technology stacks further reinforce this trend. JavaScript , CSS , HTML , and TypeScript 

are the most commonly used programming languages, while 

TypeScript , React , Tailwind CSS , and Vite are the predominant technologies adopted in vibe-coded appli-cations. This finding suggests that vibe coding is primarily used to develop web applications. We further examine whether the vibe-coded applications are actively deployed. Among all 10,517 repositories, 1,226 (11.7%) contain validated and reachable deployment links. Restricting this to web applications, the figure is 1,170 of 9,935. Therefore, vibe-coded applications are not merely repository artifacts, but also deployed systems with exter-nally observable attack surfaces.

6.1. Memory Defects 

Memory defects arise when security obligations intro-duced earlier in development are not reliably retained, prop-8agated, or revisited as the project evolves. Although coding agents can reconstruct context from prompts, files, and recent interactions, they do not maintain a complete and reli-able memory of all project-wide security requirements over time. In the vibe-coding lifecycle, these defects are primarily in later-stage development, especially during iteration , when security-relevant changes must be propagated consistently across the codebase, and deployment , when deferred obliga-tions must still be completed before release. Together, these defects account for 287 (19.5%) vulnerabilities. 

Iteration Stage: 1⃝ incomplete change propagation . ## 8. Discussion 

8.1. Sources of Insecurity 

Taken together, our findings suggest that insecurity in vibe coding emerges from the interaction of three condi-tions: agent-level limitations, workflow-level weaknesses, and responsibility misalignment. 

Agent-Level Limitations. Many of the defects above can be traced to the behavior of the agent. First, because LLMs are effectively stateless across turns, they rely on reconstructed context, which is often incomplete and can introduce incon-sistency ( memory defects ). Second, agents are optimized to follow user instructions and produce visible progress, which can deprioritize long-term security concerns ( objective de-fects ). Third, model knowledge is incomplete, and genera-tion is probabilistic, which can produce incorrect claims and omitted implicit security rules ( knowledge defects ). These limitations are structural and cannot be fully resolved by model scaling or prompt engineering alone. 

Workflow-Level Weaknesses. Experienced developers can also forget plans or write temporary code, but conventional software processes reduce the resulting risk. In human-led development, requirements are rechecked, code is reviewed, and release gates enforce security hardening before deploy-ment. In many vibe-coding settings, these controls are weak or absent. Non-expert users often cannot maintain persis-tent plans, conduct effective security reviews, or evaluate deployment configurations. As a result, defects that would normally be caught at different stages can pass through the pipeline together. 

Responsibility Misalignment. Vibe coding is asymmetric: the agent performs much of the design and implementation work, while the user often has limited technical partici-pation. However, many systems still assign responsibility for design quality, security review, deployment, and mainte-nance to the user. This creates a mismatch between respon-sibility and capability. Users who lack security expertise are not well positioned to judge secure design choices, detect security flaws in generated code, or maintain secure deploy-ments. Under this mismatch, vulnerabilities can emerge and persist without clear ownership. 

8.2. Implications for Mitigation 

Securing vibe-coded applications requires a shift from model-centric improvement to workflow-aware mitigation. Stronger models can reduce some knowledge defects , but they do not reliably address memory defects or objective defects . Therefore, mitigation requires a systematic approach that addresses the entire system. 

Security as a Workflow Property. Security should be treated as a property of the workflow, not only of the generated code. Simply incorporating security scanning into the code generation step is insufficient. Future vibe-coding systems should integrate security controls at each stage of the lifecycle, including requirements specification, code generation, and deployment configuration. 

Responsibility Aligned with Capability. Responsibility needs to match actual capability. In current vibe-coding sys-tems, users often remain implicitly responsible for security decisions they cannot evaluate. A more secure workflow would actively make the agent and the surrounding system responsible for maintaining security obligations, while ask-ing users to make decisions only when those decisions are visible, understandable, and appropriately scoped. 

From Awareness to Action. Security awareness must be connected to corrective action. Our experiment shows that agents sometimes recognize a security risk but still leave the insecure implementation in place, often by moving the risk into a comment or explanation. For vibe coding, this is insufficient because users may not understand or act on such warnings. Future agents should treat recognized risks as unfinished work and be responsible for producing a secure implementation. 

## 9. Conclusion 

In this paper, we study the prevalence, severity, and categories of vulnerabilities in vibe-coded applications, and investigate whether those vulnerabilities are caused by sys-tematic failures in the vibe-coding workflow. Our study of 1,471 vulnerabilities from real-world, publicly deployed vibe-coded applications shows that (1) vulnerabilities are widespread, with 90.0% of repositories containing at least one vulnerability; (2) these vulnerabilities are often severe, 13 with over three-quarters of the vulnerabilities rated as high or critical; and (3) they are concentrated in security-critical parts of the application stack. We identify eight recurring failure modes that introduce vulnerabilities throughout the vibe-coding development lifecycle and explain how security obligations are forgotten, underspecified, or never estab-lished. Our study suggests that securing vibe-coded appli-cations requires treating vibe coding as a workflow-level security problem more than a code-generation problem.
