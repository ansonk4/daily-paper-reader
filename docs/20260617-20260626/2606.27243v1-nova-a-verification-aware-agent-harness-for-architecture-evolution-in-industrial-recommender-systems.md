## Abstract
Industrial advertising recommender models are continuously improved through architecture evolution. Upgrades such as RankMixer, TokenMixer-Large, and MixFormer show that better structures remain a key source of quality and business gains. Yet developing such upgrades in production is expert-intensive and difficult to scale. Existing automation is insufficient: AutoML mainly tunes hyper-parameters, while effective gains often require cross-module changes under strict constraints; generic LLM coding agents optimize for runnable code, but runnable code does not imply a valid recommender architecture. Candidates may pass local tests while causing silent failures that degrade performance.   We present NOVA, a level-aware agent harness for verification-aware architecture evolution. NOVA uses an architecture gradient, an SGD-inspired, non-differentiable update signal that aggregates prior modifications, verification diagnostics, metric feedback, and trajectory memory to guide the next modification. A verification cascade checks structure semantics, local executability, offline effectiveness, and online impact; invalid candidates are blocked early, with failure patterns recorded as forbidden directions. L1--L4 task-level control matches automation to task complexity and risk, routing high-risk tasks to Copilot for human oversight. Deployed in an industrial advertising system, NOVA achieves the highest effective pass rate on L2 ScaleUp and L3 Literature-to-Production tasks (54.5% and 60.0%), reduces silent failures compared with coding-agent baselines, and shortens one literature-to-production cycle by over 13x in human-attended time. In online A/B testing, the selected L3 candidate improves GMV on three pCVR objectives by +1.25%, +1.70%, and +2.02%, while reducing pCVR bias by 58.8%, 66.7%, and 37.3%.

# NOVA: A Verification-Aware Agent Harness for Architecture Evolution in Industrial Recommender Systems — Summary

## 1. Core Problem and Overall Meaning

- **Motivation.**
  - Industrial advertising recommender models continue to improve through architecture upgrades such as RankMixer, TokenMixer-Large, and MixFormer.
  - These architectural changes remain a major source of business and quality gains, but developing them in production is **expert-intensive** and hard to scale.
  - The paper targets the gap between manual expert loops and existing automation: AutoML can tune hyperparameters but cannot easily propose cross-module structural changes, while generic LLM coding agents can produce runnable code but cannot guarantee valid recommender architecture.

- **Core problem.**
  - Existing automation either:
    - optimizes only local hyperparameters (AutoML/NAS/HPO), or
    - optimizes for software-level correctness (compilation, unit tests) but ignores recommender-specific architectural semantics.
  - This leads to **silent failures**: candidates that can be compiled and trained but degrade AUC, calibration, or business metrics because of subtle architectural errors (e.g., wrong mask semantics, wrong feature fusion, or wrong attention direction).
  - The paper aims to make architecture evolution **auditable, verification-aware, and risk-controlled** in a production advertising system serving over one billion users.

- **Overall meaning.**
  - NOVA is presented as a **level-aware agent harness** that formulates architecture evolution as a verification-aware search guided by an architecture gradient.
  - It is designed not just to generate code, but to generate and validate **architecturally feasible** modifications before expensive training and online experimentation.

## 2. Proposed Methodology

### 2.1 Core Idea

- NOVA formulates architecture evolution as a **feedback-driven search** in a discrete, constrained modification space.
- Two central mechanisms:
  1. **Architecture gradient**: an SGD-inspired, non-differentiable update signal that aggregates prior modifications, verification diagnostics, metric feedback, and trajectory memory to guide the next modification.
  2. **Verification cascade**: a multi-stage filter that checks structure semantics, local executability, offline effectiveness, and online impact, blocking invalid candidates early and recording failure patterns as forbidden directions.

### 2.2 Problem Formulation

- Candidate architecture is represented as:
  - A t = (G t , φ t , F t )
  - where G t is the model graph, φ t structural hyperparameters, and F t feature configuration.
- Initial state A 0 is derived from the production codebase C, with additional priors from paper sources P and a static knowledge base KB.
- Each iteration applies a modification e t ∈ E:
  - A t+1 = Apply(A t , e t )
- Feasibility is defined by hard production constraints Ω:
  - A Ω = {A | A satisfies Ω}
  - Ω includes interface compatibility, tensor shape consistency, dtype consistency, feature availability, training/serving compatibility, latency limits, and parameter/FLOPs budgets.

- Offline objective:
  - max J_offline(A) = AUC(A), s.t. A ∈ A Ω
- Online objective:
  - J_online(A) = Σ w i · m i (A), with m i ∈ {GMV, Bias}
  - Lower-is-better metrics such as bias are assigned negative weights.

### 2.3 Architecture Gradient

- Because A t is discrete and constrained, standard gradients are unavailable.
- NOVA defines an **architecture gradient** g t as a structured update signal:
  - g t = Grad(e t−1 , V t , ΔJ t , H t )
  - where:
    - e t−1 = previous modification
    - V t = verification diagnostics
    - ΔJ t = offline metric change
    - H t = trajectory memory of prior modifications, failures, diagnostics, and feedback
- g t encodes three types of information:
  - **Weak components**: which part of G t / φ t / F t is likely responsible for current bottlenecks.
  - **Modification directions**: which candidate directions should be explored next.
  - **Forbidden directions**: which previously failed or repeatedly negative patterns should be avoided.

- Next modification is selected by:
  - e * t ∈ arg max_{e∈E} Score(e; g t , H t) s.t. Apply(A t , e) ∈ A Ω
- Architecture is then updated:
  - A t+1 = Apply(A t , e * t )

### 2.4 Verification Cascade

- **Silent failures** are candidates that can run or train but fail to improve offline or online metrics.
- NOVA uses a **multi-stage verification cascade**:
  1. **Structure-semantic gate**
     - Checks modification logic, shape/dtype, feature-to-token mapping, attention direction, mask semantics, and logit fusion.
     - Catches runnable but structurally wrong candidates before training.
  2. **Local testing gate**
     - Checks single-machine executability.
     - Blocks engineering failures before offline training.
  3. **Offline inner loop**
     - Uses AUC to evaluate verified candidates before deployment.
  4. **Online outer loop**
     - Uses GMV/Bias for production validation.

- **Why semantic checks matter before training:**
  - A training framework can execute a graph as long as tensor shapes are compatible, but cannot determine whether masking direction, feature routing, or logit fusion matches the intended architecture.
  - Therefore, compilation and local tests show executability, not architectural validity.

- **Verification as gradient denoising:**
  - Failed candidates are recorded in trajectory memory H as forbidden directions.
  - This feedback reduces repeated semantic failures and denoises the search signal.

### 2.5 Task-Level Control

- NOVA defines **L1–L4 task levels**:
  - L1: atomic structural tuning
  - L2: constraint-aware ScaleUp
  - L3: literature-to-production transfer
  - L4: open-ended innovation
- Execution modes:
  - **AutoRun**: covered by skill specifications
  - **Copilot**: routed to human oversight for uncovered or high-risk tasks
- Mode depends on **skill-specification coverage**, not directly on task level.

### 2.6 Algorithm Flow

- Algorithm 1 implements the evolution loop:
  1. Initialize A 0, H, task level, and mode.
  2. Compute initial architecture gradient g 0.
  3. While not converged and within budget:
     - Propose K candidate modifications along g t.
     - Filter by constraints and forbidden directions.
     - Run verification cascade for each candidate.
     - If all candidates fail, record failures and update g t+1 without training.
     - Otherwise, select best surviving candidate, optionally confirm with human in Copilot mode.
     - Apply modification to obtain A t+1.
     - Offline-train and evaluate AUC.
     - Record feedback in H and compute next gradient g t+1.
  4. After offline search stops, select best feasible offline candidate.
  5. Validate selected candidate in online A/B.

### 2.7 System Implementation

- NOVA is implemented as a **Main Agent orchestrating specialized sub-agents**:
  - Initialization → Solution Design → Code Generation → Quality Assessment → Local Testing → Offline Training & Evaluation → Online Experiment & Evaluation
- Exception handling distinguishes:
  - **Temporary execution failures**: retried with bounded budget; not fed into gradient.
  - **Candidate-level failures**: recorded as forbidden or low-value directions.
  - **Rule-uncovered or high-risk failures**: routed to Copilot.

## 3. Experimental Design

### 3.1 Tasks and Scenarios

- **L2 ScaleUp**
  - Conducted on a production RankMixer-style backbone.
  - Searches over coupled structural hyperparameters:

    - token_dim
    - number of RankMixer layers
  - Model size must stay within ±10% of production baseline.
- **L3 Literature-to-Production**

### 3.2 Dataset

- Large-scale industrial advertising recommendation dataset from production traffic.
- Training corpus spans **one month** and contains **billion-scale** user–item interaction records.
- Each record has **over one thousand feature fields**, covering sequential and non-sequential signals.
- All candidate models are trained from scratch on the same dataset.

### 3.3 Budget and Protocol

- All LLM-based methods use the same base model: **Claude Sonnet 4.6**.
- Performance differences therefore reflect **harness design**, not raw LLM capability.
- All methods are evaluated under the same offline budget.
- For each method:
  - N_task = 10 independent tasks
  - At most N_iter = 10 iterations per task
  - Each iteration may generate up to K candidate modifications.
  - Candidates passing required gates are evaluated by offline AUC.

### 3.4 Metrics

- **Recommendation quality**
  - Offline: ΔAUC relative to production baseline.
  - AUC-positive defined as ΔAUC > 0.001.
  - Online: ΔGMV and ΔBias in production A/B tests.
- **Engineering validity**
  - Local Pass Rate (LPR) = N_p / N_g
- **Silent-failure rate**
  - Silent Failure Rate (SFR) = 1 − N + / N_p
- **Effective pass rate**
  - Effective Pass Rate (EPR) = N + / N_g = LPR · (1 − SFR)

### 3.5 Baselines

- **Human Expert Loop**
  - Historical iteration logs of senior engineers replayed under unified evaluation protocol.
- **ReActAgent-only**
  - Single LLM agent under standard ReAct.
  - No skill specifications, multi-candidate generation, trajectory memory, semantic verification, or architecture-gradient feedback.
- **OpenHands**
  - State-of-the-art open-source coding agent.
  - Optimizes for runnable code rather than recommender-specific architecture validity.
- **Optuna-TPE**
  - AutoML baseline using Tree-structured Parzen Estimator with population-based asynchronous early stopping.
  - Searches over a predefined switch space including learning rate, hidden dimension, block number, dropout, token dimension, and token count.
  - Applicable only to L2 ScaleUp.
- **NOVA**
  - Full system with architecture-gradient feedback, semantic verification, trajectory memory, skill specifications, and level/mode control.
