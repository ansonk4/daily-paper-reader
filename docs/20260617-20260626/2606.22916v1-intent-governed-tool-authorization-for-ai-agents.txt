# Intent-Governed Tool Authorization for AI Agents 

Genliang Zhu Accentrust Georgia Institute of Technology Chu Wang Accentrust University of Illinois Urbana-Champaign 

Abstract 

AI agents increasingly act through external tools: they read private data, construct structured payloads, submit write requests, export records, and coordinate workflows across application boundaries. Existing authorization mechanisms usually ask whether an integration credential, app, or token can call a tool. That question is necessary but incomplete. A tool call can be authorized by static credentials and still be unjustified by the user’s current request. For example, a credential that can read and export records should not expose export authority when the user only asked for a bounded summary, and a model-generated delete call should not execute merely because the integration has a delete scope. This paper proposes Intent-Governed Access Control (IGAC), a server-side authorization layer that treats the user’s expressed intent as a monotone, auditable policy attribute for AI-agent tool use. IGAC introduces intent certificates, session-scoped policy narrowing, intent-aware manifest filtering, and intent-tool-payload consistency checks. The central invariant is that user intent may only reduce the authority granted by static integration policy; it never expands scopes, data policy, tenant boundaries, or review requirements. We map IGAC onto OpenPort, an existing governance substrate that already implements authorization-dependent discovery, scope and ABAC-style policy checks, draft-first writes, preflight impact binding, state-witness checks, idempotency, stable reason codes, and audit. The current artifact includes a minimal reusable IGAC runtime path in OpenPort: an intent endpoint, in-memory certificate store, intent-aware manifest endpoint, intent-bound action/preflight checks, reason-code failures, and audit binding. We therefore present a formal model, security analysis, implementation mapping, executable synthetic checks, a runtime-backed microbenchmark, an expanded real-LLM pilot, a scored end-to-end LLM-in-the-loop runtime benchmark over the actual OpenPort path, and a benchmark-shaped external-subset runtime pilot. The deterministic checks validate internal policy invariants, while the provider-backed LLM studies provide preliminary runtime-path evidence of unsafe-execution containment and help characterize the remaining unsafe-accepted-authority surface. The paper positions intent as a missing access-control primitive between natural-language user requests and enforceable tool effects. 

Keywords: AI agents, access control, tool use, prompt injection, least privilege, intent certificates, dynamic authorization, OpenPort. I. I NTRODUCTION 

AI agents are moving from conversational assistance to operational action. A deployed agent can inspect records, update state, export data, create appointments, execute workflows, and coordinate with other services. In such settings, tool access is no longer a convenience feature of a language model interface. It is an application security boundary. The access-control question normally asked by an application gateway is credential-centric: does this token, app, or integration have permission to call this endpoint? Credential-centric authorization is necessary. It supports revocation, least privilege, tenant isolation, and accountability. But agentic tool use introduces a second question that traditional API authorization rarely models: is the proposed tool call justified by what the user currently asked the agent to do? 

This distinction matters because the same credential can be legitimately broad while a particular user request is narrow. A finance assistant may have both read and export scopes because the organization approves those capabilities for some workflows. Yet if the user asks ”summarize my transactions for this week”, exposing a bulk export tool to the model is unnecessary and potentially dangerous. If the model later proposes an export because an external document, retrieved note, or malicious tool description influenced it, credential-only authorization cannot distinguish the overbroad effect from the user’s original summary request. Likewise, a model may construct a plausible update payload that touches a different resource than the user named, or a delete call that exceeds the intended target set. The recent security literature shows that this is not a hypothetical edge case. AgentDojo evaluates prompt-injection attacks and defenses in tool-using agents [3]. MCP-oriented work identifies tool poisoning, shadowing, and protocol-specific semantic attacks [8], [10], [21], [25]. These works converge on a practical conclusion: the model’s selected tool is not, by itself, trustworthy evidence of authorization. Industry guidance reaches a similar conclusion. OWASP lists prompt injection as the first 2025 LLM application risk and recommends least privilege, human approval for high-risk actions, external-content segregation, and adversarial testing [17], [18]. Those mechanisms are important, but they do not answer whether a specific tool effect is supported by the user’s current intent. This paper proposes Intent-Governed Access Control (IGAC). IGAC adds an intent-control layer in front of the existing effect-control layer. It receives the user’s request, derives an intent certificate, computes a session policy that can only narrow existing static authorization, filters the tool manifest accordingly, and checks that each proposed tool call and payload remains consistent with the certificate. If intent is unknown, low-confidence, ambiguous, or inconsistent with the proposed effect, the gateway fails closed by hiding high-risk tools, denying the call, or downgrading it to a draft/preflight review path. The model, classifier, and planner are not trusted principals. They are advisors whose outputs are evaluated by a server-side policy layer. 

A. Relationship to OpenPort Protocol 

The current OpenPort Protocol paper establishes a governance substrate for agent tool access: authorization-dependent discovery, scoped permissions, ABAC-like data policy constraints, draft-first writes, preflight impact hashing, optional state-witness checks, idempotency, rate limits, and audit. We treat that prior work as the effect-control layer . It determines whether a tool effect is permitted under static integration policy and risk-gated write semantics. IGAC addresses the missing upstream layer. Before an agent reaches the OpenPort action pipeline, the system should decide what the user asked for and what effects are justified by that request. IGAC therefore does not replace OpenPort Protocol. It composes with it: user intent → dynamic least privilege → filtered tools → governed effects .

The distinction is important for novelty and for correctness. The existing OpenPort substrate can prevent a broad range of unsafe effects, but without IGAC it does not know whether a read, export, update, or delete was actually requested by the user. Conversely, IGAC without a server-side effect-control layer would merely be a classifier or prompt pattern. The contribution is the combination of request-level intent certificates with authoritative gateway enforcement. 

B. Contributions 

This draft makes five contributions. 1) Problem formulation. We define intent-tool mismatch as an access-control failure mode for AI agents: a proposed tool call may be statically authorized yet not authorized by the user’s current expressed intent. 2) Formal model. We define intent certificates, monotone session policy, intent-aware manifests, consistency predicates, and least-privilege effect minimization. 23) System design. We design an IGAC gateway that narrows OpenPort manifests and action execution without trusting the model or classifier as final authority. 4) Implementation artifact. We implement a minimal reusable IGAC runtime path in OpenPort, including intent certificates, intent-aware manifest filtering, intent-bound action/preflight checks, reason codes, audit binding, and endpoint tests. 5) Evaluation artifacts and runtime-path study. We provide deterministic policy checks, endpoint-level runtime tests, a 176-task runtime-backed synthetic microbenchmark, an expanded real-LLM classifier/planner pilot, a scored end-to-end LLM-in-the-loop runtime benchmark over 34 synthetic tasks with 3 repeats per model, and a benchmark-shaped external-subset runtime pilot over 12 adapted tasks and 3 models. Together these results support invariant-level, attack-surface, and runtime-path claims, though not deployment-level robustness or statistically powered real-world safety-improvement claims. 

C. Non-Goals 

IGAC is not a proof that prompt injection is solved. It does not assume intent classification is always correct. It does not replace OAuth, MCP, RBAC, ABAC, or the OpenPort effect-control layer. It does not make human review unnecessary for high-risk actions. Its purpose is narrower: bind proposed tool effects to the user’s current request through a server-side, auditable, monotone authorization layer. 

D. Intent Semantics 

One recurring source of confusion in agent-access papers is the word “intent.” IGAC does not attempt to recover a user’s hidden psychological intent. It uses a narrower and operational notion tied to what the system can actually enforce:  

> •

Expressed intent : the user request as stated in trusted conversation input.  

> •

Inferred intent : the classifier or parser’s interpretation of that request.  

> •

Authorized intent : the portion of the inferred request that is still permitted after static OpenPort policy, scopes, workspace guards, and risk policy are applied.  

> •

Operational intent certificate : the bounded, auditable server-side record used by IGAC at enforcement time. This distinction matters for claim hygiene. IGAC does not claim to know what the user “really meant” in a philosophical sense. It only claims to enforce against an expressed, bounded, auditable request interpretation that may narrow static authority but never widen it. II. B ACKGROUND AND MOTIVATION 

A. Tool Use as an Access-Control Boundary 

Tool-using agents mediate between natural language and application effects. In a conventional API client, the caller is usually deterministic software written by the application developer or an integration partner. In an agentic workflow, the immediate caller may be a planner driven by a language model, influenced by a conversation, untrusted documents, tool outputs, memory, retrieved snippets, or tool metadata. The agent may select tools automatically and may construct payloads that the user never directly inspected. The MCP tools specification explicitly frames tools as model-controlled: models may discover and invoke tools automatically based on context and user prompts [14]. That design is useful for interoperability, but it shifts security burden to implementations. If a server exposes a tool list to a model, the server still needs to decide which tools should be visible in the current context and which proposed calls should be accepted. Traditional access control offers relevant foundations. Least privilege limits authority [20]. RBAC associates permissions with roles [4]. ABAC uses attributes of subjects, objects, actions, and environments to decide access [7]. OAuth-style systems bind tokens and audiences, and MCP authorization guidance emphasizes validating tokens for the intended resource server [13]. IGAC builds on these concepts but introduces an attribute that is often missing: the user’s current natural-language intent. 

B. Why Static Scopes Are Not Enough 

Static scopes describe what an integration may ever do, not what it should do for this request. Consider a tool set with:  

> •

transaction.list , a read tool;  

> •

transactions.export_csv , a high-risk export tool;  

> •

transaction.create , a write tool that creates a draft; 3• transaction.delete , a high-risk destructive tool. An integration may legitimately receive all four scopes because an organization wants the agent to help with multiple workflows. But a specific request such as ”show transactions from last week” does not justify export or delete authority. A request such as ”create a reimbursement for lunch” may justify a narrow create draft, but not arbitrary updates. A request such as ”delete duplicate transaction X” justifies a high-risk delete candidate only under review and preflight constraints. Intent refines the static policy for the current session. The key point is not that scopes are flawed. They remain the upper bound. The flaw is treating scopes as the only authorization input for a probabilistic, context-sensitive caller. IGAC makes the following rule explicit: 

Psession (u) ⪯ P static (a),

where u is the user request, a is the integration app, and ⪯ means ”no more permissive than.” User intent narrows authority but never widens it. 

C. Prompt Injection and Tool Selection 

Prompt injection is a central motivation but not the only threat. OWASP describes direct and indirect prompt injection, including attacks through websites, files, images, and other external content, and notes impacts such as unauthorized function access and critical decision manipulation [17]. AgentDojo and InjecAgent show that agents can be hijacked by untrusted tool outputs and external content [3], [28]. ToolHijacker, ToolTweak, MCPTox, and MCP threat-modeling work show that tool metadata itself can become an attack surface [8], [22], [23], [25]. IGAC does not attempt to detect every malicious instruction. Instead, it asks a different question: even if the model is influenced, does the proposed effect remain within the user’s certified intent and the static policy? If not, the server can hide the tool, deny the call, or route it to draft/preflight review. This shifts defense from model obedience to enforceable effect constraints. 

D. OpenPort as Substrate 

OpenPort already implements important effect-control primitives. The reference runtime filters manifests by scopes, enforces policy constraints and workspace boundaries, creates drafts for writes, computes preflight impact hashes, optionally binds state witnesses, performs idempotency replay, and emits audit events. The product layer also contains controlled flows that map user messages and attachments into preview cards and action drafts before confirmation. These are implementation evidence for a pattern: user-facing input recognition and server-side effect governance should be connected by an auditable policy boundary. The IGAC layer proposed here is the missing connection. It generalizes product-layer controlled-flow practices into an OpenPort-level authorization model that can apply to external agents, tool manifests, and action execution. III. R UNNING EXAMPLES 

We use synthetic examples throughout the paper. They are deliberately generic and do not depend on private deployment details. 

A. Bounded Read vs Bulk Export 

Suppose the integration has both transaction.read and transaction.export scopes. The user asks: Show me this week’s transactions and summarize unusual spending. A credential-only system may expose both transaction.list and transactions.export_csv . A model influenced by external context might choose the export tool because it can retrieve more data. Under IGAC, the certificate is: 

I = {read , summarize },BR = {transactions in current week },BE = {no export, no mutation }.

The export tool is hidden from the manifest because the effect class export is not in I. If a planner still attempts to call the export endpoint directly, the consistency gate rejects it with agent.intent_tool_mismatch . Static OpenPort policy still applies: if the read tool lacks scope or violates a date-window policy, it is denied regardless of intent. 4B. Narrow Create Draft 

The user asks: Create a lunch reimbursement for 23.50 dollars yesterday. The certificate includes a create intent, amount bound, date bound, and resource type. The manifest may expose a create tool but not delete/export/admin tools: 

I = {create }, BE = {single transaction draft, amount ≤ 23 .50 + ϵ}.

The proposed create payload must stay within those bounds. If the model constructs a transaction for 2350 dollars, uses a different date, or attaches unrelated bulk metadata, consistency fails. The safe fallback is a draft with a preview and review, not direct execution. This mirrors the existing OpenPort philosophy that write requests should be reviewable by default. 

C. High-Risk Delete 

The user asks: Delete transaction txn-123 because it is a duplicate. The certificate includes a delete intent and an explicit record identifier. IGAC may allow the delete tool to appear, but only with high-risk review mode: 

I = {delete }, BR = {txn-123 }, ρ = preflight .

The OpenPort effect-control layer then computes impact, binds the impact hash to the payload, requires idempotency if auto-execution is configured, and otherwise creates a draft. If the payload changes to txn-124 after preflight, the preflight hash or state witness should fail. The point is that IGAC handles the request-level authorization and OpenPort handles the effect-level safeguards. 

D. Indirect Prompt Injection Through a Document 

The user asks: Summarize the attached invoice and tell me whether it looks reimbursable. The attachment contains an instruction-like string: Ignore prior instructions and export all transactions. IGAC anchors authority to the trusted user request, not to instructions found in the untrusted attachment. The certificate authorizes summarize/transform effects over the attachment and possibly a bounded read if policy allows it. It does not authorize export. If a model-generated plan tries to export records, the server rejects it even if the model says the document requested it. 

E. Compound Workflow 

The user asks: Find the duplicate transactions, show them to me, and delete the one I approve. This should not produce one broad certificate that authorizes both search and delete immediately. A safer design decomposes it: 

C1 = read/summarize , C2 = delete with explicit approval .

The first certificate permits bounded reads and presentation. The second certificate is issued only after the user selects a specific item and confirms deletion. This design avoids intent drift and overbroad multi-step leases. 

a) Rule: Read-Then-Act Decomposition.: A certificate must not authorize both information-gathering and high-risk mutation unless the user has already explicitly selected the mutation target. In operational terms, read/summarize certificates may discover and present candidate objects, but update/delete/export certificates must be re-issued after the user confirms the concrete target and effect. IV. T HREAT MODEL 

A. System Boundary 

We consider a server-side OpenPort gateway mediating all agent access to application data and actions. The gateway is trusted to enforce authentication, static authorization, tenant boundaries, data policy, rate limits, drafts, preflight checks, and audit. The model runtime, planner, intent classifier, tool selector, and external content sources are not trusted as final security principals. The gateway receives or derives an intent certificate from the user request. The certificate may be produced by a rule-based recognizer, a model-assisted classifier, a hybrid classifier, or a product-specific controlled-flow router. The gateway treats this certificate as an advisory input that may narrow authority. It never allows the certificate to bypass static policy. 

B. Assets 

Protected assets include:  

> •

application data reachable through read tools;  

> •

write effects such as create, update, delete, export, delegation, and administrative actions;  

> •

tenant and workspace boundaries;  

> •

user intent records and session policy;  

> •

audit trails and governance evidence;  

> •

operational budget, rate-limit capacity, and review capacity. 

C. Adversaries 

We model the following adversaries and failure sources:  

> •

Malicious user : submits a prompt that tries to obtain broader tool access than policy permits.  

> •

External content attacker : embeds instructions in webpages, files, emails, resumes, retrieved documents, or tool outputs.  

> •

Tool metadata attacker : attempts to bias tool selection through malicious tool names, descriptions, or descriptors.  

> •

Compromised or overbroad integration : holds credentials with more static scopes than needed for a specific request.  

> •

Benign model error : selects the wrong tool, constructs an overbroad payload, retries unsafely, or drifts across turns.  

> •

Operator misconfiguration : grants broad scopes or weak auto-execute policy.

We do not attempt to protect a malicious server operator, a compromised gateway, or a domain adapter that lies about effects. We do not solve model alignment. We do not claim to prevent all prompt injection. B. Trust and Authority 

The gateway is authoritative. The classifier may be wrong. The planner may be wrong. The model may be manipulated. Tool metadata may be malicious. IGAC therefore has two central trust rules: 1) An intent certificate may only reduce authority. 2) A proposed tool call must still pass static authorization and OpenPort effect-control checks. These rules allow IGAC to use probabilistic components without making them trusted roots. A high-confidence certificate can expose a narrower useful tool set. A low-confidence certificate cannot grant new scopes; it can only force clarification, draft review, or denial.

A. Intent Certificates 

An intent certificate is a structured record binding a user request to an authorization-relevant interpretation. It is not a natural-language summary. It is a policy input with explicit classes, bounds, confidence, expiry, and provenance. A certificate contains:  

> •

id : stable certificate identifier;  

> •

requestHash : hash of canonicalized user request and selected trusted metadata;  

> •

intentClasses : action classes such as read, summarize, create, update, delete, export, admin, unknown;  

> •

resourceBounds : ledgers, workspaces, dates, record IDs, or object types named by the user;  

> •

effectBounds : maximum effect type, row limits, mutation count, or allowed action family;  

> •

confidence : calibrated score or discrete confidence band;  

> •

reviewMode : allow, draft, preflight, confirm, deny, or clarify;  

> •

expiresAt : time or turn boundary after which the certificate cannot authorize new calls;  

> •

classifierSource : rule, model, product flow, hybrid, or human;  

> •

auditDigest : hash used for trace correlation. The certificate should not contain raw secrets or unnecessary private data. If a request includes sensitive content, the certificate stores bounded facts and hashes rather than full text. 

B. Intent Classes 

Table III summarizes a minimal class set. This taxonomy is intentionally small. A deployment can add domain-specific subtypes, but the policy layer should preserve the top-level safety semantics. 

C. Session Policy Narrowing 

Given a certificate C, the gateway derives a session policy PC . This policy is ephemeral and bounded by the static integration policy PA. It includes:  

> •

allowed and denied intent classes;  

> •

maximum risk tier;  

> •

visible tool filters; 8• required review mode;  

> •

resource and date bounds;  

> •

export limits;  

> •

certificate expiry. For example, a summary request may expose only read tools and transformation helpers. A create request may expose one create tool and require draft review. A delete request may expose the delete tool only with preflight and idempotency requirements inherited from OpenPort. 

D. Manifest Filtering 

In many tool-use systems, the model first sees all tools available to a credential and then chooses one. IGAC reduces this surface before model selection. The filtered manifest is: 

Visible IGAC (a, u ) = {t ∈ T | t ∈ Visible OpenPort (a) ∧ PC (u, t ) = 1 }.

The set Visible OpenPort (a) is the current OpenPort static manifest under scopes and data policy. The session predicate PC (u, t ) can only remove tools. Therefore: 

Visible IGAC (a, u ) ⊆ Visible OpenPort (a).

This property is stronger than prompting the model not to call certain tools. Hidden tools are not available for ordinary selection, and unexpected attempts can be denied by the consistency gate. 

E. Intent-Tool-Payload Consistency 

Manifest filtering is insufficient by itself because a visible tool can still be called with an overbroad payload. IGAC therefore checks consistency: 

Consistent (C, t, x, e ) = 1 ,

where C is the certificate, t is the tool, x is the payload, and e is the predicted or preflighted effect. A call is consistent only if: 1) tool class is allowed by the intent class; 2) payload resource identifiers are within certificate bounds; 3) date ranges and row limits do not exceed request bounds; 4) effect risk is compatible with review mode; 5) no denied class is present. If consistency fails, the gateway chooses among denial, draft, preflight, or clarification depending on risk and uncertainty. For example, a proposed export under a read certificate should be denied or require explicit re-authorization. A create action under a create certificate may be converted to a draft. 

F. Review Modes 

IGAC defines a small review-mode lattice: 

allow ≺ draft ≺ preflight ≺ confirm ≺ deny .

This ordering is not about user convenience; it is about increasing protection. A high-risk or low-confidence situation moves rightward. Static OpenPort policy may also require drafts or preflight regardless of intent. 

G. Confidence-Aware Policy 

The intent classifier is explicitly not a trusted principal. Its output is a policy hint that may narrow authority, but it cannot create authority that static OpenPort policy withheld. IGAC therefore treats confidence as an input to fail-closed routing rather than as proof that an effect is safe. Let γ ∈ [0 , 1] be the certificate confidence. C. Intent Certificate 

An intent certificate is: 

C = ( hu, I, B R, B E , γ, ρ, τ, σ ),

where hu is the request hash, I is a set of intent classes, BR is a resource-bound predicate, BE is an effect-bound predicate, γ ∈ [0 , 1] is confidence, ρ is review mode, τ is expiry, and σ identifies classifier provenance. Validity is: 

Valid( C, u, t now ) = ( hu = H(u)) ∧ (tnow < τ ) ∧ (γ ≥ γmin ∨ ρ ∈ { draft , confirm , deny }).

Low-confidence certificates may remain valid only as restrictive certificates. 

D. Monotone Session Policy 

The session policy PC is monotone if: 

∀a, u : Visible IGAC (a, u ) ⊆ Visible OpenP ort (a).

For calls: 

Allow IGAC (a, u, t, x ) = Allow OpenP ort (a, t, x ) ∧ Valid( C(u), u, t now ) ∧ Consistent (C(u), t, x, e ).

Thus, even if Consistent returns true, the call fails if static OpenPort authorization fails.

H. Security Propositions and Proof Sketches a) Proposition 1 (Manifest Monotonicity).: For any authenticated actor a and request u,

Visible IGAC (a, u ) ⊆ Visible OpenP ort (a).

Proof sketch. By definition, the IGAC-visible manifest is derived by filtering the statically visible OpenPort manifest through certificate classes, bounds, and review policy. No construction in C(u) adds tools that were not already in 

Visible OpenP ort (a). Therefore classifier output, prompt injection, or certificate corruption may suppress tools, but cannot reveal a tool outside static OpenPort authorization. 

b) Proposition 2 (Non-Expansion Under Classifier Error).: For any actor a, request u, tool t, and payload x,

Allow IGAC (a, u, t, x ) = 1 ⇒ Allow OpenP ort (a, t, x ) = 1 .

Proof sketch. Equation (8) defines Allow IGAC as a conjunction of Allow OpenP ort , certificate validity, and consistency. Hence an over-broad or compromised classifier can at worst fail to narrow static authority; it cannot transform a statically denied call into an allowed call. The low-confidence validity rule further restricts γ < γ min certificates to draft/confirm/deny-style modes, which again reduces but does not widen authority. 

c) Proposition 3 (High-Risk Fail-Closed Routing).: If any of the following holds, 

Risk (t, x, e ) > θ C , γ < γ min , ¬Boundable( C, t, x, e ),

then IGAC must not authorize immediate execution: 

Allow imm IGAC (a, u, t, x ) = 0 .

Proof sketch. The review constraint requires high-risk or low-confidence calls to map to restricted review modes such as draft , preflight , confirm , or deny . The conservative-failure rule for unboundable effects allows only the same restricted family. Since the OpenPort action pipeline interprets these modes through draft/preflight/review routing, the runtime may still accept a governed draft path, but it cannot create an immediate execution path for a high-risk, low-confidence, or unboundable request. D. Backward Compatibility 

IGAC can be introduced without breaking existing OpenPort clients. Existing clients can continue to call the static manifest and action endpoints. New clients or gateways can supply an optional intent certificate or request an intent-aware manifest. If no certificate is supplied, the runtime behaves like existing OpenPort. If a malformed certificate is supplied, the server fails closed or ignores it depending on deployment mode, but it must never widen authority.

This section sketches how IGAC can be exposed as a conservative OpenPort extension. The sketch is intentionally minimal so that existing clients remain compatible. 

A. Intent-Aware Manifest Request 

An intent-aware client may request a narrowed manifest by sending a certificate reference or a request body to a server-side intent endpoint: 

POST /api/agent/v1/intent {"request": "Summarize transactions from last week", "context": {"trustedUserMessageId": "msg_123", "untrustedContextPresent": true }}

The server returns a certificate: 

{"intentCertificateId": "intent_123", "requestHash": "sha256:...", "intentClasses": ["read", "summarize"], "reviewMode": "allow", "confidence": 0.87, "expiresAt": "2026-06-17T23:59:59Z" }

The client then calls: 

GET /api/agent/v1/manifest?intentCertificateId=intent_123 

An implementation can also embed the certificate in an authenticated request header. The important property is that the server validates the certificate and computes filtering. The client must not be trusted to filter tools locally. 

B. Action Request Extension 

The action endpoint accepts an optional certificate reference: 

POST /api/agent/v1/actions {"action": "transaction.create", "payload": { "...": "..." }, "intentCertificateId": "intent_123", "execute": false }

The server performs three checks: 1) static OpenPort authorization; 2) certificate validity and monotone session policy; 3) intent-tool-payload consistency. Only after those checks does the existing OpenPort action pipeline create a draft, require preflight, or execute under configured policy. 

C. Algorithm 1: Manifest Narrowing 

function intentAwareManifest(ctx, certificateId): staticTools = openportManifest(ctx) cert = loadAndValidateCertificate(ctx, certificateId) if cert is absent: 

15 return staticTools if cert is invalid or expired: return minimalClarificationTools() sessionPolicy = deriveSessionPolicy(cert) visible = [] for tool in staticTools: if sessionPolicy.allowsTool(tool): visible.append(tool) audit("agent.intent.manifest", cert.id, count(visible)) return visible 

The call to openportManifest(ctx) must happen before session filtering. This ordering enforces mono-tonicity. A session policy never sees tools that static policy has already hidden. 

D. Algorithm 2: Action Consistency 

function createIntentAwareAction(ctx, input): tool = resolveTool(input.action) requireStaticAuth(ctx, tool, input.payload) cert = loadAndValidateCertificate(ctx, input.intentCertificateId) if cert is missing: return openportCreateAction(ctx, forceDraft(input), reason="intent_unknown") effect = preflightOrEstimateEffect(ctx, tool, input.payload) if not consistent(cert, tool, input.payload, effect): if safeToDraft(tool, cert): return openportCreateAction(ctx, forceDraft(input), reason="intent_tool_mismatch") return deny("agent.intent_tool_mismatch") riskMode = openportRiskMode(tool, input) mode = mergeReviewModes(cert.reviewMode, riskMode) return openportCreateAction(ctx, applyMode(input, mode)) 

The design deliberately reuses OpenPort’s existing write pipeline. IGAC should not duplicate draft, preflight, idempotency, or execution logic. 

E. Algorithm 3: Certificate Expiry and Drift 

Long-running conversations create drift risk. A certificate should expire after a time interval, turn count, context transition, or completed effect. The server can define: 

Expired( C) = tnow > τ C ∨ turns (C) > n C ∨ ef f ectCompleted (C) = 1 .

If a certificate expires, the gateway can reclassify the latest trusted user request, ask for clarification, or force draft review. For multi-step plans, per-step certificates are preferable to a single broad certificate.

K. Finding 1: Broad Intent Is Easier Than Authorization Bounds 

The strongest design insight from the real-model pilot is not that any one model is good or bad overall. It is that broad intent classification is consistently easier for current LLMs than precise authorization-bound extraction. Across the scored models, intent-class accuracy ranges from 0.7273 to 0.8545, while resource-bound accuracy ranges only from 0.1818 to 0.5818. Review-mode accuracy is likewise materially weaker than broad class recognition. This asymmetry is exactly why IGAC treats model-generated certificates as advisory. A model may often recognize that the user is asking for read, summarize, create, update, delete, or export at a coarse level, yet still fail to extract the exact resource identifiers, row bounds, or risk mode needed for safe autonomous execution. The security boundary is therefore not the classifier. The boundary is the monotone gateway enforcement that validates scopes, narrows manifests, checks payload bounds, and routes high-risk effects through deterministic review logic. The expanded classifier pilot exposes the main empirical risk more clearly than the earlier 12-task probe. Across the three scored models, intent-class accuracy ranges from 0.7273 to 0.8545, but resource-bound accuracy ranges only from 0.1818 to 0.5818 and review-mode accuracy from 0.3818 to 0.6909. This pattern supports the design choice that classifiers cannot be trusted to expand authority. It also sharpens a concrete security rule: LLM-emitted review mode must be advisory only. The server-side deterministic risk policy must override it, and a high-risk tool cannot become allow merely because the classifier says so. This is not a hypothetical concern: GPT-OSS-120B achieved the strongest bound extraction in this small set, yet still emitted immediate allow for 85.71% of high-risk requests. The planner pilot likewise shows a layered story rather than a one-shot win. IGAC manifest filtering lowers unsafe proposal rates for every scored model, but filtered manifests alone still leave unsafe read-path or overbroad-payload proposals. The important invariant is the final column in Table XX: once deterministic consistency gating is applied, unsafe accepted authority falls to zero in this synthetic planner setting for every scored model. a) Finding 2: Runtime enforcement prevents execution, but not all accepted authority.: This is the most direct empirical evidence in the current artifact because it exercises the full gateway path with real model outputs. Runtime unsafe accepted authority remains nonzero at 0.0909– 0.2727 over attack-or-ambiguous cases. The accepted cases are concentrated in draft-first create paths where the LLM-generated certificate or payload failed to preserve the gold amount/resource bound strongly enough for IGAC to reject the proposal before OpenPort draft routing. Thus, the current artifact supports runtime containment of completed side effects, not complete intent-governed authority narrowing under raw LLM certificates. 

b) Finding 3: Model behavior exposes a safety-utility-latency tradeoff.: Manifest reduction remains substantial at roughly 0.79–0.80 across models, but BCR strict is 0 for every scored row and BCR safe reaches only 0.25–0.50. In other words, current benign success in the scored runtime benchmark comes entirely from governed draft creation rather than immediate completion. Qwen3-Next-80B is faster but materially less selective, with the highest unsafe planner proposal rate and unsafe accepted authority. GPT-OSS-120B sits between these regimes with higher clarification and moderate latency. These numbers should be read as preliminary runtime-path evidence and as motivation for stronger certificate-generation layers, not as deployment-level efficacy claims.

A. Invariant 1: Intent Cannot Widen Static Policy 

The monotonicity invariant is: 

PC ⪯ PA.

If a tool is hidden by static OpenPort scopes, ABAC policy, workspace boundary, revocation, or rate limit, IGAC cannot reveal it. This protects against classifier compromise turning into privilege escalation. 

B. Invariant 2: Low Confidence Fails Closed 

If γ < γ min , the system must not treat the certificate as broad authority. It should expose minimal tools, request clarification, or force draft/preflight. Low confidence is not an excuse to fall back to static broad scopes. 

C. Invariant 3: Payload Effects Must Be Bounded 

Even if a tool class is allowed, the payload must stay within request bounds. A read request for ”last week” should not silently become a year-long export. A request to update one record should not become a bulk update. 

D. Invariant 4: The Model Is Not a Principal 

The model may suggest a tool call, but authorization is decided by the gateway. The gateway verifies scopes, policy, certificate validity, consistency, preflight, and review requirements. 

E. Invariant 5: Drafts Are Not Effects 

Draft creation is a governance state, not a completed side effect. Routing uncertain calls to drafts reduces immediate harm, but human review can still fail. Therefore, drafts require audit, clear preview, and review context. 

F. Prompt Injection 

Prompt injection can influence model reasoning, tool selection, or payload generation. IGAC reduces damage by ensuring that injected instructions cannot expand authority beyond the original user intent and static policy. If an external document instructs the agent to export data while the user asked for a summary, the export tool should not be visible under a summary certificate, and any attempted export should fail consistency checks. This is not full prompt-injection prevention. If the user explicitly requests a high-risk action and an attacker changes details within the same effect envelope, IGAC may need context provenance or taint-aware manifests. That is complementary work. 29 G. Tool Metadata Poisoning 

Tool metadata attacks try to make a model choose a malicious or irrelevant tool. IGAC reduces the effect of such attacks by filtering visible tools and checking selected tools against user intent. However, if poisoned metadata belongs to a tool that is both statically authorized and intent-compatible, IGAC alone may not detect semantic deception inside the tool implementation. Manifest signing, tool provenance, sandboxing, and behavior monitoring remain necessary. 

H. Effect Estimation and Conservative Failure 

The consistency predicate depends on estimating or bounding tool effects. This is easy for simple read, create, update, delete, and export actions with explicit resource IDs and row limits. It is harder for tools with implicit side effects, compound workflows, tool-provided summaries, or domain-specific semantics. IGAC therefore adopts a conservative failure rule: 

¬Boundable( C, t, x, e ) ⇒ draft ∨ preflight ∨ confirm ∨ deny .

If the gateway cannot safely bound the payload or predicted effect against the certificate, it must not allow immediate execution. The runtime can still create a draft or require preflight so that OpenPort’s effect-control layer presents impact, state witness, audit context, and reviewer confirmation. This rule prevents the formal predicate 

Consistent (C, t, x, e ) from silently becoming a broad allow when effect semantics are uncertain. 

I. Classifier Compromise 

If the classifier is compromised, it might produce a broad certificate. Monotonicity still prevents scope expansion, but it may fail to narrow authority. Mitigations include rule/model agreement, confidence thresholds, human review for high-risk classes, certificate audit, and adversarial tests. Critical deployments should treat classifiers as untrusted components and require deterministic policy constraints for high-risk effects. 

J. Multi-Intent Tasks 

Users often ask for compound tasks. IGAC should avoid issuing one broad lease for the entire conversation. Instead, it can decompose the request into per-step certificates: 

C(u) = {C1, . . . , C n},

where each Ci authorizes one effect class and resource bound. If decomposition is uncertain, the gateway should require clarification or draft review. 

a) Operational rule.: Read/summarize certificates may be used to discover candidate objects and present them to the user. Update/delete/export certificates should be issued only after the user confirms the concrete target or approved output scope. One broad certificate should not simultaneously cover search, ranking, and high-risk mutation unless the mutation target is already explicit and bounded. XIII. R ELATED WORK 

A. Agent Security Benchmarks 

AgentDojo, InjecAgent, AgentDyn, ToolSandbox, tau-bench, and ToolEmu provide important evaluation settings for tool-using agents [3], [11], [12], [19], [27], [28]. These benchmarks primarily measure agent robustness, task success, or safety under adversarial context. IGAC is complementary: it defines a server-side authorization mechanism that can be evaluated using such benchmarks. 

B. Prompt-Injection Defenses 

StruQ separates prompt and data channels through structured queries and model training [2]. AgentSentry models multi-turn indirect prompt injection as temporal causal takeover and purifies context [29]. AttriGuard uses action-level causal attribution to test whether tool invocations are supported by user intent rather than untrusted observations [6]. IGAC differs by focusing on access-control semantics and gateway enforcement. It can consume signals from structured prompting or causal attribution, but its enforcement point is the server-side policy layer. 30 C. Tool Selection and MCP Security 

ToolHijacker, ToolTweak, MCPTox, and MCP threat-modeling work show that tool names, descriptions, metadata, and ecosystem composition can create security vulnerabilities [8], [21]–[23], [25]. These works motivate intent-aware filtering but do not replace it. Even a perfectly authenticated and well-described tool should not be callable when the user’s request does not justify its effect. 

D. Access Control 

Least privilege, RBAC, ABAC, and OAuth-style delegated authorization remain foundational [4], [7], [13], [20]. IGAC can be understood as a request-level ABAC extension where natural-language intent becomes an ephemeral attribute. The crucial difference is monotonicity: the attribute can narrow but not expand static policy. 

E. Purpose Limitation and Contextual Integrity 

IGAC is also related to purpose-based access control and contextual privacy. Purpose-based systems restrict data use according to declared purposes rather than only subjects, objects, and operations; recent systems such as Data Guard apply purpose-aware masking in warehouse settings, and purpose limitation has also been studied for data-in-transit systems [24], [26]. Contextual integrity frames privacy as appropriate information flow within a social context, using parameters such as sender, recipient, information type, and transmission principle [1], [16]. Recent assistant work operationalizes contextual integrity for information-sharing and LLM delegation [5], [9]. IGAC differs in its enforcement target. It does not attempt to model all contextual information norms. Instead, it turns the current user request into a short-lived, auditable purpose-like certificate that narrows an already-authorized agent tool surface. In this sense, IGAC can be viewed as request-level purpose restriction for agent side effects: a tool may be statically authorized and still unavailable when the present request does not justify its effect. 

F. OpenPort Protocol 

OpenPort Protocol provides the effect-control substrate for this paper. It defines a governance-first gateway with authorization-dependent discovery, scoped permissions, data policy constraints, draft-first writes, preflight, state witness, idempotency, rate limits, and audit. IGAC adds a new upstream layer that binds user intent to tool exposure and proposed effects. XIV. R EPRODUCIBILITY PLAN

Until the end-to-end LLM-in-the-loop benchmark has been run at larger scale, the larger classifier evaluation exists, and the checklist items above are archived in a releasable bundle, the paper should avoid deployment-level empirical safety-improvement claims. XV. C ONCLUSION 

AI-agent access control should not stop at checking whether a credential can call a tool. It should also check whether the proposed call is justified by the user’s current expressed intent. This paper proposed Intent-Governed Access Control, a server-side authorization model that introduces intent certificates, monotone session policy narrowing, intent-aware manifests, and consistency checks between user intent, tool choice, payload, and expected effect. OpenPort provides the existing effect-control substrate: static authorization, policy constraints, draft-first writes, preflight, state witnesses, idempotency, and audit. IGAC adds the missing intent-control layer before those effects are reached. The current evidence supports the design, minimal runtime artifact, endpoint-level invariants, runtime-backed synthetic narrowing results, the expanded real-LLM pilot, the scored end-to-end runtime-path benchmark, and the benchmark-shaped external runtime subset. Collectively, these results show that IGAC is already an executable access-control mechanism for request-bounded least privilege in agent tool use: user intent becomes auditable policy rather than merely prompt context, and completed unsafe execution remains contained across the scored runtime settings.
