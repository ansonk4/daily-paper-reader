## Abstract
AI agents are driving a new software paradigm, with the ability to autonomously call tools, extract information, manage memory, and complete tasks that span applications and data sources. Most existing end-user operating systems, however, are designed for application-centric workflows and offer little native support for AI agents. This mismatch limits the wider adoption of agents and leads to execution overhead and safety risks when running agents on conventional systems. While the concept of agent-native operating systems is emerging, the research community lacks an open testbed to explore the architectural primitives desired for agent-mediated interaction. We present AOHP (Android Open Harness Project), an OS-level agent harness built on the Android Open Source Project (AOSP). The core design principle of AOHP is to treat agents as first-class OS actors, enabling adaptive user interfaces and agent-friendly runtime environments. AOHP preserves the mature Android software and hardware ecosystem while introducing three agent-oriented system mechanisms: personalized service composition, efficient agent interfaces, and secure information flow. Based on preliminary experiments on challenging tasks covering key capabilities of OS agents, AOHP shows clear advantages in task completion (+21.12% completion rate), execution cost (-51.55% token cost), and security-policy compliance.

# AOHP: An Open-Source OS-Level Agent Harness for Personalized, Efficient and Secure Interaction — Summary

## 1. Core Problem, Meaning, and Motivation

- **Problem.** Conventional personal operating systems (e.g., stock Android) are designed around **app-centric workflows**: interfaces are fixed by developers, GUIs are rendered for human visual operation, the software lifecycle assumes one active app at a time, and permission models protect app boundaries but cannot track sensitive data across an agent’s context and tool calls. This creates a structural mismatch with modern AI agents that need to observe services, invoke capabilities across apps, and enforce user intent spanning multiple data sources.
- **Consequences.** Running agents on conventional OSes leads to high execution overhead (excessive GUI navigation, redundant visual processing), safety risks (private plaintext exposed to agents, no cross-app information-flow control), and limited composability (no native way to build task-level, cross-service interfaces).
- **Gap in the community.** While “agent-native operating systems” are an emerging concept, there is no open testbed for exploring the architectural primitives needed for agent-mediated interaction.
- **Goal of AOHP.** Introduce an open-source, OS-level agent harness built on AOSP that treats **agents as first-class OS actors**, enabling adaptive user interfaces, agent-friendly runtimes, and system-enforced security, while preserving Android’s hardware support, ecosystem, and app compatibility.

## 2. Proposed Methodology

AOHP redesigns Android around three agent-oriented system mechanisms, organized in a four-layer architecture.

### 2.1 Core Architectural Idea

- **Four vertical layers:**
  1. **Android ecosystem** (bottom): unchanged apps, system services, hardware, APIs — the compatibility base.
  2. **Unified interaction interface**: normalizes traditional Android interfaces and agent interfaces into four invocation modes — **API, CLI, Structured UI, and Rendered GUI** — so agents can use compact symbolic paths when available and fall back to visual operation when needed.
  3. **AOHP capabilities layer**: reorganizes services and agent functions into **system memory**, **skills**, and **UI utilities**.
  4. **Personalized service composition** (top): assembles capabilities into user-facing, task-specific service entrances.
- **Two horizontal cross-layer mechanisms:**
  - **Efficient agent interfaces** for low-overhead access to resources, state, files, events, and isolated execution.
  - **Secure information flow** for default isolation of sensitive data and system-level information-flow tracking.

### 2.2 Personalized Service Composition

- **Generated service entrances** are user-facing shells backed by OS-managed service composition. Each entrance has:
  - A **task schema** (what the user wants, e.g., “compare shoes under a budget”).
  - A **service graph** mapping the task to concrete capabilities.
  - A **presentation policy** deciding what is shown to the user vs. kept agent-internal.
- **Capability discovery and composition.** AOHP discovers capabilities across API, CLI, and GUI channels, representing each with input/output schemas, preconditions, side effects, and policy labels. The OS agent composes them into higher-level workflows. Composition is policy-constrained: e.g., product search can be parallelized, but purchase submission requires explicit confirmation.
- **Cross-service personalization via system memory.** Preferences learned in one service can improve another, subject to policy. Memory is OS-mediated and split into:
  - Persistent profile memory (stable preferences).
  - Task-local memory (temporary state).
  - Sensitive memory (private values stored behind sandbox indices).

### 2.3 Efficient Agent Interfaces

- **Parallel background interaction.** Decouples execution from the physical display via lightweight virtual displays, allowing agents to run waiting-heavy or independent workflows in the background without preempting the foreground session.
- **Agent-aware UI enhancement.** Abstracts GUIs into structured representations with lower redundancy and richer semantics, while retaining rendered GUI fallback.
- **Native sandbox runtime.** An OS-managed sandbox for local computation, transformation, and tooling independent of app-facing interfaces; can be created and reset as an execution surface.
- **Unified file shortcut.** Treats files as first-class task objects at the OS boundary. GUI storage actions are reflected back as structured file observations; agents can hand artifacts to other interfaces via direct programmatic access or system UI flows, unifying file-producing GUI steps and file-consuming programmatic steps into a single cross-app data plane.
- **Event stream abstraction.** Lets agents subscribe/unsubscribe to continuous data sources:
  - **Dynamic notification capture:** buffers transient events (toasts, pop-ups, push notifications).
  - **Sensor streaming access:** streams hardware sensor data (accelerometer, gyroscope, microphone, camera).

### 2.4 Secure Information Flow

- **Policy enforcement over runtime data use.** For each sensitive operation, the policy layer evaluates data source, requested purpose, destination, action sensitivity, and approval state. Non-sensitive flows proceed; private-data transfers or state-changing actions require consent or are blocked. Explanations are given in terms of source, purpose, destination, and downstream effect.
- **Sensitive source sanitization.** Before sensitive content enters the agent context, AOHP replaces plaintext with typed placeholders such as `<payment-card:uuid>` or `<home-address:uuid>`. A **data vault** stores sensitive values behind opaque identifiers. Sanitization covers application pages, files, event streams, API responses, system memory, and user interactions. Developer annotations can mark sensitive fields; otherwise conservative detection rules are applied.
- **Trusted vault and execution.** When an agent needs to operate on sensitive information, it submits an intent to a **trusted vault executor** using the placeholder. The executor checks policy, obtains user approval when needed, and performs operations (formatting, comparison, validation, composition) inside the trusted environment. If the result is still sensitive, it returns another placeholder. The executor also mediates sensitive transfers to external interfaces (e.g., filling a GUI field or substituting plaintext at the API/CLI boundary).
- **Data-flow taint tracking.** Once sensitive data enters AOHP, it is associated with **taint metadata** that follows the value through copying, transformation, composition, and transfer. At system exits and policy-relevant boundaries, AOHP checks tainted data before it is displayed, stored, submitted, or transmitted. The taint path provides an audit trail explaining which source reached which sink through which steps.

## 3. Experimental Design

- **Agent and settings.** AOHP is evaluated using the **OpenClaw agent** in two settings:
  - **Stock Android (baseline):** agent receives conventional GUI observations and operates through ordinary app interactions.
  - **AOHP (agent-native):** the same agent additionally uses generated service entrances, structured UI observations, system CLIs/APIs, virtual execution, and sandboxed information flow.
- **Benchmark.** 30 real-world mobile tasks built exclusively from real applications, spanning six categories (5 tasks each):
  1. GUI operation.
  2. Non-GUI operation.
  3. Event capture.
  4. Multi-source information retrieval.
  5. Memory management (two-stage: stage A executes, stage B asks memory-related questions).
  6. Hybrid (capability composition into longer cross-capability workflows).
- **Metrics.** Task completion (checkpoint-weighted completion rate), success count, tool calls, duration, token usage, LLM requests, and security-policy behavior.
- **Security evaluation.** Uses a purpose-built **annotated payment application** exposing account info, payment credentials, transfer inputs, confirmation actions, invoice files, and transaction-related event streams. Case-driven tests exercise source sanitization, sink/action mediation, vault-token use, taint propagation, file handling, event handling, and fail-closed behavior.

## 6. Main Conclusions and Findings

- **Functionality.** On AOHP, OpenClaw reaches a **75.56% average completion rate** (20 tasks fully solved, 5 partially), versus **54.44% on stock Android** (13 fully solved, 7 partially). AOHP raises completion rate by **+21.12 percentage points** and fully solves 7 more tasks. Gains concentrate on tasks involving transient notifications, fine-grained in-app GUI manipulation, and multi-step cross-app or memory-dependent workflows.
- **Efficiency.** On the commonly-solved subset, AOHP reduces:
  - Tool calls by **44.64%** (129 vs 233).
  - Total duration by **44.21%** (18.93 min vs 33.94 min).
  - Total token consumption by **51.55%** (3.44M vs 7.10M).
  - LLM requests by **47.62%** (143 vs 273).
- **Security.** In the prototype evaluation on the annotated payment app, AOHP enforced all five tested security cases as expected: sensitive display redaction, ordinary actions allowed, sensitive actions requiring consent, unsupported access failing closed, and sensitive events redacted with taint preserved.
- **Overall conclusion.** Treating agents as first-class OS actors via personalized service composition, efficient agent interfaces, and secure information flow significantly improves task completion, execution cost, and security-policy compliance, providing an open testbed for agent-native OS research.

## 7. Strengths in Method or Experimental Design

- **System-level approach.** Instead of proposing only a stronger agent policy, AOHP redesigns the OS interfaces and enforcement mechanisms that agents depend on — addressing the root architectural mismatch.
- **Backward compatibility.** Built on AOSP, preserving Android’s hardware support, open-source framework, and app compatibility.
- **Unified interaction model.** Four invocation modes (API, CLI, Structured UI, Rendered GUI) let agents choose the most efficient path while retaining visual fallback.
- **Strong security model.** Combines source sanitization, trusted vault execution, and end-to-end taint tracking across UI, files, event streams, and generated entrances — going beyond static app permissions.
- **Fair efficiency comparison.** Restricting efficiency analysis to tasks solved completely by both settings avoids confounding by task difficulty.
- **Open-source release.** The code is publicly available, enabling the community to build on an agent-native OS testbed.

## 8. Weaknesses and Limitations

- **Compatibility coverage.** Apps with custom rendering, anti-automation logic, or undocumented behavior may not be fully supported. The paper itself notes the need for stronger structured UI extraction, predictable rendered-GUI fallback, and clearer developer compatibility guidelines.
- **Capability discovery dependency.** AOHP relies on accurate service descriptions, side-effect labels, and policy metadata. Legacy apps may require manual annotation; automatic capability inference is not yet mature.
- **Resource scheduling not fully addressed.** Background execution via virtual displays and sandbox runtimes must respect mobile resource, thermal, and memory limits; the paper acknowledges that a fuller prototype needs explicit scheduling policies.
- **Policy usability.** Fine-grained information-flow control depends on user approvals; excessive interruptions could harm usability. Approval UI, trace review, and policy explanations need further refinement.
- **Evaluation scale.** 30 tasks and a single agent (OpenClaw) are sufficient for a preliminary demonstration, but broader validation across more agents, more diverse apps, and longer-horizon tasks is lacking.
- **No training/compute details.** Since no model training is involved, there is no discussion of training cost or environmental footprint, but also no evidence of scalability to large-scale or cloud-deployed agent workloads.
- **Security evaluation is case-driven.** The security study demonstrates correctness on a purpose-built annotated app, but does not provide large-scale or adversarial robustness guarantees.
