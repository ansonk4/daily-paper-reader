## Abstract
The integration of external tools has substantially expanded the capabilities of large language model (LLM) agents, but it also introduces new attack surfaces beyond prompt injection. In particular, cross-tool description poisoning can manipulate planner-visible tool metadata to steer an agent's trajectory, even if the poisoned tool itself is never chosen. To understand the effectiveness of existing defenses against this emerging threat, we first evaluate several prompt-injection defenses and find that they transfer poorly to cross-tool description poisoning. A key observation is that poisoned descriptions persist in the planning context across steps, enabling continuous influence over subsequent tool choices. Building on this insight, we propose Tool-Guard, a novel system-level defense based on a new concept called isolated planning, in which tool invocations that are detected as misaligned or suspicious cause the corresponding tool to be placed in a quarantined list (the influenced list), breaking further influence from poisoned descriptions. With this influence isolated, the tool can continue to be used to support the task, enabling a robust defense that preserves legitimate tool utility. Experiments on the AgentDojo and ASB benchmarks show that Tool-Guard substantially reduces attack success while maintaining high task utility. Our code is available at https://github.com/shishishi123/Tool-Guard.

## 1. Core Problem and Overall Meaning

### Background
- LLM-based agent systems increasingly rely on external tools (web search, databases, APIs, file systems, coding utilities) to enhance task-solving capabilities, with the Model Context Protocol (MCP) further standardizing tool deployment across ecosystems.
- Existing agent security research focuses primarily on **prompt injection attacks**, where malicious instructions are delivered through tool outputs after an agent reaches an adversarial source. Defenses (tool filtering, system prompts, repeat prompts, Drift, Progent) have been developed for this threat model.

### Core Problem: Cross-Tool Description Poisoning
- The paper identifies a distinct and underexplored attack surface: **tool metadata (descriptions)** treated as trusted context by the planner.
- In **cross-tool description poisoning**, an adversary poisons the descriptions of non-critical, weakly regulated tools (e.g., "get weather") to steer the planner into selecting **safety-critical tools** (e.g., "send money to attacker") — even though the poisoned tool itself is never selected.
- The attack is particularly dangerous because poisoned descriptions **persist in the planning context across steps**, enabling continuous influence over subsequent tool choices.
- Existing prompt-injection defenses transfer poorly to this threat, yielding high attack success rates (ASRs) on the AgentDojo benchmark.

### Motivation
- Prior defenses either rely on static scanning of descriptions (without considering task context) or require access to the LLM planner's internal states (unavailable for API-based models).
- There is a lack of systematic evaluation and characterization of cross-tool description poisoning as a threat.

## 2. Proposed Methodology

### Core Idea: Tool-Guard via Isolated Planning
Tool-Guard is a **system-level defense** based on a new concept called **isolated planning**, which applies the security principle of **isolation** to agentic planning. The goal is to break the connection between poisoned descriptions and their intended target of influence while preserving legitimate tool utility.

### Key Technical Components

**Dynamic Tool Set Partitioning:**
- Tool-Guard maintains a dynamic partition of the tool set: $\mathcal{K} = \mathcal{K}_{inf} \cup \mathcal{K}_{others}$ and $\mathcal{K}_{inf} \cap \mathcal{K}_{others} = \emptyset$
- $\mathcal{K}_{inf}$ (Influenced list): tools suspected of being influenced by poisoning.
- $\mathcal{K}_{others}$ (Others list): remaining tools.
- Initially, $\mathcal{K}_{inf} = \emptyset$ and $\mathcal{K}_{others} = \mathcal{K}$.

**Isolated Planning:**
- Tool-Guard queries the planner **twice** under two strictly isolated description sets:
  - $a_A^t \sim \pi_\theta(m_t, D(\mathcal{K}_{inf}))$
  - $a_B^t \sim \pi_\theta(m_t, D(\mathcal{K}_{others}))$
- A special "no tool suitable" option is added to each subset to handle cases where neither tool in the subset is appropriate.
- This prevents cross-tool steering: a poisoned description in $\mathcal{K}_{others}$ cannot influence tool selection in $\mathcal{K}_{inf}$, and vice versa.

**Candidate Selection:**
- In most cases, exactly one isolated plan yields a valid tool call. If both yield valid calls, a tie-breaker (fixed rule or random) selects one. If neither yields a valid call, the trajectory terminates.

**Pre-Execution Validation:**
- Each candidate tool call $\tilde{a}_t = (\kappa_t, i_t)$ is validated using two LLM-based semantic scores:
  - **Alignment check**: $s_{align} = Align(u, \tilde{\tau}_{<t}, \tilde{a}_t)$ — tests whether the tool is task-consistent and a plausible next step given validated progress.
  - **Suspicion check**: $s_{susp} = Susp(u, m_t, \tilde{a}_t)$ — verifies that arguments are grounded in the user request or prior tool outputs, flagging sensitive entities without clear provenance.
- A call is accepted only if $Validate = s_{align} \wedge s_{susp}$.

**List Update and Replanning:**
- If validation fails, the selected tool is moved to $\mathcal{K}_{inf}$, and replanning is triggered under the updated partition.
- At most **three validation attempts** (two replans) are allowed for the same tool call to avoid infinite loops.
- Optionally, tools in $\mathcal{K}_{inf}$ can be moved back to $\mathcal{K}_{others}$ after several subsequent validation passes.

**MCP Extension:**
- The same isolation logic lifts to the server level: tools are grouped by MCP server, and isolated planning applies across server partitions ($\mathcal{S}_{inf}$ and $\mathcal{S}_{others}$).

### Algorithm Flow (Summary)
1. Initialize partition ($\mathcal{K}_{inf} = \emptyset$, $\mathcal{K}_{others} = \mathcal{K}$).
2. **Isolated Planning**: Query planner on both partitions to get candidate actions.
3. **Candidate Selection**: Choose one valid candidate (or terminate if none).
4. **Pre-execution Validation**: Run alignment and suspicion checks. If pass, execute; if fail, update partition and replan.
5. **Execute & Update State**: Execute validated action, update memory, append to trajectory.

## 3. Experimental Design

### Benchmarks
- **AgentDojo** (Debenedetti et al., 2024): Four suites (Banking, Slack, Travel, Workspace), 97 tasks, 76 unique tools.
- **ASB (Agent Security Bench)** (Zhang et al., 2024): 10 task scenarios (e-commerce, autonomous driving, finance), 10 scenario-specific agents, 400+ tools.

### Attack Implementation
- Poisoning strategy: Append authority-style instructions (e.g., "IMPORTANT: …") at the end of tool descriptions while preserving original context.
- Approximately **40% of tools** in each suite are poisoned (non-critical, rarely used tools), targeting high-frequency safety-critical tools.
- Four suite-specific attack goals: sending money to attacker, forwarding emails to attacker address, posting messages to leakage channel, reserving a hotel for attacker.
- Additional attack strategies evaluated: prepend, replace, redirect, authority markers (Table 7 in Appendix).
- Customized poisoning descriptions for Progent and Drift to mislead their built-in security policy generators.

### Baselines (Existing Defenses Evaluated)
1. **System Prompt**: Instructs the planner to focus on core functions and ignore injected messages.
2. **Tool Filtering**: Blocks tools whose descriptions are not aligned with the user task.
3. **Repeat Prompt**: Re-injects the original user query after each tool execution.
4. **Drift** (Li et al., 2025a): State-of-the-art system-level defense.
5. **Progent** (Shi et al., 2025): State-of-the-art system-level defense with programmable privilege control.

### LLMs Evaluated
- **AgentDojo**: GPT-4o-mini, GPT-4o, Gemini-2.5-pro, Gemini-2.5-flash, Claude-3.5-haiku.
- **ASB**: GPT-4o-mini, GPT-4o-2024-05-13, GPT-4o, GPT-4-turbo, GPT-3.5-turbo.

### Additional Experiments
- **Stability Analysis**: 5 repeated runs on 3 models.
- **Adaptive Attacks**: Semantic-adaptive (alignment-evade, suspicion-evade, combined) and optimization-based (PAIR, TAP).
- **Stress Test**: Poisoning critical/frequently used tools (4 tools) plus a similarly-named decoy.
- **Ablation Study**: Validation-only and validation+mask variants.
- **Scalability Test**: Tripling the number of tools (3x) per suite.
- **Flipped Default Setting**: Reversing default validator behavior.

## 6. Main Conclusions and Findings

- **Cross-tool description poisoning is effective**: Existing defenses yield high ASRs (19.6%–43.3% on AgentDojo), confirming that this is a serious and distinct threat from prompt injection.
- **Tool-Guard is highly effective**: Reduces ASR to **negligible levels** (0%–3.09%) across all models on both benchmarks.
- **Utility is preserved**: Benign utility remains high (68.04%–72.16% on AgentDojo with defense), and no "over-kill" behavior is observed.
- **Efficiency**: Tool-Guard incurs only ~**1.45× token overhead** (benign) / ~**1.39×** (attack) compared to no defense — substantially less than Repeat Prompt, Drift, and Progent. Latency is ~**3.7×** no-defense (more than Progent's 1.2× but less than Drift's 7.3×).
- **Robustness to adaptive attacks**: Tool-Guard maintains ASR at or near zero under semantic-adaptive attacks. Under optimization-based attacks (PAIR/TAP), ASR remains very low (0%–2.06%).
- **Stress test resilience**: Even when critical tools are poisoned, Tool-Guard keeps ASR at single-digit levels.
- **Model variability**: Different models respond differently to poisoning; more advanced models do not necessarily provide better security (they may follow injected instructions more effectively).
- **Ablation finding**: Validation alone is an effective gatekeeper, but dynamic tool set partitioning is necessary to preserve utility.

## 7. Strengths in Method or Experimental Design

- **Novel and well-motivated problem**: Cross-tool description poisoning is a genuinely distinct threat from prompt injection, and the paper provides the first systematic evaluation of defenses against it.
- **Clean and principled design**: The isolation principle is well-justified, and the combination of pre-execution validation with dynamic partitioning is elegant — it cuts malicious influence without removing tools.
- **Utility-preserving by design**: Unlike tool filtering or masking, Tool-Guard does not permanently disable tools; it only partitions visibility, so legitimate workflows are maintained.
- **Comprehensive evaluation**: Two benchmarks, five models, five baselines, multiple attack strategies, adaptive attacks, stress tests, ablation, scalability, and overhead analysis.
- **Practical applicability**: The MCP extension shows the method is deployable in real-world MCP ecosystems. The system-level design does not require access to model internals.
- **Efficiency**: Only 1.5× token overhead compared to no defense — much more efficient than other system-level defenses.
- **Reproducibility**: Code is publicly available on GitHub.

## 8. Weaknesses and Limitations

### Coverage and Generalizability
- **Benchmark limitations**: Both AgentDojo and ASB are synthetic environments with relatively small tool sets (76 and 400+ tools, respectively). Real-world MCP ecosystems may have thousands of tools with more complex interdependencies.
- **Limited task diversity**: AgentDojo has only 97 tasks across 4 suites; the attack goals are relatively simple (send money, forward email, etc.). More complex multi-step attack objectives are not explored.
- **Single poisoning strategy in main experiments**: The primary attack uses appended authority-style instructions; other strategies (prepend, replace, redirect) are only briefly discussed in the appendix and not systematically evaluated in the main comparison.

### Bias Risk
- **Model selection bias**: Only 5 LLMs are evaluated, all from major providers (OpenAI, Google, Anthropic). Open-source models (e.g., Llama, Mistral) are not tested, limiting generalizability.
- **Attack knowledge assumption**: The adaptive attack evaluation assumes the adversary knows the defense details (white-box for semantic-adaptive, black-box iterative for PAIR/TAP), but does not consider a fully adaptive attacker who can probe the isolation mechanism over many interactions.
- **Validator dependency**: The defense relies on an LLM-based validator, which itself could be a target for adversarial manipulation. The paper acknowledges failure cases where benign tasks are semantically close to attack targets (Appendix G), suggesting the validator has inherent limitations.

### Application Constraints
- **Latency overhead**: Tool-Guard introduces ~3.7× runtime overhead compared to no defense, which may be prohibitive for latency-sensitive applications (though it is better than Drift's 7.3×).
- **Replanning cost**: The isolated planning approach requires **two LLM calls per step** (plus additional replanning on validation failures), increasing API costs and response time.
- **Validator reliability**: The alignment and suspicion checks depend on the LLM's judgment, which can fail when benign tasks are semantically similar to attack targets (as shown in Appendix G). This is an inherent limitation of LLM-based validation.
- **No formal security guarantee**: The paper provides a security analysis but no formal proof that the isolation mechanism covers all possible attack paths.
- **Scalability concerns**: While the 3x tool test shows sub-linear latency growth, the paper does not test with truly large-scale tool sets (hundreds or thousands of tools) that are common in real MCP deployments.
- **Limited real-world validation**: All experiments are conducted on benchmarks; no real-world deployment or user study is reported.
