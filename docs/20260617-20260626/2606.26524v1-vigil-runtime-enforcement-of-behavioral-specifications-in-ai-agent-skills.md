## Abstract
Agentic systems increasingly act through third-party skills, allowing model-generated decisions to affect files, communication channels, and cyber-physical devices. These skills often include natural-language specifications that define access permissions, disclosure limits, execution privileges, and required preconditions. Although such specifications describe the intended boundaries of skill behavior, they do not by themselves provide executable runtime enforcement. Enforcing them raises a contextual granularity challenge: even when a policy is written for a particular task context, a monitor must still decide which events to observe, what state to retain, how far across the execution to reason, and where to intervene. Choosing the wrong granularity can either block benign executions or miss violations that emerge only across multiple actions. Most existing enforcement mechanisms, however, assume a fixed event model or enforcement point.   In this work, we present VIGIL, an end-to-end runtime enforcement framework for agentic systems. VIGIL checks an agent's actual execution trace against behavioral policies from skill specifications, operator-defined constraints, and global rules spanning multiple skills. To make such policies executable, VIGIL introduces a policy language that captures context-specific enforcement requirements over agent-tool events, including temporal dependencies, argument constraints, and value-flow conditions. The language is paired with symbolic evaluation rules that translate policies into SMT constraints over finite traces, allowing VIGIL to detect violations that depend on event order, argument relationships, or cross-call value flow rather than relying on fixed single-call filters. On real LLM-agent runs spanning office-document, operational, and engineering tasks, VIGIL detects policy violations with over 95% recall and a false-positive rate below 10%.

## 1. Core Problem and Overall Meaning

- **Problem motivation.** Modern LLM agents execute tasks by invoking third-party *skills* (packaged tool bundles with natural-language behavioral specifications). These specifications declare what the agent may or must not do — confirmation before destructive actions, validation before data release, prerequisite checks, and so on. However, today's deployments blindly trust the agent to follow these rules; nothing enforces them at runtime. Real-world incidents (data exfiltration via prompt injection, rogue agent behavior on inboxes) show this trust is already broken.
- **Structural blind spot.** The most dangerous violations are *trace-level*: each individual tool call looks benign and schema-valid, but the *sequence* breaks the specification (e.g., an intermediate artifact is consumed without ever being validated). Existing defenses — sandboxes, action-boundary monitors, information-flow controls, offline LLM judges — cannot reason across multiple events, artifact identities, and temporal orderings simultaneously.
- **Contextual granularity challenge.** A monitor must decide *which* events to observe, *what* state to retain, *how far* across the execution to reason, and *where* to intervene. Choosing wrong granularity either blocks benign runs or misses multi-action violations.
- **Goal.** VIGIL is an end-to-end runtime reference monitor that makes behavioral specifications executable — it intercepts each pending tool invocation, checks the full execution trace against a formal policy, and blocks violations before their effects land.

## 2. Proposed Methodology

### 2.1 Core Idea

VIGIL strictly separates a behavioral specification's *generic meaning* from its *run-specific instantiation*. It operates in a four-step lifecycle:

1. **Trace Abstraction** — Raw tool calls are parsed into strongly typed events `e = (a(e), e[·], s(e), o(e))` capturing action, named arguments, status, and bounded output. Events are ordered into a finite trace `τ = e₁ ≺ e₂ ≺ …`.
2. **Policy Grounding** — Abstract natural-language specifications are compiled into a policy over the run's own vocabulary (its *signature* Σ of actions, argument names, and values).
3. **SMT Check** — Each policy statement is reduced to a quantifier-free SMT constraint over the trace's fact base; a single Z3 query decides whether allowing the pending action would violate the policy.
4. **Intervention** — On violation, the solver's unsatisfiable core yields a localized witness naming the offending invocation; the runtime blocks, rolls back, replans, or escalates.

### 2.2 Behavioral Policy Language

The language provides six temporal forms over patterns:

| Form | Meaning |
|---|---|
| `abs(ψ)` | Prohibition — ψ must never occur |
| `prec(ψ, ψ')` | Precedence — ψ allowed only after ψ' has happened |
| `resp(ψ, ψ')` | Response — every ψ must be followed by ψ' |
| `bresp_ℓ(ψ, ψ')` | Bounded response — within next ℓ events |
| `rslv(ψ, ψ')` | Final-state resolution — every ψ resolved by run end |
| `until(ψ, ψ', ψ_b)` | Interval guard — after ψ, no ψ_b before ψ' |

Patterns select events by action, argument values/text patterns, output, and status. Value variables (`x`) tie two patterns to the same artifact identity across calls.

### 2.3 Formal Reduction

Each statement φ compiles to a requirement R_φ via syntax-directed rules (Figure 5 in the paper). The key rule for precedence:

```
R_prec = ⋀_e [ ψ(e) ⇒ ⋁_{e'≺e} ψ'(e') ]
```

The fact base JτK is a complete set of ground atoms over the trace. The decision procedure is:

```
unsat(JτK ∧ R_φ) ⇔ τ ⊭ φ
```

Localization extracts the unsatisfying core's named events:

```
loc(φ) = { e ∈ E | R_φ(e) ∈ muc(JτK ∧ R_φ) }
```

### 2.4 Algorithm

```
procedure VIGIL(L, D):
    (Σ, JτK) ← ABSTRACT(L)
    P ← COMPILE(D, Σ)
    W ← ∅
    for each φ ∈ P:
        R_φ ← encode(φ, Σ)
        if unsat(JτK ∧ R_φ):
            W ← W ∪ {(φ, loc(φ))}
    return (safe, ∅) if W=∅ else (unsafe, W)
```

An LLM (Claude Opus 4.6) is used *only* in COMPILE to draft policy statements; all grounding, validation, and decision-making is deterministic. The LLM is constrained to use only vocabulary recovered from the trace, and ungroundable clauses are rejected.

## 3. Experimental Design

### 3.1 Datasets and Benchmarks

- **SB+SI (primary):** 152 labeled real agent executions (72 violating, 80 benign) drawn from SkillsBench and Skill-Inject, spanning office-document, operational, and engineering tasks.
- **AgentDojo:** 629 trajectories for prompt-injection attack/defense evaluation.
- **SafeAgentBench:** 546 trajectories for safe task planning in embodied agents.
- **Real-world study:** 216 executions from deployed skill bundles (NVIDIA, Databricks, Trail of Bits, Cloudflare, Anthropic, Microsoft).

### 3.2 Baselines

| Baseline | Paradigm |
|---|---|
| AgentSpec | Action-boundary enforcement with temporal triggers |
| Progent | Per-tool-call permission allowlists |
| LLM-as-judge | Full-trace LLM verdict (Claude Opus 4.6) without formal grounding |

### 3.3 Metrics

- TP, FP, Precision, Recall, F1 for violation detection.
- Balanced Enforcement Error (BEE) = ½(FNR + FPR) — punishes both missed violations and false blocks equally.
- End-to-end latency and per-stage latency.

## 6. Main Conclusions and Findings

- **Detection effectiveness on SB+SI:** VIGIL achieves 95.8% recall, 89.6% precision, 92.6% F1 — leading the strongest baseline (LLM-as-judge) by 15.8 F1 points. Action-boundary defenses (AgentSpec 57.6% F1, Progent 50.3% F1) miss roughly half of trace-level violations.
- **Cross-benchmark transfer:** VIGIL ranks first on both AgentDojo (94.2% F1) and SafeAgentBench (96.3% F1) without per-benchmark tuning.
- **Balanced enforcement error:** VIGIL attains the lowest macro BEE (5.8%) across all three benchmarks — less than half the next-best complete baseline.
- **Real-world violations:** 34 confirmed policy-violating runs out of 44 alerts across 216 real-world executions, spanning specification gaps (NVIDIA, Trail of Bits, Cloudflare) and agent execution gaps (Databricks, Anthropic, Microsoft). NVIDIA acknowledged one specification defect.
- **Ablation:** Removing argument grounding costs 31.9 recall points; removing policy structure costs 5.5 recall points — both components are essential.
- **Overhead:** Online verification decides in 0.27 s; total end-to-end cost plateaus as trace vocabulary saturates (~10 KB), making it negligible relative to agent tool-call timescales.

## 7. Strengths

- **Principled formal foundation.** The reduction of finite-trace temporal safety properties to SMT is sound and complete for the supported property class; the unsatisfiable core provides free localization.
- **Separation of concerns.** Using the LLM only for policy drafting (not for verdict decisions) keeps the enforcement path deterministic and auditable.
- **Practical architecture.** The four-step lifecycle (abstract → ground → check → intervene) maps cleanly onto existing agent runtimes with sandbox/tool-mediation hooks.
- **Comprehensive evaluation.** Combines labeled benchmarks, cross-benchmark transfer, ablation, error analysis, and a real-world study with responsible disclosure — a rare breadth.
- **Strong empirical results.** Consistently outperforms both formal action-boundary and informal LLM-judge baselines, especially on the hardest trace-level violations.

---

## 8. Weaknesses and Limitations

- **Trace observability boundary.** Violations hidden inside opaque utility scripts (e.g., `secure_archive.py` performing unlink without a trace-level validation event) are invisible to VIGIL. The closed-world assumption is only as good as the instrumentation.
- **Compilation fidelity.** The LLM-based COMPILE step can over-generalize specifications (e.g., dropping the "bulk" qualifier from a calendar-edit rule), causing false positives. This is the dominant source of false alarms.
- **Benchmark size.** The primary SB+SI benchmark has only 152 runs; while manually labeled and high-quality, this limits statistical confidence for subgroup analyses.
- **No GPU/compute reporting.** The paper does not disclose whether LLM API calls were made to cloud infrastructure, preventing full reproducibility assessment.
- **Supported property class.** VIGIL targets finite-trace safety properties (precedence, response, absence, bounded response, resolution, until). It does not handle liveness properties, probabilistic guarantees, or properties requiring infinite-trace reasoning.
- **Single-agent focus.** The framework assumes a single untrusted agent's trace; multi-agent or concurrent-systems scenarios are not addressed.
- **Deployment assumptions.** Requires a trusted collector that records all tool calls completely and a runtime that can pause and block pending invocations — assumptions that may not hold in all deployment environments.
